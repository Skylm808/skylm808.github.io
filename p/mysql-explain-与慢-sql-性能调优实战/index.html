MySQL Explain 与慢 SQL 性能调优实战

Mar 06, 2026
阅读时长: 10 分钟

在 MySQL 性能优化中，Explain 是最重要的分析工具。它能够展示 MySQL 如何执行一条 SQL 语句，帮助我们发现性能瓶颈并进行针对性优化。

一、Explain 基础

1.1 基本用法 

1

EXPLAIN SELECT * FROM users WHERE age > 25;

Explain 会返回一个执行计划表，包含以下关键字段：

字段 说明
id 查询的执行顺序标识
select_type 查询类型（SIMPLE、PRIMARY、SUBQUERY 等）
table 正在访问的表名
type 访问类型（性能从好到坏：system > const > eq_ref > ref > range > index > ALL）
possible_keys 可能使用的索引
key 实际使用的索引
key_len 使用的索引长度
ref 索引的哪一列被使用了
rows 预计扫描的行数
Extra 额外信息（Using index、Using filesort、Using temporary 等）

1.2 type 字段详解（重点）

type 是判断查询性能的关键指标，从最优到最差：

  1. system：表只有一行记录（系统表），这是 const 的特例
  2. const：通过主键或唯一索引查询，最多返回一条记录
  3. eq_ref：唯一索引扫描，对于每个索引键，表中只有一条记录匹配（常见于主键或唯一索引的 JOIN）
  4. ref：非唯一索引扫描，返回匹配某个单独值的所有行
  5. range：索引范围扫描（使用 >、<、BETWEEN、IN 等）
  6. index：全索引扫描（扫描整个索引树）
  7. ALL：全表扫描（最差，需要优化）

优化目标：至少达到 range 级别，最好是 ref 或更好。

二、实战案例分析

案例 1：全表扫描优化

场景：用户表查询，根据年龄筛选

1
2

-- 原始 SQL
SELECT * FROM users WHERE age > 25;

Explain 分析：

1

EXPLAIN SELECT * FROM users WHERE age > 25;

结果：

1
2
3
4
5

+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows   | Extra       |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
|  1 | SIMPLE      | users | ALL  | NULL          | NULL | NULL    | NULL | 100000 | Using where |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+

问题诊断：

  • type = ALL：全表扫描，性能最差
  • rows = 100000：需要扫描 10 万行
  • key = NULL：没有使用任何索引
  • Extra = Using where：在 Server 层进行过滤

优化方案：为 age 字段添加索引

1
2
3
4
5

-- 创建索引
CREATE INDEX idx_age ON users(age);

-- 再次 Explain
EXPLAIN SELECT * FROM users WHERE age > 25;

优化后结果：

1
2
3
4
5

+----+-------------+-------+-------+---------------+---------+---------+------+-------+-----------------------+
| id | select_type | table | type  | possible_keys | key     | key_len | ref  | rows  | Extra                 |
+----+-------------+-------+-------+---------------+---------+---------+------+-------+-----------------------+
|  1 | SIMPLE      | users | range | idx_age       | idx_age | 4       | NULL | 50000 | Using index condition |
+----+-------------+-------+-------+---------------+---------+---------+------+-------+-----------------------+

优化效果：

  • type 从 ALL 提升到 range
  • 使用了 idx_age 索引
  • 预计扫描行数减少到 5 万行（实际执行会更快）

案例 2：索引失效问题

场景：根据姓名模糊查询

1
2

-- 原始 SQL（索引失效）
SELECT * FROM users WHERE name LIKE '%张%';

Explain 分析：

1
2
3
4
5

+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows   | Extra       |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
|  1 | SIMPLE      | users | ALL  | NULL          | NULL | NULL    | NULL | 100000 | Using where |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+

问题诊断：

  • 即使 name 字段有索引，但 LIKE '%张%' 导致索引失效
  • 原因：前缀通配符 % 无法利用 B+Tree 索引的有序性

优化方案：

  1. 改用前缀匹配（如果业务允许）：
1

SELECT * FROM users WHERE name LIKE '张%';
  1. 使用全文索引（更适合关键词搜索，FULLTEXT 常见于 MySQL 5.6+；ngram 解析器常见于 5.7+）：
1
2

-- 1) 建立全文索引（英文或已做好分词的文本可直接用）
ALTER TABLE users ADD FULLTEXT INDEX ft_name(name);

分词设置（非常关键）：

  • 全文索引能否命中，取决于“分词后的词项”。
  • InnoDB 常见参数：innodb_ft_min_token_size（最小词长，默认通常是 3）。
  • MyISAM 常见参数：ft_min_word_len（最小词长，默认通常是 4）。
  • 中文建议用 ngram 分词器，否则单字（如“张”）可能搜不到。
1
2
3
4
5
6
7

-- 查看分词相关参数
SHOW VARIABLES LIKE 'innodb_ft_min_token_size';
SHOW VARIABLES LIKE 'ft_min_word_len';
SHOW VARIABLES LIKE 'ngram_token_size';

-- 中文场景建议：直接用 ngram 建全文索引（与上面的普通 FULLTEXT 二选一）
ALTER TABLE users ADD FULLTEXT INDEX ft_name(name) WITH PARSER ngram;

修改最小词长参数后，通常需要重启 MySQL 并重建全文索引才会生效。

MATCH + AGAINST 用法：

1
2
3

-- 2) 自然语言模式：按相关度排序（默认模式）
SELECT id, name
FROM users

-- 3) 布尔模式：可用 + - * 控制关键词

  • AGAINST(...)：指定搜索词和搜索模式（自然语言 / 布尔模式）。

  1. 使用 Elasticsearch（推荐）：将搜索需求迁移到专业搜索引擎

案例 3：联合索引优化

场景：多条件查询

1
2
3
4
5

-- 原始 SQL
SELECT * FROM orders
WHERE user_id = 1001
  AND status = 'paid'
  AND create_time > '2026-01-01';

Explain 分析（假设只有单列索引）：

1
2
3
4
5

+----+-------------+--------+------+------------------+--------------+---------+-------+------+-------------+
| id | select_type | table  | type | possible_keys    | key          | key_len | ref   | rows | Extra       |
+----+-------------+--------+------+------------------+--------------+---------+-------+------+-------------+
|  1 | SIMPLE      | orders | ref  | idx_user_id      | idx_user_id  | 4       | const | 5000 | Using where |
+----+-------------+--------+------+------------------+--------------+---------+-------+------+-------------+

问题诊断：

  • 只使用了 idx_user_id 索引
  • status 和 create_time 条件在 Server 层过滤（Using where）
  • 需要扫描 5000 行后再过滤

优化方案：创建联合索引（遵循最左前缀原则）

1
2
3
4
5
6
7
8

-- 创建联合索引（顺序很重要！）
CREATE INDEX idx_user_status_time ON orders(user_id, status, create_time);

-- 再次 Explain
EXPLAIN SELECT * FROM orders
WHERE user_id = 1001
  AND status = 'paid'
  AND create_time > '2026-01-01';

优化后结果：

1
2
3
4
5

+----+-------------+--------+-------+----------------------+----------------------+---------+------+------+-----------------------+
| id | select_type | table  | type  | possible_keys        | key                  | key_len | ref  | rows | Extra                 |
+----+-------------+--------+-------+----------------------+----------------------+---------+------+------+-----------------------+
|  1 | SIMPLE      | orders | range | idx_user_status_time | idx_user_status_time | 14      | NULL | 100  | Using index condition |
+----+-------------+--------+-------+----------------------+----------------------+---------+------+------+-----------------------+

优化效果：

  • 预计扫描行数从 5000 降到 100
  • 三个条件都在索引中完成过滤

联合索引顺序原则：

  1. 等值条件优先：user_id = 1001（精确匹配）
  2. 范围条件靠后：create_time > '2026-01-01'（范围查询会终止索引使用）
  3. 区分度高的字段优先：如果都是等值，选择区分度高的

案例 4：覆盖索引优化

场景：只查询索引列

1

3
4

-- 原始 SQL
SELECT user_id, status, create_time
FROM orders
WHERE user_id = 1001;

Explain 分析（使用上面的联合索引）：

1

+----+-------------+--------+------+----------------------+----------------------+---------+-------+------+-------------+
| id | select_type | table  | type | possible_keys        | key                  | key_len | ref   | rows | Extra       |

| id | select_type | table | type | possible_keys | key        | key_len | ref            | rows | Extra       |

CREATE INDEX idx_status ON orders(status);

-- 2. 确保 users.id 有主键索引（通常已有）

-- 再次 Explain
EXPLAIN SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'paid';

优化后结果：

|  1 | SIMPLE      | o     | ref    | idx_status    | idx_status | 10      | const          | 100 | NULL  |

+----+-------------+-------+--------+---------------+------------+---------+----------------+-----+-------+

优化效果：

  • orders 表使用 idx_status 索引，type = ref
  • users 表使用主键索引，type = eq_ref（最优 JOIN 类型）
  • 预计扫描 100 × 1 = 100 次
  • ref = const 表示 o.status = 'paid' 是“与常量比较”；rows = 100 表示预计有约 100 行满足该常量条件，这两者不冲突

三、Extra 字段详解

Extra 是 Explain 里最容易“暴露问题”的字段。
同样的 type 下，Extra 往往决定了 SQL 实际快不快。

建议看 Explain 时按这个顺序判断：

  1. 先看有没有 Using filesort / Using temporary / Using join buffer
  2. 再看有没有 Using index / Using index condition
  3. 最后结合 rows 看扫描量是否可接受

3.1 好的信号

  • Using index

    • 含义：覆盖索引，只读索引就能拿到结果，不用回表。
    • 何时出现：SELECT 的列都在索引里。
    • 优化意义：通常是非常好的信号，随机 I/O 少。

  • Using index condition

    • 含义：索引下推（ICP），把一部分 WHERE 条件下推到存储引擎层过滤。
    • 何时出现：范围扫描或联合索引中，部分条件可在索引层先过滤。
    • 优化意义：比单纯 Using where 更好，可减少回表次数。

  • Using where

    • 含义：MySQL Server 层还需要再做条件过滤。
    • 何时出现：条件不能完全靠索引完成，或本来就没有可用索引。
    • 说明：它本身不是“坏信号”，要结合 rows 一起看；rows 大才危险。

3.2 需要优化的信号

  • Using filesort

    • 含义：无法按索引顺序直接返回结果，需要额外排序（不一定落盘，但有额外开销）。

    • 常见原因：ORDER BY 字段没有匹配索引，或排序方向/字段顺序与索引不一致。

    • 优化动作：让 WHERE + ORDER BY 命中同一个索引，必要时建联合索引。

      1
2

      -- 示例
SELECT * FROM users ORDER BY age;  -- 如果 age 无索引

– 优化：创建索引

1

CREATE INDEX idx_age ON users(age);

  • Using temporary

    • 含义：执行过程中需要临时表，常见于 GROUP BY / DISTINCT / UNION。

    • 常见原因：分组或去重列没有合适索引，或查询列过多导致无法走索引聚合。

    • 优化动作：给分组/去重字段建索引，尽量减少不必要列，必要时改写 SQL。

      1
2

      -- 示例
SELECT status, COUNT(*) FROM orders GROUP BY status;

– 优化：为 GROUP BY 字段创建索引

1

CREATE INDEX idx_status ON orders(status);

  • Using join buffer

    • 含义：JOIN 不能有效使用被驱动表索引，只能用 join buffer 做匹配。

    • 常见原因：ON 条件字段缺索引，或类型不一致导致索引失效。

    • 优化动作：给被驱动表 JOIN 列建索引，保证连接字段类型一致。

      1
2

      -- 优化：为 JOIN 字段创建索引
CREATE INDEX idx_user_id ON orders(user_id);

3.3 常见组合怎么判断

  • Using where; Using index

    • 通常表示“覆盖索引 + 条件过滤”，一般不错。

  • Using index condition; Using where

    • 表示先 ICP 过滤，再做部分 Server 层过滤，通常比只有 Using where 好。

  • Using where; Using filesort

    • 过滤后还要额外排序，数据量一大就容易慢，优先优化排序索引。

  • Using temporary; Using filesort

    • 高风险组合（分组/去重 + 排序），通常是慢 SQL 常见形态，优先处理。

五、性能调优检查清单

5.1 索引优化

  • 为 WHERE、JOIN、ORDER BY、GROUP BY 字段创建索引
  • 使用联合索引，遵循最左前缀原则
  • 避免索引失效：不使用函数、不使用 !=、不使用前缀通配符 %
  • 利用覆盖索引，减少回表
  • 定期删除冗余索引

5.2 SQL 优化

  • 避免 SELECT *，只查询需要的字段
  • 使用 LIMIT 限制返回行数
  • 子查询改 JOIN
  • 避免在 WHERE 中使用函数：WHERE DATE(create_time) = '2026-01-01' → WHERE create_time >= '2026-01-01' AND create_time < '2026-01-02'
  • 使用 UNION ALL 代替 UNION（如果不需要去重）

5.3 表结构优化

  • 选择合适的数据类型（INT 比 VARCHAR 快）
  • 使用 NOT NULL + DEFAULT（避免 NULL 值）
  • 大字段（TEXT、BLOB）拆分到单独的表
  • 垂直分表、水平分表（数据量大时）

5.4 配置优化

  • 调整 innodb_buffer_pool_size（建议设置为物理内存的 70-80%）
  • 调整 max_connections（根据并发量）
  • 查询缓存仅适用于 MySQL 5.7 及以下，且高并发场景通常不建议开启（MySQL 8.0 已移除）

七、总结

Explain 核心关注点：

  1. type：至少达到 range，最好 ref 或 eq_ref
  2. key：确保使用了索引
  3. rows：预计扫描行数越少越好
  4. Extra：避免 Using filesort 和 Using temporary

优化思路：

  1. 先用 Explain 分析执行计划
  2. 找出全表扫描（type = ALL）和大量扫描行（rows 大）
  3. 为过滤、排序、分组字段创建索引
  4. 优化 SQL 写法（子查询改 JOIN、避免函数等）
  5. 再次 Explain 验证优化效果

记住：索引不是越多越好，每个索引都会增加写入成本。根据实际查询场景，创建合适的索引才是关键。
