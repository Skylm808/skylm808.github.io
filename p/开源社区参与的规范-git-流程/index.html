开源社区参与的规范 Git 流程

阅读时长: 11 分钟

开源社区参与的规范 Git 流程

这份文档以 GitHub 上最常见的 Fork + 分支 + Pull Request 工作流为主，适合你现在这种参与 Apache 项目、给官方仓库提 PR 的场景。

文中会重点说明：

  • 每一步应该做什么
  • 每个命令的作用是什么
  • 什么时候要同步官方仓库
  • origin、upstream、fetch、rebase、PR 分别是什么意思
  • 结合 dubbo-go-pixiu / dubbo-go-pixiu-samples / dubbo-website 这类实际场景，应该怎么操作

1. 先理解几个核心概念

在开始之前，先把几个最容易混淆的概念分清楚。

1.1 Fork 是什么

Fork 是 GitHub 上“把别人的仓库复制一份到你自己账号下”的动作。

例如：

  • 官方仓库：apache/dubbo-website
  • 你的 fork：Skylm808/dubbo-website

Fork 之后：

  • 你有了自己可写的仓库副本
  • 你可以在自己的仓库里自由建分支、提交、推送
  • 最后再从自己的 fork 向官方仓库发起 PR

注意：

  • Fork 不是自动实时同步官方最新代码
  • 你 fork 的那一刻通常是最新状态
  • 但后面官方仓库继续更新时，你自己的 fork 不会自动跟着变

1.2 origin 是什么

origin 通常指向你本地 clone 下来的默认远程仓库。

如果你是从自己的 fork clone 下来的：

1

git clone https://github.com/Skylm808/dubbo-website.git

那么本地默认的：

  • origin = https://github.com/Skylm808/dubbo-website.git

也就是：你自己的 fork

1.3 upstream 是什么

upstream 一般是你手动加上的“官方源仓库”。

例如：

1

git remote add upstream https://github.com/apache/dubbo-website.git

这时：

  • origin = 你自己的 fork
  • upstream = 官方仓库

这样做的意义是：

  • 你平时 push 到 origin
  • 你平时从 upstream 获取官方最新代码

1.4 main 和 master 是什么

这是仓库的主分支名。

不同仓库不一样：

  • 有的仓库主分支叫 main
  • 有的仓库主分支叫 master

你实际操作时，必须以该仓库真实分支名为准。

例如你这次：

  • dubbo-website 用的是 master

所以命令里就应该写：

1

git rebase upstream/master

而不是：

1

git rebase upstream/main

1.5 git fetch 是什么

git fetch 的作用是：

  • 把远程仓库的最新提交信息下载到本地
  • 但不会直接改动你当前分支的代码

例如：

1

git fetch upstream

它做的事情是：

  • 把官方仓库最新信息取回来
  • 更新本地的远程跟踪分支，例如 upstream/master
  • 但不会自动把这些内容合并进你当前分支

你可以把它理解成：

  • “先把远程最新情况拿回来”
  • “但先别动我现在正在写的代码”

1.6 git rebase upstream/master 是什么

这个命令的核心含义是：

  • 把你当前所在的本地分支，变基到 upstream/master 最新代码之上

例如你当前在 feature/pixiu-saml-docs 分支上：

1

git rebase upstream/master

意思就是：

  • 先拿 upstream/master 作为新的起点
  • 再把你当前分支上的提交，重新“接”到这个新起点后面

它是对“当前分支”生效的，不是对所有分支生效。

所以要特别注意：

  • 你在哪个分支上执行
  • 它就会改哪个分支的提交历史

1.7 git branch 和 git branch -a 的区别 

1

git branch

只看 本地分支。

1

git branch -a

看 本地分支 + 远程跟踪分支。

例如：

1
2
3

* master
  remotes/origin/master
  remotes/upstream/master

含义分别是：

  • * master：你当前所在的本地分支是 master
  • remotes/origin/master：你本地记录的“你自己 fork 上的 master”
  • remotes/upstream/master：你本地记录的“官方仓库上的 master”

2. 最规范、最常见的开源提交流程

下面是最推荐的流程。

适合场景：

  • 你要给官方仓库提 PR
  • 你没有官方仓库写权限
  • 你是通过 fork 参与开源

3. 第一次参与某个仓库时的完整流程

步骤 1：先在 GitHub 页面上 Fork 官方仓库

例如把：

  • apache/dubbo-website

Fork 到你的账号下，得到：

  • Skylm808/dubbo-website

这个步骤是网页操作，不是命令行。

作用：

  • 获得一个你自己能推送代码的仓库副本

步骤 2：把你自己的 fork clone 到本地 

1
2

git clone https://github.com/Skylm808/dubbo-website.git
cd dubbo-website

命令解释：

  • git clone <url>：把远程仓库完整下载到本地
  • cd dubbo-website：进入这个本地仓库目录

此时本地默认远程仓库只有一个：

  • origin

它指向你的 fork。

步骤 3：给本地仓库配置官方上游仓库 upstream 

1
2

git remote add upstream https://github.com/apache/dubbo-website.git
git remote -v

命令解释：

  • git remote add upstream <url>：新增一个叫 upstream 的远程仓库地址
  • git remote -v：查看当前所有远程仓库配置

正常会看到类似：

1
2
3
4

origin    https://github.com/Skylm808/dubbo-website.git (fetch)
origin    https://github.com/Skylm808/dubbo-website.git (push)
upstream  https://github.com/apache/dubbo-website.git (fetch)
upstream  https://github.com/apache/dubbo-website.git (push)

注意：

  • 这不代表你真能往 upstream push
  • 这里只是 Git 配置里记录了这个地址
  • 你有没有权限 push，要看 GitHub 权限

步骤 4：先同步一次官方最新代码 

1
2
3

git fetch upstream master
git checkout master
git rebase upstream/master

命令解释：

  • git fetch upstream master

    • 只获取官方仓库的 master 分支信息
    • 比 git fetch upstream 更省一些
    • 尤其是大仓库时，能避免把很多无关分支一起拉下来

  • git checkout master

    • 切换到本地 master 分支
    • 如果你已经在 master，它会提示你已经在这个分支

  • git rebase upstream/master

    • 把你当前本地 master 分支，更新到官方 master 的最新提交之上

如果提示：

1

Current branch master is up to date.

意思是：

  • 你的本地 master 已经和官方 master 一样新了

步骤 5：可选，把你 fork 上的主分支也同步成最新 

1

git push origin master

命令解释：

  • 把你本地更新后的 master 推到你自己的 fork 上

作用：

  • 让你 GitHub 上的 fork 也保持和官方主分支接近同步

这一步不是强制，但推荐做。

步骤 6：从最新主分支切出你自己的功能分支 

1

git checkout -b feature/pixiu-saml-docs

命令解释：

  • git checkout -b <branch>
    • 新建一个本地分支
    • 并立刻切换到这个新分支

为什么不要直接在 master 上开发：

  • master 应该尽量保持干净，方便同步官方更新
  • 你的实际改动应该放在独立 feature 分支里
  • 后续提 PR 也更清晰

步骤 7：开始修改代码或文档

开发过程中最常用的是：

1
2

git status
git diff

命令解释：

  • git status

    • 查看当前工作区状态
    • 哪些文件改了
    • 哪些文件已暂存
    • 当前在哪个分支

  • git diff

    • 查看还没 git add 的修改内容

如果你想看已经 add 但还没 commit 的内容，可以用：

1

git diff --cached

步骤 8：先本地测试，再提交

这一步非常重要。

在开源社区里，提交前最好先确认：

  • 功能能跑
  • 测试通过
  • 文档格式没问题

例如 Go 项目里常见：

1

go test ./...

或者像你这次的 sample：

1

go test -v ./dubbogo/simple/saml/test

命令解释：

  • go test ./...：运行当前模块下所有 Go 测试
  • go test -v <path>：详细模式运行指定目录下的测试

步骤 9：把改动加入暂存区 

1
2

git add content/en/overview/reference/pixiu/other/user/httpfilter/saml.md
git add content/zh-cn/overview/reference/pixiu/other/user/httpfilter/saml.md

或者：

1

git add .

命令解释：

  • git add <file>：把指定文件加入本次提交
  • git add .：把当前目录下所有改动都加进去

规范建议：

  • 优先使用精确的 git add <file>
  • 这样不容易把不该提交的文件一起带上去

步骤 10：提交到本地 Git 历史 

1

git commit -m "docs(pixiu): add saml filter docs"

命令解释：

  • git commit -m "<message>"
    • 生成一个本地提交
    • -m 后面是提交说明

好的提交信息应该：

  • 简洁
  • 准确
  • 一看就知道这次改了什么

常见风格：

  • feat(...): 新功能
  • fix(...): 修 bug
  • docs(...): 文档修改
  • test(...): 测试修改

步骤 11：把功能分支推到你自己的 fork 

1

git push -u origin feature/pixiu-saml-docs

命令解释：

  • git push origin <branch>

    • 把本地分支推到远程 origin

  • -u

    • 给当前本地分支设置“上游跟踪关系”
    • 设置后，后面你通常只需要：
1
2

git push
git pull

为什么你有时直接 git push 会报错：

  • 因为这个新分支第一次推送时，还没有和远程分支建立跟踪关系
  • 所以第一次通常要用：
1

git push -u origin <branch>

步骤 12：去 GitHub 页面发起 Pull Request

通常你 push 成功后：

  • 你自己的 fork 页面会出现 Compare & pull request
  • 或者 Open a pull request

这时候你要检查 PR 页面顶部四个位置：

  • base repository
  • base branch
  • head repository
  • compare branch

标准 fork 提 PR 应该像这样：

  • base repository: 官方仓库，例如 apache/dubbo-website
  • base: 官方主分支，例如 master
  • head repository: 你自己的 fork，例如 Skylm808/dubbo-website
  • compare: 你的功能分支，例如 feature/pixiu-saml-docs

这表示：

1
2
3

Skylm808/dubbo-website:feature/pixiu-saml-docs
  ->
apache/dubbo-website:master

也就是：

  • 改动来自你的 fork
  • 目标是官方仓库

4. 后续如果官方仓库更新了，应该怎么同步

这是开源协作里非常常见的情况。

例如：

  • 你已经 fork 了仓库
  • 也已经在本地写了一半
  • 但这期间官方仓库又有新提交

这时就需要同步。

场景 A：你只是想更新本地 master 

1
2
3
4

git fetch upstream master
git checkout master
git rebase upstream/master
git push origin master

作用：

  • 本地 master 跟上官方 master
  • 你 fork 上的 master 也跟上最新

这个流程适合：

  • 你准备从主分支重新开新功能分支

场景 B：你已经在功能分支开发了，还想把官方最新代码同步进当前分支

先确认你当前就在功能分支，例如：

1
2
3

git checkout feature/pixiu-saml-docs
git fetch upstream master
git rebase upstream/master

命令解释：

  • git checkout feature/pixiu-saml-docs

    • 切到你自己的开发分支

  • git fetch upstream master

    • 获取官方 master 最新内容

  • git rebase upstream/master

    • 让你当前分支的提交，重新排到最新官方主分支之后

如果你这个分支之前已经 push 过到远程，rebase 之后通常还要：

1

git push --force-with-lease origin feature/pixiu-saml-docs

为什么需要这个命令：

  • rebase 会改写提交历史
  • 远程分支和你本地分支的提交哈希会变得不一致
  • 普通 git push 往往会被拒绝

为什么推荐 --force-with-lease，而不是 --force：

  • --force-with-lease 更安全
  • 它会先检查远程分支是不是你预期的状态
  • 能减少误覆盖别人提交的风险

5. 如果 rebase 过程中遇到冲突怎么办

这也是常见情况，不用慌。

当你执行：

1

git rebase upstream/master

如果 Git 提示冲突，大致流程是：

第一步：看哪些文件冲突了 

1

git status

你会看到哪些文件处于冲突状态。

第二步：打开冲突文件，手动处理

冲突文件里会出现类似标记：

1
2
3
4
5

<<<<<<< HEAD
你的内容
=======
对方内容
>>>>>>> upstream/master

你需要：

  • 判断最终保留什么内容
  • 删除这些冲突标记
  • 保存文件

第三步：告诉 Git 这个文件已解决 

1

git add <冲突文件>

第四步：继续 rebase 

1

git rebase --continue

如果还有冲突，就重复上面的流程。

如果你不想继续了

可以放弃这次 rebase：

1

git rebase --abort

命令解释：

  • 回到 rebase 之前的状态

6. git fetch upstream 为什么有时会下载很多东西

这是你这次真实遇到过的问题。

原因通常有两种：

原因 1：你第一次拉这个远程仓库的信息

第一次 fetch 时，本地还没有这些对象和引用，所以量会大。

原因 2：你拉的是整个远程，而不是单独一个分支

例如：

1

git fetch upstream

有时会把 upstream 上很多分支的引用都拉下来。

对于很大的仓库，这可能比较重。

更省一点的方式是：

1

git fetch upstream master

这样更聚焦，只拉 master 相关信息。

7. fork 完是不是就永远不用配 upstream

不是。

如果你只是想自己单机改着玩，可以不配。

但如果你要长期参与开源项目，强烈建议配置 upstream，因为：

  • 你需要知道官方最新代码
  • 你需要把自己的分支同步到官方最新基线
  • 你需要减少 PR 冲突

所以规范流程里，upstream 基本是标准配置。

8. git rebase upstream/master 到底会改谁

它只会改：

  • 你当前所在的本地分支

例如你当前是：

1
2
3

git branch
* feature/pixiu-saml-docs
  master

这时执行：

1

git rebase upstream/master

被改写历史的是：

  • feature/pixiu-saml-docs

不是 master

所以执行前一定先看：

1

git branch

或者：

1

git status

确认自己现在到底在哪个分支。

9. 一次完整、规范的日常工作流示例

下面给一个最常用的模板。

假设：

  • 官方仓库：apache/dubbo-go-pixiu-samples
  • 你的 fork：Skylm808/dubbo-go-pixiu-samples
  • 你要新增一个 SAML sample

9.1 拉最新主分支 

1
2
3
4

git fetch upstream main
git checkout main
git rebase upstream/main
git push origin main

如果这个仓库主分支不是 main，就换成真实名字。

9.2 从最新主分支切功能分支 

1

git checkout -b feature/saml-filter-sample

9.3 开发、测试、检查改动 

1
2
3

git status
go test -v ./dubbogo/simple/saml/test
git diff

9.4 提交 

1
2
3
4

git add README.md
git add README_CN.md
git add dubbogo/simple/saml
git commit -m "feat(samples): add SAML authentication sample"

9.5 推送分支 

1

git push -u origin feature/saml-filter-sample

9.6 在 GitHub 上创建 PR

检查：

  • base repository = 官方仓库
  • base branch = 主分支
  • head repository = 你的 fork
  • compare branch = 你的功能分支

10. PR 发出后你还要做什么

PR 发出后，不是就结束了。

你通常还需要：

10.1 看 CI 是否通过

如果 PR 页面显示：

  • checks pending：还在跑
  • checks failed：有检查失败
  • checks passed：检查通过

如果失败，就点进去看日志。

10.2 回复 reviewer

如果 reviewer 提了修改意见：

  • 先在本地改
  • 测试
  • git add
  • git commit
  • git push

同一个 PR 会自动更新，不需要重新开一个 PR。

10.3 如果 reviewer 让你补 follow-up PR

这也是你这次实际碰到的情况。

例如：

  • 原始 PR 在 apache/dubbo-go-pixiu

  • 文档单独提到 dubbo-website

11. 什么时候用 merge，什么时候用 rebase

在给开源社区提 PR 时，通常更常见的是：

  • 同步官方最新代码时，优先使用 rebase

原因：

  • 提交历史更线性
  • PR 更干净
  • 不容易出现一堆无意义 merge commit

你可以这样简单理解

  • merge：把两条历史“并起来”
  • rebase：把你的提交“挪到最新基线后面”

对于个人 fork 开源协作场景，通常推荐：

  • 更新自己分支时用 rebase

13. 你这类开源参与里最推荐记住的操作原则

原则 1：永远不要直接在主分支上开发

主分支只用于：

  • 同步官方最新代码
  • 作为新功能分支的起点

原则 2：先测试，再提交，再推送

不要把“还没验证过”的东西直接 push 上去。

原则 3：PR 的目标看 base repository

这点特别重要。

真正决定 PR 提到哪里的，是：

  • base repository

不是你从哪个页面点进去的。

所以即使你是在自己 fork 页面看到 Open a pull request：

  • 只要最终 base repository 选的是官方仓库
  • 那这个 PR 就是提给官方仓库的

原则 4：同步官方更新时，先 fetch，再 rebase

推荐顺序：

1
2

git fetch upstream master
git rebase upstream/master

原则 5：第一次 push 新分支时，用 -u 

1

git push -u origin feature/xxx

这样后面会省很多事。

14. 一句话总结

对于你现在这种参与 Apache 项目的方式，最标准的流程就是：

  1. Fork 官方仓库到自己账号
  2. clone 自己的 fork 到本地
  3. 配置 upstream 指向官方仓库
  4. 先同步官方主分支最新代码
  5. 从最新主分支切功能分支
  6. 开发、测试、提交
  7. push 到自己的 fork
  8. 从自己的 fork 分支向官方仓库主分支发起 PR
  9. 根据 reviewer 意见继续修改并 push
  10. 必要时补 sample PR、website PR 等 follow-up PR

如果你把这一套流程熟了，以后参与大多数 GitHub 开源项目都够用了。
