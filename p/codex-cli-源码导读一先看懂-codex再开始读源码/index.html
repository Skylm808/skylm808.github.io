Codex CLI 源码导读（一）：先看懂 Codex，再开始读源码

在一个代码仓库里，真正棘手的地方常常不在某一行代码。约定散在 AGENTS.md、脚本、测试和 Git 历史里；有些修改能直接验证，有些命令却不该随便执行。一个编码 Agent 若想参与这类工作，光会生成代码远远不够。

codex 出现在终端里时，看上去像一段对话。它实际接住的是一处本地工作环境：当前目录、项目指令、可调用的工具，以及一套权限边界。模型决定下一步值得查什么、做什么；Codex CLI 负责把这个决定变成一次受控的本地操作，再把结果带回下一轮。

这是理解 Codex 的起点。本文不从某个 crate 开始，也不试图猜模型如何思考。先把任务怎样跑起来讲清楚。后面的源码阅读才有地图。

Codex 到底是什么

openai/codex 的 README 将 Codex CLI 描述为运行在本机上的编码 Agent。这里的“本机”决定了它和普通聊天窗口的区别。

在聊天窗口里，模型只能根据对话中已有的材料回答。它看不到你没有贴出的测试，也不能自己运行项目里的命令。CLI 工作在你启动它的目录中。它可以在授权范围内读取文件、搜索代码、运行测试，必要时写入修改。任务因此不止是一条建议，而是一个可以检查结果的过程。

把这件事拆开，只有三个角色：

  • 模型提出下一步，例如“读取这个测试”“搜索这个函数”“运行这个命令”。
  • 工具完成实际操作，并返回文件内容、命令输出或修改结果。
  • Codex 运行时管理两者之间的信息与边界：给模型哪些上下文，允许哪些工具动作，怎样保存任务进度。

“运行时”听起来有点抽象。这里可以把它理解成任务现场的协调者。它不替模型写结论，也不替 Shell 执行命令；它负责让模型的请求有机会落地，同时不越过项目和机器的边界。

一项任务需要六部分

如果把 Codex 看成“模型加一个终端”，很多源码位置都会显得莫名其妙。它实际要同时处理六类事情。

flowchart LR
    U["开发者"] --> E["入口\nCLI、IDE 或富客户端"]
    E --> C["任务与上下文\n问题、目录、指令、历史"]
    C --> A["Agent 循环\n决定下一步"]
    A <--> M["模型连接\n请求与响应"]
    A <--> T["工具\n读、搜、改、执行"]
    T --> W["本地工作区"]
    P["权限与沙箱"] -.约束.-> T

图里没有画一个叫“智能”的方框，因为智能不是单独的模块。任务的效果来自这六部分不断配合。先逐个认识它们。

1. 入口：人从哪里发起任务

CLI 是终端优先的入口。你在仓库目录中输入 codex，它把输入、配置和当前目录交给后面的运行时。

Codex 还有其他入口。README 把 codex app 列为桌面端体验，也提供 IDE 使用方式。它们改变的是人如何提交任务、如何观看过程，不会改变一个编码任务仍要面对上下文、工具和权限的事实。

本系列聚焦公开源码里的 CLI 本地执行路径。app-server 会在后文单独出现：它是面向富客户端的本地服务接口，公开 README 以 Codex VS Code 扩展为例。它不等于 Codex App 的全部实现，也不应被简单叫成“桌面端功能”。

2. 任务与上下文：模型眼前有什么

“修复登录测试”这句话太短。模型还需要知道自己在哪个目录，项目有哪些约定，前一轮刚读到了什么，命令执行后发生了什么。

因此，一次任务的上下文会逐步增长：用户的要求、当前工作区、项目指令、已读取的文件、工具输出和会话历史都会加入。与此同时，运行时还得控制上下文的体积。把整个仓库一次性塞进去既不可行，也不一定有用。

这一部分决定了模型此刻能据什么作判断。它不会替模型保证结论正确，但会决定模型能看到多少相关证据。

3. Agent 循环：怎样从一句话走到下一步

模型的回答不总是最终答复。它也可能请求一次搜索、一次文件读取，或者一次命令执行。运行时收到请求后调用工具，把结果写回当前任务，再让模型继续。

这个循环就是这里所说的 Agent。它没有神秘之处：选择动作，执行动作，把观察结果带回下一轮。一次小任务可能只转两三轮；排查问题时，循环会持续到证据足够、改动完成，或用户要求停止。

4. 模型连接：请求怎样出去，结果怎样回来

在 Agent 循环里，运行时要把任务和上下文组织成请求，接收模型的流式输出、工具请求和错误信息。模型是重要的一环，但不是整个系统。

从源码可以观察请求怎样准备、响应怎样处理、失败怎样回到任务流。看不到的部分也要划清边界：公开客户端代码不能还原模型的私有推理过程。后文会分析可验证的控制流，不把猜测写成源码结论。

5. 工具：让“去做”成为事实

读取文件、全文搜索、执行测试、应用补丁、查看 Git 状态，这些都是工具。模型说“检查失败测试”时，真正读取文件的是工具；模型说“运行测试”时，真正启动进程的也是工具。

工具返回的内容不只是给人看的日志。它是下一轮上下文的一部分。测试失败的输出会改变下一步搜索方向，补丁是否应用成功也会影响任务能否继续。这条来回路径比“模型调用了函数”更接近真实的编码工作。

6. 权限与沙箱：哪些动作现在可以发生

能执行命令不等于应该执行命令。读取仓库里的一个测试文件，和向工作区外写文件、联网安装依赖，不是同一种风险。

Codex 会把权限策略和沙箱放在工具路径旁边。它们决定可写范围、命令怎样运行，以及何时需要确认或升级。这个设计不是给 Agent 外面再套一层说明书，而是决定工具请求最终是否能落地。没有这一层，模型生成的一条命令就可能直接碰到不该碰的地方。

用一次测试修复把它们串起来

假设仓库里有一个失败的测试。用户的任务很简单：找出失败原因，修复它，并运行相关测试。开始时，Codex 并不知道问题在哪。

sequenceDiagram
    participant D as 开发者
    participant R as Codex 运行时
    participant M as 模型
    participant W as 工作区与工具

    D->>R: 修复失败测试
    R->>M: 任务、目录、项目指令
    M->>R: 请求读取测试并搜索相关代码
    R->>W: 执行读取与搜索
    W-->>R: 测试内容、代码位置
    R->>M: 返回观察结果
    M->>R: 请求修改并运行目标测试
    R->>W: 在权限允许的范围内执行
    W-->>R: 修改结果、测试输出
    R->>M: 返回验证结果
    R-->>D: 说明改动与测试状态

这张图有两个容易被忽略的点。

第一，模型通常不会先看遍整个仓库，再一次性给出答案。它根据当前证据选择下一步，工具结果再改变后续选择。第二，模型的请求没有绕过运行时直接落到工作区。工具调用和安全检查都在中间，这也是为什么 Codex 的源码不只有一个 API 客户端。

CLI、App、IDE：先分清入口，再谈源码

读 README 时容易把几个名字混在一起。它们服务的使用场景不同。

名称 面向的使用方式 本系列中的位置
Codex CLI 在终端和本地仓库里工作 主线
Codex App 桌面端任务体验 只说明产品边界，不分析私有实现
IDE 集成 在编辑器中使用 Codex 用来理解富客户端需求
app-server 富客户端可使用的本地协议服务 后文分析公开接口与事件流

把 app-server 和 App 分开，后面会轻松很多。前者是源码中的一个服务接口；后者是用户看到的产品入口。名称相近，不表示二者是一一对应的实现关系。

现在再看目录名

理解任务流程后，再打开仓库。根目录下的 codex-rs 是主要 Rust workspace，可以把它看成多个 Rust crate 组成的大工程。第一轮只需要记住五个路标：

目录 先这样理解 后面在哪篇读
cli 命令行入口 第二篇
tui 终端中的交互界面 第二篇、第九篇
app-server 面向富客户端的本地服务接口 第二篇、第九篇
core 任务运行时的主要协调逻辑 第三篇起
tools 工具定义与调度的一部分 第五篇

目录不是架构图。一个模块可能参与多个职责，一项职责也可能散在多个 crate 中。以后读源码时，先问“这个任务步骤在哪儿发生”，再去找目录；不要看到 core 就把所有 Agent 逻辑都塞进一个概念里。

MCP、Skill、Plugin、Hook 和 Memory 也在仓库中，但它们解决的是扩展与定制问题。主任务循环还没看清之前，先不急着背这些名字。第七篇和第八篇会把它们拆开。

带着这些问题进入下一篇

  • 用户输入的一段任务文本，最早由哪个入口接住？
  • 一条任务怎样从“已创建”变成一串可以显示给界面的事件？
  • 终端里的 tui 为什么不是桌面 App？
  • 富客户端通过 app-server 提交任务时，和 CLI 共享了哪些概念？
