阅读时长: 5 分钟

Codex CLI 源码导读（一）：先看懂 Codex，再开始读源码

假设你在一个仓库里看到失败测试，想让 Codex 帮忙处理。你给出的要求可能只有一句：读一读这个失败，找到原因，修改后运行相关测试。

这句话不会直接变成一段代码。Codex 要先看见仓库里的文件和项目约定，再决定读什么、搜什么、是否修改，最后还要用测试确认结果。理解 Codex 的第一步，不是记住 Rust 目录，而是看清这条任务链上每个人和每个组件各做什么。

本文固定使用 openai/codex 的 268288a 版本。公开源码能说明任务如何流动、工具如何执行；它不能还原模型内部的思考过程。

Codex 是什么

Codex CLI 是运行在本机代码仓库中的编码 Agent。这里的“本机”很重要：它工作的材料不只有聊天记录，还包括当前目录、项目文件、测试命令和可用权限。

先把四个角色分开：

角色 它负责什么
开发者 给出目标，决定是否接受改动
模型 根据已经拿到的材料，选择下一步该做什么
Codex 运行时 组织任务、调用模型、执行工具，并检查边界
本地工作区 提供文件、命令、测试结果和最终改动

模型可以提出“读取这个测试”或“运行这个命令”。真正读取文件和启动命令的是 Codex 的工具。运行时夹在中间，负责把结果带回模型，也负责决定这项操作在当前权限下能不能发生。

一句话概括：模型决定下一步，Codex 负责让这一步在本地可控地发生。

一项任务怎样走完

先不看任何源码名，只看一次普通任务的过程。

flowchart LR
    U["开发者提出目标"] --> C["Codex 记录任务和已知信息"]
    C --> M["模型选择下一步"]
    M --> T["工具读取、搜索、修改或执行命令"]
    T --> W["本地工作区"]
    W --> C
    C --> R["向开发者汇报结果"]
    P["权限与沙箱"] -.限制工具操作.-> T

这张图从左到右可以这样读：

  1. 开发者说明想做什么，例如“找出这个测试为何失败”。
  2. Codex 收集眼前需要的信息，例如当前目录、项目指令和先前结果。
  3. 模型选择一个小动作，例如读测试文件或搜索函数名。
  4. 工具在本地完成动作，把输出交回 Codex。
  5. 新结果成为下一次选择的依据，直到任务完成、无法继续，或开发者叫停。

这就是 Agent 的基本含义：它不是一次性生成答案，而是在“决定、执行、观察”之间反复往返。一次简单的问题也许只需要两轮；排查问题时，轮数会更多。

权限与沙箱也在这条链上。读一个文件和向仓库外写文件的风险不同，运行时必须在工具真正执行前判断边界。第六篇会专门拆开这一层。

六个部分，六个问题

为了读源码方便，可以把这条链分成六个部分。它们不是六个彼此隔离的程序，只是六个不同的问题。

部分 它要解决的问题
入口 开发者从哪里提交任务？
任务与上下文 模型此刻能看到哪些材料？
Agent 循环 一次工具结果怎样引出下一步？
模型通信 任务怎样发给模型，回答怎样回来？
工具 怎样真正读、搜、改和运行命令？
安全边界 哪些操作可以直接做，哪些需要限制或确认？

后面的九篇会一层层回答这些问题。现在只需要记住，它们共同让一句自然语言任务变成可检查的本地工作。

CLI、App 和 IDE 是什么关系

它们都是使用 Codex 的入口，区别在于人从哪里发起任务、在哪里看过程。

名称 可以先这样理解
Codex CLI 在终端中启动 Codex 的命令行程序，本系列的主线
TUI 终端里显示出来的交互界面，例如消息、进度和确认提示
IDE 集成 编辑器里的 Codex 入口
Codex App 桌面端产品入口；公开 CLI 仓库不是它的全部实现

还有一个很容易混淆的名字：app-server。它不是 Codex App。它是公开 CLI 源码中的一个本地服务：其他界面，例如终端界面或编辑器，可以把任务交给它，再从它那里接收进度。可以先把它想成“界面和任务运行时之间的接线员”。第二篇会沿着这条线解释它。

现在再看目录名

理解了任务链后，codex-rs 里的目录才有位置可放。codex-rs 是主要 Rust workspace，可以理解为一个统一构建的 Rust 工程集合。里面可独立编译、声明自己依赖关系的模块叫 crate；不熟悉 Rust 时，可以近似看作 Go 项目中的一个包或程序。它通常对应一个目录和一份 Cargo.toml，但“目录”与 crate 不是完全等同的概念。

第一轮只需要记住下面五个路标：

目录 它在任务链中的位置
cli 接住终端命令的入口
tui 终端交互界面
app-server 给其他界面使用的本地服务
core 协调任务、模型和工具的主要运行时
tools 读取、搜索、执行等工具能力

目录名不是完整的架构图。一个目录可能服务多件事，一件事也可能跨多个目录。读源码时，先问“这个步骤在任务链的哪里发生”，再去找对应目录。

MCP、Skill、Plugin、Hook 和 Memory 都属于扩展与定制能力。它们会在第七、八篇单独解释；刚开始不必把它们和主任务链混在一起。

  • 如果模型不能直接碰文件和 Shell，工具结果是怎样回到它手里的？
  • 当前目录、项目指令和已经读过的文件，为什么都会影响下一步？
  • 为什么“能运行命令”还不足以说明一个编码 Agent 可以安全工作？
