Harness

Harness

Apr 06, 2026
阅读时长: 8 分钟

一、这篇文章到底在讲什么

OpenAI 在 2026 年 2 月 11 日发布了一篇文章：《Harness engineering: leveraging Codex in an agent-first world》。

文章的核心不是“AI 会写代码”这么简单，而是在讨论一个更本质的问题：

  • 当工程团队的主要工作，不再是手写代码，而是设计环境、定义约束、提供上下文、构建反馈回路时，软件工程会变成什么样？

OpenAI 在文中给出的答案可以浓缩成一句话：

  • 人类负责掌舵，Agent 负责执行；工程效率的关键，不再只是代码能力，而是 harness 能力。

这里的 harness，可以理解成一整套“让 Agent 稳定产出结果的工程支架系统”。

这篇文章里提到，他们用一个很小的工程团队，驱动 Codex 在一个全新仓库里持续开发，形成了大约百万行代码、上千个 PR，而且强调：

  • 不是为了刷产出；
  • 产品真的被内部用户和外部测试者使用；
  • 人类几乎不直接写代码；
  • 真正重要的是如何让 Agent 持续、稳定、可控地工作。

所以，这篇文章的本质，是在提出一种新的工程范式：

  • 从 code-first software engineering，转向 agent-first software engineering。

二、什么是 Harness Engineering

严格说，OpenAI 在原文里并没有给出一个像教科书那样的标准定义。下面这个定义是我基于原文总结出来的工程化表述：

  • Harness Engineering：为了让 AI Agent 能够在真实软件项目中稳定完成设计、编码、测试、验证、修复、评审、合并等工作，而专门构建的一套环境、规则、工具、知识库和反馈回路体系。

它不是单一工具，也不是单一 prompt，而是一套系统工程。

你可以把它理解成：

1

Agent 能力 = 模型能力 × 工程环境质量

如果模型很强，但工程环境很差，Agent 依然会失控、误解、漂移、产出不稳定。

所以 Harness Engineering 的目标，不是“让 AI 更聪明”，而是：

  • 让 Agent 看得见
  • 让 Agent 走得对
  • 让 Agent 错了能被发现
  • 让 Agent 修得回来
  • 让 Agent 长期不烂

这就是文章的主线。

三、这篇文章最核心的 8 个观点

3.1 工程师的角色变了

过去工程师主要自己写代码。

在 agent-first 世界里，工程师的核心工作逐渐变成：

  • 定义任务目标；
  • 拆分任务；
  • 设计仓库结构；
  • 提供足够上下文；
  • 编码约束和规范；
  • 设计验证链路；
  • 处理需要人类判断的少数高价值决策。

所以工程师的价值并没有下降，而是从“直接产出代码”转向“设计一个能稳定产出代码的系统”。

一句适合面试的总结：

  • 未来高级工程师的核心竞争力，不只是写代码快，而是能不能把团队经验、架构规则和质量标准沉淀成可执行系统。

3.2 问题失败时，不是让 Agent“再试一次”，而是补能力

文中一个很关键的思想是：

  • 当 Agent 失败时，正确的修复方式通常不是 “try harder”；
  • 而是反问：到底缺了什么能力、上下文、工具或约束？

这是一种非常工程化的思路。

也就是说，Agent 失败的根因常常不是“模型笨”，而是：

  • 仓库知识不足；
  • 规则不可见；
  • 依赖边界不明确；
  • 测试不充分；
  • 工具链没接通；
  • 反馈信号不够直接。

3.3 仓库不只是代码仓库，而是知识系统

OpenAI 明确反对把所有规则都堆进一个超长的 AGENTS.md。

原因很现实：

  • 上下文窗口是稀缺资源；
  • 大而全的说明书会稀释重点；
  • 文档容易腐化；
  • 难以校验其新鲜度和正确性。

所以他们的做法是：

  • AGENTS.md 只做目录和导航；
  • 真正的知识系统放在仓库里的 docs/；
  • 设计文档、执行计划、产品规格、质量评分、可靠性规范、安全规范都版本化。

这背后的思想是：

  • Repository is the system of record.

也就是：

  • 对 Agent 来说，仓库才是事实来源；
  • 不在仓库里的知识，等价于不存在。

3.4 Agent Legibility 比 Human Elegance 更重要

原文有一句非常重要的意思：

  • 对一个完全由 Agent 生成和维护的代码库来说，首要目标是让 Agent 能读懂，而不是先追求人类审美上的“优雅”。

这里的 legibility 可以翻译成：

  • 可读性；
  • 可理解性；
  • 可导航性；
  • 可推理性。

换句话说，代码和文档不是写给“抽象的人类工程师”看的，而是首先写给“未来还会继续操作这个仓库的 Agent”看的。

所以他们偏向：

  • 稳定的结构；
  • 明确的边界；
  • 可推理的依赖关系；
  • 可验证的规则；
  • 尽量少黑盒。

这也是为什么文章里提到：

  • 有时候宁可自己实现一小段受控功能；
  • 也不愿意引入一个对 Agent 来说过于黑盒、行为不透明的外部库。

3.5 让应用本身也对 Agent 可读

这点特别重要，很多人会忽略。

OpenAI 不是只让 Agent 读代码，而是让 Agent 能直接观察：

  • UI；
  • DOM；
  • 截图；
  • 页面导航；
  • 日志；
  • 指标；
  • Trace。

也就是说，他们把“应用运行时世界”暴露给 Agent。

这样 Agent 就不只是会改代码，而是能：

  • 自己复现 bug；
  • 自己验证修复；
  • 自己检查性能指标；
  • 自己完成回归验证。

这是从“代码生成器”走向“工程执行体”的关键一步。

3.6 架构约束越早编码，Agent 速度越快

传统团队常常会觉得：

  • 早期先快写，后面再治理架构。

但文章给出的经验恰好相反：

  • 对 Agent 来说，强约束不是减速器，而是加速器。

因为 Agent 最怕的是：

  • 边界模糊；
  • 依赖随意；
  • 模式不统一；
  • 文件风格漂移；
  • 质量标准口头化。

所以 OpenAI 选择：

  • 固定分层；
  • 固定依赖方向；
  • 用自定义 lint 和结构测试强制执行；
  • 把错误信息本身写成“带修复指导的反馈”。

这其实已经不是传统意义上的代码规范，而是在做“Agent 编程环境设计”。

3.7 高吞吐下，合并哲学也要变

文章里有一个很反直觉但很现实的观点：

  • 当 Agent 的产出速度远远快于人类注意力时，很多传统的重型阻塞式流程反而会拖累整体效率。

比如：

  • PR 尽量短命；
  • Merge gate 尽量少阻塞；
  • flaky test 不一定要无限期卡住主流程；
  • 便宜的错误更适合快速修复，而不是长时间等待。

这并不是说“质量不重要”，而是说：

  • 在高吞吐系统里，等待本身就是成本；
  • 如果纠错成本很低，快速前进可能比过度阻塞更优。

但这有前提：

  • 自动验证链路要足够强；
  • 回滚和修复能力要足够快；
  • 系统可观察性要足够好。

3.8 自治越高，越需要持续垃圾回收

文章最后提到一个很接地气的问题：

  • Agent 会复制已有模式；
  • 一旦仓库里出现坏模式，它会被持续放大。

这就是所谓的代码熵增。

OpenAI 的处理方式是：

  • 定义 golden principles；
  • 持续后台扫描偏差；
  • 自动更新质量评分；
  • 自动发起小型重构 PR；
  • 把技术债当成“高利贷”，持续小步偿还。

这特别像垃圾回收：

  • 不是等系统烂到无法维护再集中治理；
  • 而是持续、小步、机械化清理。

四、文章里的关键术语，面试必须讲清楚

4.1 Agent-First Engineering

含义

  • 把 AI Agent 当成软件交付主力之一来设计工程流程，而不是把它当成一个临时辅助工具。

核心变化

  • 不是“我写代码，AI 帮我补全”；
  • 而是“我定义目标和约束，Agent 负责执行和迭代”。

面试怎么说

  • Agent-first engineering 的核心，不是用了 AI，而是工程流程从面向人类执行优化，转向面向人类决策 + Agent 执行优化。

4.2 Harness

含义

  • 原意像“马具、控制装置、束具”。
  • 在工程语境里，可以理解成“把能力约束并导向正确输出的一套支架系统”。

在本文中的具体指代

  • 仓库结构；
  • 知识文档；
  • 工具链；
  • lint；
  • CI；
  • 测试；
  • 评审流；
  • 可观察性；
  • 运行时验证能力。

面试怎么说

  • Harness 不是一个 prompt，而是让 Agent 在真实项目里可控工作的整体工程基础设施。

4.3 Agent Legibility

含义

  • 让 Agent 易于理解系统、定位信息、推理边界、做出正确修改。

它关注什么

  • 文档是否在仓库内；
  • 信息是否结构化；
  • 依赖关系是否清晰；
  • 规则是否可执行；
  • 运行反馈是否可观测。

为什么重要

  • Agent 不是靠“感觉”工作，它依赖上下文和可见信号。
  • 看不见的信息，对 Agent 来说就等价于不存在。

面试怎么说

  • 人类面对不清晰系统时还能靠经验兜底，但 Agent 更依赖可见、结构化、可验证的信息，因此 legibility 是 agent-first 时代的核心工程指标。

4.4 System of Record

含义

  • 某类信息的权威事实来源。

在这篇文章里

  • 仓库就是 system of record；
  • 文档、计划、规则、质量状态必须回到仓库并版本化。

面试怎么说

  • 如果知识只存在于会议、聊天、脑子里，那对 Agent 来说不可用；只有进入仓库并版本化，才能真正参与自动化执行。

4.5 Progressive Disclosure

含义

  • 渐进式暴露信息。

具体做法

  • 先给 Agent 一个短入口；
  • 再通过目录、索引、链接引导它逐层深入；
  • 而不是一开始塞一个几千行总说明书。

好处

  • 节省上下文；
  • 降低噪声；
  • 提高查找精度；
  • 减少过度提示造成的误导。

4.6 Structural Invariants

含义

  • 结构性不变量，也就是无论谁改代码都必须满足的底层规则。

例子

  • 分层方向不能反；
  • 数据校验必须发生在边界；
  • 日志必须结构化；
  • 文件不能无限膨胀；
  • 某些类型命名必须统一。

面试怎么说

  • 高级工程治理的核心，不是反复口头提醒，而是把关键不变量编码成 lint、测试和 CI，让系统自动拒绝错误形态。

4.7 Taste Invariants

含义

  • 团队长期形成的风格偏好和可维护性偏好，被进一步抽象成可以自动执行的规则。

它和普通 code style 的区别

  • 普通 code style 更偏格式；
  • taste invariants 更偏长期维护哲学。

例子

  • 偏向共享工具而不是随手造 helper；
  • 偏向显式 schema 校验，而不是 YOLO 式猜数据结构；
  • 偏向稳定、透明、可推理的依赖。

4.8 Entropy / Garbage Collection

含义

  • Entropy 指系统会自然走向混乱和漂移；
  • Garbage collection 在文中是一个类比，指持续清理 Agent 产生的坏模式和技术债。

为什么会发生

  • Agent 会模仿仓库里已存在的模式；
  • 一旦旧模式不好，它会被高频复制。

面试怎么说

  • AI 时代的技术债传播速度比人工时代更快，所以治理必须从阶段性大扫除，变成持续性自动回收。

五、OpenAI 这套方法论，落到工程里到底长什么样

需求/问题
-> 人类把目标写成 prompt 或验收标准
-> Agent 读取仓库内文档、架构图、计划、规则
-> Agent 修改代码/测试/CI/文档
-> Agent 启动应用并驱动 UI
-> Agent 查询日志/指标/Trace
-> Agent 验证结果是否达标
-> Agent 发 PR 并响应反馈
-> 系统通过 lint/CI/结构测试拦截错误
-> 后台清理任务持续修复坏模式和技术债

这个流程说明：

  • Agent 并不是在“盲写代码”；
  • 它是在一个高度工程化的闭环里工作。

所以真正的门槛并不只是模型 API 接入，而是你是否有能力把下面这些东西系统化：

  • 需求表达；
  • 文档沉淀；
  • 规则编码；
  • 自动验证；
  • 反馈回路；
  • 漂移治理。

六、为什么这篇文章对面试很重要

因为它回答了一个现在越来越常见的问题：

  • AI 会不会取代工程师？

更成熟的回答不是情绪化地说“不会”，而是说：

  • AI 会重塑工程师的工作重心；
  • 低水平重复编码的价值会下降；
  • 高水平的系统设计、规则抽象、质量治理、反馈闭环设计会更重要。

如果你能讲出下面这层意思，面试官通常会觉得你不是在追热点，而是真的理解了工程演进：

  • **在 agent-first 时代，工程能力从“实现某个功能”上移到“构建一个可持续产出正确功能的系统”。
