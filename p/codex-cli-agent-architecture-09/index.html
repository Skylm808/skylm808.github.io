Codex CLI 源码导读（十）：任务进度怎样送到终端和富客户端

让 Codex 在仓库里找一个调用点、改一处文件、再跑一条测试时，界面不会等到任务结束才给出一整段结果。你会先看到任务开始，接着看到命令、文件改动和文字回复，最后看到完成或中断。支撑这个体验的是一串带有任务标识和状态的结构化事件。

本文的判断很简单：运行时负责产生事实，界面负责呈现事实；两者之间由 app-server 的本地协议连接。app-server 不是桌面版 Codex App 本身。它是公开 CLI 中的本地服务，供 TUI 和 VS Code 扩展这类富客户端使用。桌面 App 可以使用类似能力，但其完整实现不在这份公开仓库中。

本文固定使用 openai/codex 的 268288a。第十篇只讨论公开源码里事件如何流动，不把某个界面的具体设计当作协议承诺。

先把“进度”看成状态，不是日志

日志是给人或诊断工具看的文本，常常只说明“发生过什么”。结构化事件则会明确说明：哪一个 Thread、哪一轮 Turn、哪一个 Item 发生了什么变化。

这里的三个名词来自第二、三篇：Thread 是一段持续的任务会话；Turn 是用户提交一次请求后的一轮执行；Item 是这一轮里一项可记录的输入或输出，例如用户消息、模型回复、命令执行和文件改动。

假设客户端提交“运行 go test ./...，修复第一个失败的测试”。它关心的不只是终端里出现了哪些字符，还需要知道下面几件事：

看到的变化 对应的状态含义
任务开始转圈 某个 Turn 已进入执行中状态
命令卡片出现 一个命令执行 Item 已开始
终端内容逐步增加 这个 Item 正在产生输出增量
补丁或文件卡片完成 一个文件改动 Item 已有最终结果
任务结束 Turn 已完成、失败或被中断

“增量”是一次消息中新增的一小段内容。例如命令输出可以一行一行到达，模型回复也可以逐段出现。界面将这些增量追加到已有 Item 上，最终再以 item/completed 收束它。

sequenceDiagram
    participant C as TUI 或富客户端
    participant A as app-server
    participant R as Core 运行时

    C->>A: turn/start（用户输入）
    A-->>C: 响应：Turn 已创建
    A->>R: 提交本轮任务
    R-->>A: TurnStarted、ItemStarted、输出增量、ItemCompleted
    A-->>C: turn/started、item/*、各类增量通知
    R-->>A: TurnComplete 或 TurnAborted
    A-->>C: turn/completed

图里的“响应”和“通知”不同。响应是客户端发起请求后得到的回答；通知是服务端主动推送的一条单向消息，不需要客户端逐条回复。这样，turn/start 可以很快返回一个 Turn，而耗时的工作继续通过通知发送给界面。

App Server：本地协议服务，不是另一个 Agent

codex app-server 在 codex-rs/app-server 中实现。它用 JSON-RPC 2.0 组织请求、响应和通知。JSON-RPC 是一种以 JSON 表达消息的协议约定：方法名决定要做什么，参数描述输入，请求标识把响应对应回原始请求。

默认连接方式是标准输入输出上的 JSONL。JSONL 指每行一个 JSON 对象，因此一边可以连续写入请求，另一边也可以连续读到通知。公开源码还提供 WebSocket 和 Unix socket 等连接方式；其中 WebSocket 被标注为实验性接口，不能把它当作稳定集成边界。

客户端先发送 initialize，再发 initialized，随后才能操作 Thread。创建会话用 thread/start，继续已有会话用 thread/resume；提交用户输入用 turn/start。turn/start 的响应只表示这轮任务已经建立，真正开始执行时会再收到 turn/started，随后才是 Item 和增量通知。

app-server 的职责不是替模型决定下一步。它把运行时已经产生的状态变成客户端能稳定消费的消息，并维护客户端对哪些 Thread 订阅了进度。thread/start、thread/resume 和 thread/fork 会让当前连接订阅相应 Thread；客户端也能通过 thread/unsubscribe 停止接收该 Thread 的 Turn 和 Item 事件。

从 Core 事件到协议通知

第三篇中的 Core 运行时使用 EventMsg 表示任务内部发生的事情。codex-rs/app-server/src/bespoke_event_handling.rs 是一次重要的转换点：它接收带有 Turn 标识的 Core 事件，补上 Thread 信息，再发出 ServerNotification。

最常见的映射可以这样理解：

Core 中发生的事 客户端收到的消息 界面通常怎么用
Turn 开始 turn/started 设置运行中状态
一个 Item 开始 item/started 新建命令、文件改动或消息区域
模型文字到达 item/agentMessage/delta 追加模型回复
命令输出到达 命令输出增量通知 追加终端输出
一个 Item 结束 item/completed 固化结果和最终状态
Turn 完成或中断 turn/completed 结束加载，显示最终结果

codex-rs/app-server-protocol/src/protocol/v2/item.rs 为 item/started 和 item/completed 定义了相同的定位骨架：thread_id、turn_id、item，再加上开始或完成的时间。客户端因此不必从一行文本猜测“这是谁的输出”。

这里还有一个容易读错源码的地方。Core 中仍保留了较早的 ExecCommandBegin、ExecCommandEnd、PatchApplyBegin、PatchApplyEnd 事件，用于兼容原始事件和历史记录。app-server 的 v2 客户端不依赖它们来构建界面，而是使用统一的命令执行 Item 和文件改动 Item 的开始、完成生命周期。这样客户端面对命令和补丁时遵循同一套数据形状。

权限请求也不是普通的进度通知。需要用户决定时，app-server 会发送一个 Server Request，客户端必须给出允许、拒绝或其他响应。第六篇里的审批边界因此仍然保留在协议上：界面不能只把确认框当成一条可忽略的日志。

TUI 怎样消费这些事件

tui 是终端里的交互界面。它并不直接观察 Core 的所有内部事件，而是作为 app-server 客户端消费 AppServerEvent。codex-rs/tui/src/app/app_server_events.rs 先处理应用范围的通知，再按照 thread_id 把其余消息送到正确的 Thread。

这一步很实际。一个 TUI 可能同时显示主任务、子任务和旁路会话。若一个子任务的 item/completed 被误送到主任务，命令输出和改动记录就会混在一起。codex-rs/tui/src/app/app_server_event_targets.rs 把通知中的 Thread 标识取出来，thread_routing.rs 再决定是立刻交给当前会话，还是先保存起来。

flowchart LR
    A["app-server 通知"] --> B{"带有哪个 thread_id？"}
    B --> C["当前正在看的 Thread\n立刻更新 ChatWidget"]
    B --> D["未激活的 Thread\n放入事件缓冲区"]
    D --> E["用户切换回来\n重放仍有意义的事件"]

codex-rs/tui/src/app/thread_events.rs 中的 ThreadEventStore 保存每个 Thread 的会话快照、Turn、缓冲事件和当前输入状态。它会在 turn/started 记下活动 Turn，在对应的 turn/completed 清除活动状态。体积较大、不会用于重放的消息，例如原始响应内容或实时音频增量，不会复制进每个 Thread 的缓冲区。

这也说明“实时”并不等于“绝不丢失每一条展示消息”。当 TUI 消费事件落后时，源码会记录延迟并丢弃明确不影响后续状态的事件，同时刷新必要的状态。客户端要以 Thread、Turn 和 Item 的最终状态为准，不能把每个增量都当成唯一可靠的历史来源。

再回到 CLI、扩展和 App

终端交互模式下，TUI 是一个 app-server 客户端；VS Code 扩展等富客户端也可以通过同一协议驱动 Thread 和 Turn。它们可以有不同的布局、快捷键和展示方式，却不必各自实现 Agent 循环、工具执行和审批流程。

这正是 app-server 的位置：它把“如何执行任务”和“怎样看见任务”隔开。公开仓库能让我们观察到 TUI 与本地服务的边界，也能看到该服务面向富客户端的协议。至于 Codex App 自身的全部界面和产品逻辑，不能从这些 CLI 源码中推导出来。
