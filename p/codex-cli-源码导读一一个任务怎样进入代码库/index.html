阅读时长: 6 分钟

Codex CLI 源码导读（一）：一个任务怎样进入代码库

假设你正在维护一个 Go 服务。线上偶发空指针，测试还没有覆盖这个分支。你进入仓库根目录，输入：

1

codex "找出订单接口的空指针来源，修复它，并补上回归测试"

这看上去像把一句话交给模型，但真正开始工作的，是一个运行在本机、知道自己身处哪个代码库的程序。它要先找到项目指令和当前会话，决定哪些文件可以读取，执行测试，再把测试输出带回下一轮。模型提出下一步；Codex 让这一步在仓库里发生。

这就是读 Codex 源码时最该抓住的主线：把 Codex 看成一套运行时更准确。它让模型在真实工程环境里观察、行动和校验。

本文不从 core 目录或一堆 Rust crate 开始。先认识 Codex CLI，再看它和 App 的关系，最后沿着这条任务把整体结构走一遍。目录名会出现，但它们是路标，不是正文。

Codex CLI：任务发生在本地

openai/codex 的 README 将它定位为运行在本机的 Codex CLI。这个“本机”并不是一个无关紧要的部署细节。

在普通聊天窗口里，模型只能根据你贴进去的内容回答。CLI 则站在当前工作目录中。它可以查看 Git 状态、读取项目文件、运行已有测试，并在权限允许时写入修改。于是，“修一个空指针”不再是一段建议，而是一段可以被验证的工作。

不过，模型并不会天然看到整个仓库。一次任务通常是逐步展开的：先搜索 OrderService，再读取相关实现与测试，然后运行目标测试。每次工具返回的内容，才会进入下一轮决策。

sequenceDiagram
    participant D as 开发者
    participant C as Codex 本地运行时
    participant M as 模型
    participant R as 代码库与工具

    D->>C: 修复订单接口的空指针
    C->>M: 任务、项目指令和当前上下文
    M->>C: 搜索并读取相关代码
    C->>R: 搜索文件、读取实现和测试
    R-->>C: 代码与命令输出
    C->>M: 新的观察结果
    M->>C: 修改代码并运行测试
    C->>R: 应用修改，执行测试
    R-->>C: 测试结果
    C-->>D: 修改与验证结果

图中的 Codex 是关键。它夹在模型和工作区之间，既负责传递信息，也负责控制动作。没有这一层，模型只能给出建议；有了它，模型才可以参与一次完整的软件任务。

CLI 和 Codex App：两个入口，各自分工

第一次看到 codex app 时，我也会产生一个疑问：既然仓库主要叫 Codex CLI，为什么又有 App？

答案不复杂。CLI 和 App 是使用 Codex 的两个入口。

CLI 以终端为中心。开发者在代码库里发起任务，直接看到命令、文件改动和测试输出。它的节奏和日常开发很接近。

App 以桌面界面为中心。它更适合查看任务、过程与结果。当前 README 也将 codex app 列为桌面端体验的启动方式，仓库因此也包含通向桌面体验的入口。

两者真正共用的重点，是本地任务运行时：会话怎样建立，模型怎样获得上下文，工具怎样执行，哪些动作需要被限制。界面层决定人怎样发起和查看任务；运行时决定任务怎样进入代码库。

入口 用户主要看到什么 本文关心什么
CLI 命令、终端对话、文件改动、测试输出 任务如何在本地仓库中运行
App 任务视图、执行过程、结果 与本地运行时相连的部分

后面的文章会一直坚持这个边界。本文只研究 Codex 的本地 Agent 运行时；桌面产品和云端服务的内部实现不在讨论范围内。

一次任务经过哪些部分

回到开头的空指针问题。为了完成这件小事，Codex 不只需要一个模型接口。它至少要处理五类工作。

    I["入口\nCLI、终端界面、App 连接"] --> S["会话与上下文\n任务、项目指令、历史、配置"]
    S --> A["Agent 核心\n组织模型响应与任务推进"]
    A <--> M["模型与后端\n发送上下文，接收响应"]
    A <--> T["工具运行时\n读文件、搜索、改代码、执行命令"]
    T --> W["本地工作区"]
    G["安全边界\nSandbox、执行策略、权限升级"] -.约束.-> T

入口：把人的意图变成一个任务

在 codex-rs 工作区里，cli、tui 和 app-server 分别对应命令行、终端交互和面向其他客户端的服务入口。它们接住任务，并把执行过程呈现给用户。

这也是 CLI 与 App 的差异最容易落到源码里的地方：它们的交互可以不同，但都需要把一个明确的任务交给后面的运行时。

会话与上下文：决定模型此刻知道什么

项目理解必须由运行时逐步拼装。任务文本、当前目录、项目配置、AGENTS.md、已读文件和前几轮工具输出，都会成为当前会话的一部分。

core 周围的 thread-store、context-fragments、prompts、state 等模块都和这件事有关。阅读这里时，要先记住一个限制：上下文是有限的。把所有文档塞给模型通常不会带来更好的结果，运行时必须决定哪些信息现在相关。

Agent 核心：把“下一步”接进循环

core 和 rollout 一类模块负责推进任务。它们收到模型的回答后，要判断这是一段最终回复，还是一次读取文件、运行命令、应用补丁的请求；工具完成后，又把结果放回会话，继续下一轮。

这里的 Agent 是一条反馈回路：模型选择动作，运行时执行动作，执行结果改变模型下一轮能看到的内容。

模型与后端：让上下文和响应来回流动

codex-api、backend-client、model-provider 等模块处理模型或后端通信。它们把核心层准备好的请求发出去，也把流式文本、工具请求和失败信息带回来。

这一层能告诉我们请求怎样构造、响应怎样解析、失败怎样恢复。可分析的边界到此为止：任务循环可以观察，模型的私有推理无法从源码中还原。

工具与安全边界：让动作发生，也让它止步

tools、exec、shell-command、file-search、apply-patch、git-utils 负责对工作区做事。它们把“搜索 OrderService”“读取测试”“运行 go test”变成真正的本地操作。

同一条路径旁边，还有 sandboxing、execpolicy、shell-escalation、process-hardening。它们存在的理由很朴素：模型可以提出动作，不代表动作就应该直接执行。一个能修改仓库的 Agent，必须知道哪些目录可写、哪些命令要确认、失败时怎样把原因交回会话。

工具让模型获得行动能力；安全边界让这种能力不会越过工作区和权限范围。两部分应该一起读，拆开看会误以为 Agent 只是“会跑 Shell”。

为什么第一张架构图要这样画

仓库里还有 skills、plugin、codex-mcp、hooks、memories 等模块。它们很重要，但不该在第一章抢走主线。

先看清任务怎样进入运行时、怎样调用模型、怎样使用工具，MCP 和 Skill 才有合适的位置。Skill 影响 Agent 如何做事，MCP 为它接入外部工具和数据；两者都建立在前面这条任务与工具链路之上。

后续读源码时，可以带着四个问题往下走：

  • 当前任务的上下文是从哪里组装出来的？
  • 工具请求怎样从模型响应变成一次本地调用？
  • 哪一层决定一条命令可以直接执行，还是需要确认？
  • 工具结果怎样重新进入下一轮会话？

这四个问题会比目录树更早把你带到关键代码。下一篇就从任务启动和会话创建开始。
