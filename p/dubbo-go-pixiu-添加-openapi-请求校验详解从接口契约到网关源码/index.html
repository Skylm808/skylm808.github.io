dubbo-go-pixiu 添加 OpenAPI 请求校验详解

这份文档不是单纯的 changelog，而是把 OpenAPI 的基础概念、Pixiu 里这次功能的代码写法、reviewer 指出的坑，以及最终修复思路串起来。读完之后，你应该能回答三个问题：

  • OpenAPI 到底描述了什么，为什么可以拿来做请求校验。
  • PR #948 为什么把 OpenAPI 校验从 apiconfig 中拆出来，最终代码怎么跑。
  • 这类网关热路径功能容易踩哪些坑，reviewer 为什么抓这些点。

0. OpenAPI 是什么

API 是 Application Programming Interface，泛指接口；HTTP、gRPC、Dubbo 都可以叫 API。OpenAPI 不是“新版本 API 的统一名字”，而是一种用 YAML/JSON 描述 HTTP API 契约的规范。

可以把它理解为一份机器可读的接口说明书：不仅写给人看，也能用于生成文档、生成客户端、Mock，以及在网关中校验请求。

它和 Pixiu 的路由不是一回事：

组件 回答的问题 例子
OpenAPI filter 请求是否合法？ source 是否必填、role 是否在 enum 内
apiconfig / Route 请求去哪里？ POST /users 命中哪个 Pixiu API/cluster
cluster 选哪个实例？ 在健康 endpoint 中选一个

最短记忆法：OpenAPI 决定“是否合法”，路由决定“去哪里”。

1. OpenAPI 先从 0 理解

OpenAPI 是一种 HTTP API 契约。它把“一个接口长什么样”写成机器可读的 YAML/JSON 文件。最小可以理解成四类信息：

  • paths：有哪些 URL，例如 /users、/users/{id}。
  • operation：每个 URL 支持哪些 HTTP method，例如 get、post。
  • parameters：path/query/header/cookie 参数长什么样，是否必填，有没有枚举、类型、长度、数值范围。
  • requestBody / responses：请求体和响应体的 schema。

这份契约表达的是：

  • 只有 POST /users 这个 operation 被声明。
  • query 参数 source 必填，只能是 web 或 app。
  • JSON body 必须是 object，且必须有 name 字段。
  • name 必须是字符串，长度在 3 到 8 之间。

所以 POST /users?source=web 携带 {"name":"tom"} 会通过；POST /users 少了 source，应该被拦。

1.1 在网关里用 OpenAPI 做什么

在 Pixiu 这种网关里，请求进入上游服务前会经过 filter chain。OpenAPI 校验 filter 的职责是：

  1. 启动时加载 OpenAPI 文件。
  2. 请求进来时先判断这个 path + method 是否在 OpenAPI 中声明。
  3. 如果没声明，跳过校验，把请求交给后续 filter 或上游。
  4. 如果声明了，就校验 path/query/header/body。
  5. 校验失败时本地返回 400 Bad Request，停止过滤链。

注意：这不是路由创建，也不是鉴权。OpenAPI 的 security 在本 PR 中故意关闭，认证授权仍由 JWT、OPA、SAML 等专门 filter 处理。

2. PR #948 做了什么

这次合入的核心变化是：新增独立的 dgp.filter.http.openapi，并删除原来塞在 dgp.filter.http.apiconfig 里的自研 OpenAPI 校验代码。

主要文件：

文件 作用

pkg/filter/http/openapi/openapi_test.go 新 filter 的单元测试
pkg/filter/http/apiconfig/api_config.go apiconfig 不再做 OpenAPI 校验，只做 API 路由元信息

3. 为什么要从 apiconfig 拆出来

旧实现把两件事混在一起：

  • apiconfig：Pixiu 自己的 API 路由配置，负责匹配 API，并把 API 元信息写入请求上下文。
  • OpenAPI request validation：按 OpenAPI 契约校验请求参数和 body。

这两个职责不是一回事。把 OpenAPI 校验做成独立 filter 后，配置和行为更清楚：

- name: dgp.filter.http.apiconfig
  config:
    path: configs/api_config.yaml

- name: dgp.filter.http.openapi
  config:
    path: configs/openapi_users.yaml
    max_request_body_bytes: 1048576

这意味着：

  • apiconfig 继续管 Pixiu 路由元数据。
  • openapi 只管 OpenAPI 文件中声明过的 operation。
  • 两个 filter 可以按需要组合，也可以单独使用。

迁移兼容的关键点是：旧字段 openapi_path 和 enable_openapi_validation 不能直接删掉，否则老配置会被 YAML 静默忽略，用户以为校验还开着，实际已经没生效。最终实现保留旧字段，但只要它们出现在配置里就启动失败，并提示迁移到 dgp.filter.http.openapi。

4.2 Apply：启动阶段加载和预处理

FilterFactory.Apply() 是最重要的启动阶段入口。最终逻辑是：

  1. cleanOpenAPIPath(factory.cfg.Path) 清理并校验 OpenAPI 文件路径。
  2. effectiveMaxRequestBodyBytes() 计算 request body 限制，默认 1MB，负数报错。
  3. loadValidatorFromFile(path) 读取 OpenAPI 文件，构建 SDK validator。
  4. 把 validator、model、validationOptions、maxRequestBody 存到 factory，后续每条请求复用。

这一步选择 fail fast：OpenAPI 文件缺失、路径不安全、$ref 不安全、模型构建失败，都会让 filter 启动失败，而不是运行时再出问题。

这里有几个关键决策：

  • 用 libopenapi-validator 负责请求校验，不再维护自研 schema 校验器。
  • BuildV3Model() 的 error 必须检查，不能拿半成品 model 继续跑。
  • WithoutSecurityValidation() 明确关闭 OpenAPI security 校验。
  • 手动构建并复用 PathTree，避免每个请求都做全量 path 扫描。

4.4 Decode：请求阶段校验

Filter.Decode() 是每个请求的入口：

  1. 如果 validator/model 没准备好，直接 Continue。
  2. findRequestOperation(req) 查找 OpenAPI 是否声明了当前 path + method。
  3. 如果没声明，跳过校验，Continue。
  4. 如果声明了，先 prepareRequestBodyForValidation(req) 做 body 大小限制和 body reset。
  5. 调用 ValidateHttpRequestSyncWithPathItem(req, pathItem, foundPath) 校验。
  6. 校验失败返回本地 400 Bad Request，并停止 filter chain。
  7. body 超限返回 413 Payload Too Large。

对 POST /users?source=web，findRequestOperation 先在规范中找到 /users 的 post operation；接着 body 限制逻辑读取并重新放回 req.Body；最后 validator 校验 query/body。任一步失败都会设置 local reply 并 Stop。HTTP connection manager 看到 local reply 后不会构造上游响应，因此非法请求不会到业务服务。

4.5 为什么要先判断 path + method

文档契约说的是“未声明的 path + method 跳过校验”。所以只找到 path 不够，还必须确认 method 也声明了。

最终实现的 hasRequestOperation() 按 HTTP method 检查 PathItem：

  • GET 必须有 pathItem.Get
  • POST 必须有 pathItem.Post
  • HEAD 必须显式有 pathItem.Head
  • 其它 method 走 GetOperations() 兜底

这解决了 reviewer 指出的行为不一致问题：如果 OpenAPI 只声明 GET /users，请求来了 POST /users，它应该被视为“未声明 operation”，跳过校验，而不是被 validator 当成错误请求拦掉。

4.6 为什么 HEAD 不 fallback 到 GET

HTTP 语义里 HEAD 常常和 GET 相关，但这里最终选择更保守的 OpenAPI 契约语义：只有显式声明 head operation 才校验 HEAD。

原因是 filter 的文档契约已经定义为“未声明 path + method 就跳过”。如果只声明了 get 却让 HEAD 走 GET 的参数/body 校验，会让行为变得隐式，甚至被 GET 的 requestBody 或参数约束误伤。

4.7 为什么要读完 body 再 reset

validator 要读取 request body 才能校验 JSON schema。Go 的 req.Body 是 stream，读一次就没了。如果 filter 读了 body 又不放回去，后续 filter 或上游就读不到 body。

最终实现：

1
2
3

body, err := readRequestBodyWithinLimit(req.Body, f.maxRequestBody)
req.Body = io.NopCloser(bytes.NewReader(body))
req.ContentLength = int64(len(body))

这保证了 validator 可以读，后续链路也还能读。

这里三个对象的分工很容易混淆：

1

req.Body = io.NopCloser(bytes.NewReader(body))
部分 作用
body []byte 真正保存已经读出的 JSON 字节
bytes.NewReader(body) 从这份内存字节的开头重新创建一个可读流
io.NopCloser(...) 给 reader 补一个什么都不做的 Close()，使其满足 req.Body 要求的 io.ReadCloser 接口

io.NopCloser 不保存 body，也不会重新读取网络；它只是适配类型。原始 HTTP body 通常关联网络资源，读完要关闭；现在的 body 已经在内存里，关闭时没有资源需要释放，所以 Close() 直接返回 nil 即可。

例如请求体是 {"name":"tom"}：filter 先读出字节到 body，原始 req.Body 此时已经 EOF；bytes.NewReader(body) 让下一位读取者又能从 { 开始读；io.NopCloser 让这个内存 reader 可以赋值给 req.Body。

4.8 为什么要限制 body 大小

旧写法或 SDK 内部如果直接 io.ReadAll(req.Body)，遇到很大的 JSON 或 chunked body，会在网关热路径上放大内存和延迟风险。

最终实现分两层：

  • 如果 ContentLength > maxRequestBody，不用读 body，直接 413。
  • 如果 ContentLength == -1 或未知，用 io.LimitReader(body, limit+1) 最多读 limit+1 字节，超过就 413。

默认限制是 1 << 20，也就是 1MB。配置为 0 使用默认值；负数启动失败。

5. 路径和 $ref 安全

OpenAPI 文件路径看起来只是本地配置，但在网关场景中，配置可能来自远程配置中心或被不同部署流程生成。reviewer 指出：如果直接 os.ReadFile(path)，就可能被配置成读取任意本地文件。

最终做了两层防护。

5.1 主 OpenAPI 文件路径

cleanOpenAPIPath() 要求：

  • path 不能为空。
  • 必须是相对路径，拒绝 /etc/passwd 这种绝对路径。
  • 拒绝包含 .. 的父目录跳转。
  • filepath.Clean() 规范化路径。
  • 用 filepath.EvalSymlinks() 解析 symlink。
  • 拒绝敏感 base 目录，例如 /、/etc、/proc、/sys、/dev、/run、/var/run。

重点是 symlink：如果只检查字符串，specs/current 可以是指向 /etc 的符号链接。最终实现会解析真实路径，再判断是否落到敏感目录里。

5.2 OpenAPI 外部 $ref

OpenAPI 支持：

1

$ref: ./schemas/user.yaml#/User

这很常见，但也意味着：OpenAPI 文件里的一个字符串，会影响网关启动时读取哪个本地文件。 因此它不能只交给 SDK 随意解析，必须先划定读取边界。

假设主文件是 configs/openapi/api.yaml，它所在目录就是本次允许读取的根目录 configs/openapi。

$ref 结果 原因
./schemas/user.yaml#/User 允许 相对当前文件，真实路径仍在根目录内
common.yaml#/Error 允许 同目录下的本地文件
https://example.com/schema.yaml#/User 拒绝 不让启动过程发起远程读取
/etc/passwd#/User 拒绝 绝对路径可能读取宿主机敏感文件
../../secret.yaml#/User 拒绝 .. 试图越出 OpenAPI 目录
./schemas/current/user.yaml#/User，其中 current -> /etc 拒绝 字符串看似安全，真实路径已逃出根目录

validateExternalRefs() 的关键调用如下（省略 err 的返回处理）：

spec, _ := os.ReadFile(path)
yaml.Unmarshal(spec, &raw)
refs := collectExternalRefs(raw, nil)

for _, ref := range refs {
    validateExternalRefPath(path, rootDir, ref)
    refPath, _, ok := splitRefTarget(ref)

    resolved := resolveSafeReferencePath(filepath.Dir(path), refPath, rootDir)

可以按四步理解：

  1. collectExternalRefs() 递归遍历 YAML 的 map 和 list，找到所有 $ref；只引用当前文件内部的 #/components/... 不需要读新文件，所以跳过。
  2. validateExternalRefPath() 先拒绝带 scheme 或 host 的远程地址、绝对路径和 ..。例如 https://... 的 scheme 是 https，会直接报错。
  3. resolveSafeReferencePath() 用当前文件目录和相对路径拼出候选文件，再用 filepath.EvalSymlinks() 得到真实位置；最后用 filepath.Rel() 判断真实位置是否仍属于 rootDir。这一步专门防 symlink 绕过。
  4. 校验通过后才递归读取被引用文件，继续检查它里面的 $ref。visited 记录已经检查过的文件：多个 schema 都引用 common.yaml 时不重复扫描；A 引 B、B 又引 A 时也不会无限递归。

这部分是 reviewer 建议后逐步补强的。最初只限制主 OpenAPI 文件还不够，因为 SDK 随后会顺着外部 $ref 继续读文件；安全边界必须覆盖这条完整读取链路。配置加载阶段只要发现一个不安全引用，就让 filter 启动失败（fail fast），而不是带着不确定的文件来源继续运行。

- name: dgp.filter.http.apiconfig
  config:
    path: configs/api_config.yaml

- name: dgp.filter.http.openapi
  config:
    path: configs/openapi.yaml

兼容实现的细节在 pkg/filter/http/apiconfig/config.go：

  • 字段 OpenAPIPath 和 EnableOpenAPIValidation 仍然保留。
  • 自定义 UnmarshalYAML 遍历 YAML node，记录旧 key 是否出现过。
  • hasDeprecatedOpenAPIConfig() 同时检查字段值和 key presence。

为什么要检查 key presence？因为这两个配置即使写成下面这样，也应该提示用户迁移：

1
2

openapi_path: ""
enable_openapi_validation: false

如果只看 Go 零值，空字符串和 false 会被当成“没有配置”，导致老配置静默通过。这是 reviewer 点出的兼容性坑。

7. Reviewer 评论暴露出的坑

7.1 P0：BuildV3Model() error 不能丢

Reviewer: AlexStocks
讨论: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3308060049

问题：doc.BuildV3Model() 可能同时返回部分 model 和非 nil error。如果只拿 model、不处理 error，就会让残缺 OpenAPI model 进入 validator。

风险：

  • 本该拒绝的请求被放行。
  • 本该放行的请求被误拦。
  • 启动看起来成功，实际 validator 状态不可信。

最终修法：只要 BuildV3Model() 返回 error，Apply() 阶段直接失败：

1
2
3
4

model, err := doc.BuildV3Model()
if err != nil {
    return nil, nil, nil, errors.Wrap(err, "build openapi model")
}

经验：初始化阶段拿到“不完整但可用”的对象时要非常谨慎。网关校验类功能最好 fail fast，不要带病运行。

7.2 path 存在不等于 operation 存在

Reviewer: Copilot
讨论: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3307899137

问题：只检查 /users 这个 path 是否存在，不检查 POST /users 是否存在，会让“未声明 method 应跳过”的文档行为失效。

最终修法：findRequestOperation() 找到 PathItem 后再调用 hasRequestOperation()，确认 method-specific operation 存在。

经验：OpenAPI 的最小校验单位不是 path，而是 path + method，也就是 operation。

7.3 FindPath 错误和“未声明路径”不能混在一起

Reviewer: AlexStocks
问题：FindPath 返回的 validationErrs 如果直接当成“不匹配”吞掉，会让排查困难。

最终修法：如果 validationErrs 非空，记录 Debug 日志，包含 method、path 和错误详情；业务上仍按未匹配跳过，避免把 lookup 异常暴露给客户端。

经验：客户端响应要克制，但内部日志要能定位问题。

7.4 os.ReadFile(path) 前必须有路径边界

Reviewer: AlexStocks / Similarityoung
相关后续回复: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3369027070

问题：OpenAPI path 来自配置，直接读文件可能让配置注入变成本地文件读取。

最终修法：

  • 主 OpenAPI path 必须相对。
  • 拒绝 .. 和敏感目录。
  • 解析 symlink 后再判断。
  • 外部 $ref 也做同样的相对路径和根目录限制。

经验：配置文件路径也是输入。只要输入能影响 ReadFile，就需要边界。

7.5 body 读取不能无限制

Reviewer: Similarityoung
问题：请求校验前读取 body，如果没有大小限制，会导致大 body 或 chunked body 消耗过多内存。

最终修法：新增 max_request_body_bytes，默认 1MB；已知 Content-Length 超限直接 413，未知长度用 LimitReader(limit+1)。

经验：网关热路径不要无界 io.ReadAll。

7.6 FindPath(nil options) 会绕过 PathTree 优化

Reviewer: Similarityoung
问题：每次请求都传 nil options，SDK 可能走全量 path 扫描和正则匹配。

最终修法：Apply() 阶段构造 ValidationOptions，并显式构建 PathTree，请求阶段复用。

经验：能在启动时做的索引构建，不要放到每个请求里做。

7.7 旧配置不能静默失效

Reviewer: Alanxtl / Similarityoung
问题：删除 apiconfig 中旧字段会导致老配置被忽略。用户以为 OpenAPI 校验还在，实际已经没开。

最终修法：保留旧字段，并检测 key 是否出现；只要出现就启动失败，提示迁移。

经验：配置迁移时，最危险的不是报错，而是静默降级。

7.8 文档不能夸大能力边界

Reviewer: Similarityoung
相关后续回复: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3369028350

问题：OpenAPI 3.2 有 additional operations，但当前 filter 只接入标准 HTTP operation。如果文档写成“完整支持 OpenAPI 3.2”，就和实际能力不一致。

最终修法：文档收窄为“OpenAPI 3.2 文档中使用标准 HTTP operation 的部分可被当前 filter 覆盖”。

经验：文档是契约。实现没有覆盖的行为，不要提前承诺。

7.9 HEAD 行为要明确

Reviewer: Similarityoung
相关后续回复: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3369027921

问题：HEAD 是否可以 fallback 到 GET 容易产生歧义。

最终修法：HEAD 必须显式声明 head operation 才校验；只声明 GET 时，HEAD 跳过校验。

经验：遇到协议习惯和 OpenAPI 契约语义冲突时，要选一个清晰、可测试的规则，并写入文档。

7.10 CI 和 import 分组

Reviewer: Copilot / Similarityoung
Issue comment: https://github.com/apache/dubbo-go-pixiu/pull/948#issuecomment-4608535037

问题：新增 import 分组不符合仓库风格，CI 要求格式化。

最终处理：使用仓库匹配的 import formatting 修复。

经验：Go 代码不只是 gofmt，有些仓库还依赖 imports-formatter 或自定义 import 分组规则。

7.11 一个未在本 PR 修的系统性问题

讨论: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3369035830
补充: https://github.com/apache/dubbo-go-pixiu/pull/948#discussion_r3369354582

问题：FilterManager.ReLoad() 中如果某个 filter 的 Apply() 失败，错误传播和失败 factory 安装逻辑存在系统性风险，可能影响所有会在 Apply() 失败的 filter，不只是 OpenAPI。

本 PR 的边界：OpenAPI filter 已经正确返回 error；FilterManager 的 reload 行为建议单独 follow-up 修。

经验：review 中发现的问题要分清“本 PR 内的职责”和“系统性 follow-up”。不要把一个功能 PR 扩成无边界重构，但要把风险记录清楚。

8. 如果从 0 写这个功能，推荐顺序

  1. 先把 filter 边界想清楚：只做请求校验，不做路由、不做鉴权、不做响应校验。
  2. 先写最小 happy path：加载 spec，构建 validator，POST /users 合法请求通过。
  3. 加失败路径：必填 query 缺失、body schema 不符合时返回 400。
  4. 加“未声明跳过”：path 未声明跳过，method 未声明跳过。
  5. 加启动 fail fast：无效 spec、无效 $ref、validator nil 都不能运行。
  6. 加热路径保护：body size limit，body reset，PathTree 复用。
  7. 加配置迁移：旧 apiconfig 字段存在就报错，不能静默忽略。
  8. 最后写用户文档：明确支持什么、不支持什么、怎么配置、怎么迁移。

9. 最后记住这些设计原则

  • OpenAPI 校验单位是 operation，也就是 path + method。
  • OpenAPI 文件和 $ref 都是输入，不可信。
  • 网关热路径不能有无界读、无界扫描、无界解析。
  • 初始化阶段错误要 fail fast，不要拿部分构建结果继续跑。
  • 客户端错误信息要克制，内部 Debug 日志要足够定位。
  • 配置迁移时宁可启动失败，也不要让老配置静默失效。
  • 文档不要写超过实现能力的承诺。
