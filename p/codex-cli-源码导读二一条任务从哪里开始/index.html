阅读时长: 4 分钟

Codex CLI 源码导读（二）：一条任务从哪里开始

在仓库根目录输入 codex，屏幕上会出现一个可以继续对话的终端界面。你输入任务，它显示读取文件、执行命令和最终结果。表面上看，像是一个终端程序直接在工作。

源码里的分工比这多一层：命令行程序负责启动，终端界面负责展示和收集输入，app-server 负责接住结构化任务，再把任务进度送回界面。先把这三层分开，后面看任务循环才不会迷路。

本文固定使用 openai/codex 的 268288a 版本。

先认识三个名字

这三个词都和终端有关，却不是同一件事。

名称 用人话说 它不负责什么
CLI 你输入 codex 后启动的命令行程序 不独自完成整项任务
TUI 终端中的交互界面，显示消息、进度和确认提示 不是桌面 App
app-server 在本机接收任务并回传进度的服务 不是 Codex App

TUI 的全称是 Terminal User Interface，意思就是“终端里的界面”。它负责让人看见和操作任务。app-server 更接近后台接线员：界面把“开始这项任务”的请求交给它，它把“任务正在读文件”“命令执行完了”这类进度消息送回来。

Codex App 是另一种桌面端产品入口。公开 CLI 仓库里的 app-server 能为编辑器等富客户端提供本地服务，但不能据此推断它就是桌面 App 的全部实现。

flowchart LR
    U["开发者"] --> C["CLI：启动 codex"]
    C --> T["TUI：终端界面"]
    T --> S["app-server：本地任务服务"]
    S --> R["core：任务运行时"]
    I["IDE 等其他界面"] --> S
    R --> S
    S --> T

图中的 core 暂时只要理解为“实际协调任务、模型和工具的运行时”。第三篇才会进入它。这里最重要的是：TUI 和 IDE 都是界面；它们可以经由同一个本地服务提交任务、接收进度。

从 codex 命令开始

源码入口在 codex-rs/cli/src/main.rs。它先读取参数和配置，再决定以哪种方式启动。

flowchart TD
    A["终端输入 codex"] --> B["CLI 解析参数与配置"]
    B --> C{"本次要做什么？"}
    C -->|"没有子命令"| D["启动 TUI"]
    C -->|"codex exec"| E["执行一次非交互任务"]
    C -->|"codex app-server"| F["启动本地任务服务"]
    D --> G["进入同一套任务运行时"]
    E --> G
    F --> G

最常见的是第一条路：不带子命令时，CLI 启动 TUI。codex exec 更适合脚本里的一次性调用，因此不需要在终端里持续画界面。codex app-server 则单独启动服务，供编辑器一类的客户端连接。

三条路径的外观不同，任务仍会落到同一类运行时能力上。它们不是三套不同的 Agent。

任务怎样从界面交给服务

这里会第一次遇到 thread/start 和 turn/start。不用把它们当作陌生术语，它们只是协议里的“对象/动作”写法：

写法 直白含义
thread/start 创建一条新的任务记录
turn/start 在这条记录中开始处理这一次用户输入

start 就是“开始”；斜杠前是操作的对象。类似地，turn/completed 表示“这一轮已经结束”。

这里的 Thread 可以先理解为一条可保存的长期任务记录；Turn 是其中的一次请求。例如，你先说“看看这个失败测试”，再说“继续修复”，它们可以属于同一个 Thread，但会是两个 Turn。下一篇会把这两个概念放进完整任务过程里。

客户端和 app-server 用一种约定好的消息格式通信。这个格式叫 JSON-RPC；现在只需记住它的作用：**客户端发一个有名字的请求，服务返回结果，并持续发送进度事件。**具体的 JSON、socket 和传输方式，等读到事件层时再看，不需要先背下来。

sequenceDiagram
    participant U as 开发者
    participant T as TUI 或 IDE
    participant S as app-server

    U->>T: 输入“修复这个失败测试”
    T->>S: thread/start（第一次时创建任务记录）
    T->>S: turn/start（提交这次输入）
    S-->>T: 任务已开始、工具进度、文本结果
    T-->>U: 在终端或编辑器中展示

默认 TUI 也会走这套接口。源码中的 tui/src/app_server_session.rs 保存了这层客户端会话，负责把 TUI 的输入转换成服务请求。这个文件名现在不必细读，只要知道：终端界面不是绕开服务直接操纵任务运行时。

到源码里找什么

本章只需要记住三个起点：

位置 先观察什么
codex-rs/cli/src/main.rs 命令如何分流到不同启动方式
codex-rs/tui 终端界面怎样提交输入、展示事件
codex-rs/app-server 服务怎样接收请求、发出进度

看到 AppServerSession、JSON-RPC、JSONL 或 socket 时，可以先把它们归到同一个问题：**界面怎样把任务交出去，又怎样拿回过程？**它们是实现这件事的不同部件，不是新的 Agent 概念。

接下来可以继续想

  • 为什么 TUI 不直接调用任务运行时，而要经过一层服务接口？
  • 同一个 Thread 中连续发出两个 Turn，第二个 Turn 应该看见哪些前一个任务留下的信息？
  • codex exec 没有 TUI 时，怎样拿到任务完成或失败的结果？

下一篇从 turn/start 开始。我们会把它放回一项完整任务里，看它怎样变成模型、工具和进度事件之间的多轮接力。
