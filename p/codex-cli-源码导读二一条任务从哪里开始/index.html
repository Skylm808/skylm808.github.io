Codex CLI 源码导读（二）：一条任务从哪里开始

在终端里输入 codex 后，第一件发生的事不是调用模型。程序要先回答一个很朴素的问题：这次启动到底想做什么？是打开交互式终端，执行一条非交互任务，还是作为一个本地服务等待其他客户端连接？

这个分流点在 codex-rs/cli/src/main.rs 里。它看似只是命令行解析，实际上规定了任务从哪个入口进入系统，也解释了为什么 tui 和 app-server 都出现在 Codex 的源码中。

本文只追任务的入口。上下文如何组装、工具如何执行、模型如何多轮推进，留给后面的章节。先把“谁接住了任务”弄明白。

codex 不是一条固定路径

cli crate 定义了一个多命令入口。输入没有子命令时，程序走交互模式；codex exec 走非交互执行；codex app-server 则启动一个面向客户端的服务。还有 mcp、plugin、review、resume 等管理或专用命令。

flowchart TD
    A["终端输入 codex"] --> B["cli 解析参数与配置"]
    B --> C{"有没有子命令？"}
    C -->|没有| D["启动交互式 TUI"]
    C -->|exec| E["非交互执行路径"]
    C -->|app-server| F["启动本地协议服务"]
    D --> G["任务运行时"]
    E --> G
    F --> G

图中的三条路径不是三套 Agent。它们面向不同的使用方式：

  • 交互式 TUI 适合人在终端里持续对话、查看命令和确认操作。
  • exec 适合脚本或 CI 中的一次性调用，结果需要按约定输出。
  • app-server 适合 IDE 等富客户端，它们需要提交任务并持续接收结构化事件。

第一篇把 CLI、App 和 IDE 称作“入口”，这里可以再精确一点：cli 是可执行文件的总入口；TUI 和 app-server 是它分流后的两种本地交互通道。桌面 App 是另一种产品体验，不能用一个 Rust 目录替代它的全部实现。

默认启动：CLI 先把控制权交给 TUI

在 cli/src/main.rs 中，根命令没有子命令时会调用 run_interactive_tui。这个函数做几件启动前的事：规范化传入的提示文本，检查终端是否可交互，解析可选的远程端点，然后进入 codex_tui::run_main。

这里的 tui 是 Terminal User Interface。它不是日志输出，也不是桌面图形界面。它负责在终端里渲染消息、输入框、命令进度、批准请求和任务状态。

如果只看界面，容易误以为 TUI 自己就是 Agent 运行时。当前源码的做法更有意思：TUI 会创建一个 AppServerSession，通过它启动或连接 app-server，再以客户端身份发起任务。也就是说，终端界面和任务服务之间已经有一层明确接口。

sequenceDiagram
    participant U as 用户
    participant C as cli
    participant T as tui
    participant S as AppServerSession
    participant R as app-server

    U->>C: codex
    C->>T: run_main
    T->>S: 创建会话
    S->>R: 初始化并创建 Thread
    U->>T: 输入任务
    T->>S: 发起 Turn
    S->>R: turn/start
    R-->>T: 任务事件与输出

这条路径有一个值得记住的细节。默认的本地交互模式可以启动嵌入式 app-server，也可以按配置复用本地 daemon 或连接远程端点。嵌入式不等于没有服务层，它表示客户端和服务端通过进程内通道通信，而不是通过一个可见的 TCP 端口或标准输入输出通信。

所以，TUI 仍然是界面层。它只是没有绕开任务服务，而是通过 AppServerSession 使用同一套 Thread、Turn 和事件协议。这种拆分让终端界面、IDE 和其他客户端有机会谈同一种“任务语言”。

app-server 不是桌面 App

这个名字很容易造成误解。源码里的 app-server 是一个可由 codex app-server 启动的本地服务。它的公开 README 说得很直接：它为 Codex VS Code 扩展这类富客户端提供接口。

它使用双向 JSON-RPC 2.0 协议。默认传输方式是标准输入输出上的 JSONL；也支持 Unix socket，WebSocket 处于实验状态。这里的重点不是记住传输名，而是理解角色：客户端负责展示与交互，app-server 负责接受结构化请求，并把任务过程变成结构化事件。

一个富客户端的最小工作过程大致如下：

sequenceDiagram
    participant I as IDE 或 TUI
    participant S as app-server

    I->>S: initialize
    I->>S: initialized
    I->>S: thread/start
    S-->>I: thread/started
    I->>S: turn/start
    S-->>I: item/started、文本增量、工具进度
    S-->>I: item/completed
    S-->>I: turn/completed

初始化不是装饰步骤。协议要求连接先完成 initialize 和 initialized，之后才能发送其他请求。随后，客户端用 thread/start 创建会话，用 turn/start 把用户输入送入该会话。任务跑起来后，客户端不需要猜测终端该显示什么，而是消费 item/started、文本增量、工具进度和 turn/completed 等事件。

这就是为什么 IDE 集成不必重写一套 Agent。它只需实现协议客户端：发送请求，订阅事件，把事件渲染成编辑器里的消息、改动和批准界面。

Thread 和 Turn：入口层已经在使用的任务单位

app-server README 用三个词描述一次交互：Thread、Turn 和 Item。现在只需要掌握前两个。

  • Thread 是一次持续的会话。它可以被恢复或分叉。
  • Turn 是会话中的一轮交互，通常从用户输入开始，到 Agent 完成或中断结束。

TUI 里的 app_server_session.rs 很直观地保留了这两个动作：start_thread 发送 thread/start，turn_start 发送 turn/start。它还把当前目录、模型设置、审批策略和权限覆盖传入请求。

这不意味着界面层决定所有运行时规则。TUI 收集启动所需的信息，并把它交给服务端。真正的任务循环、工具调用和安全判定仍在后面的路径中发生。第三篇会从 Turn 进入运行时后开始追。

三个名字，三个层次

最后再压缩一次，避免后面读源码时混淆：

名称 它是什么 不是什么
cli 命令行程序的分流入口 一整套 Agent 逻辑
tui 终端里的交互界面与客户端 桌面 App
app-server 给富客户端使用的本地协议服务 Codex App 的同义词

这三个模块都和“任务开始”有关，但职责并不重叠。CLI 选择路径，TUI 负责终端交互，app-server 提供任务与事件协议。把它们分清楚，后面看到 Thread、Turn、事件流和工具调用时，才不会把界面代码误当成 Agent 核心。
