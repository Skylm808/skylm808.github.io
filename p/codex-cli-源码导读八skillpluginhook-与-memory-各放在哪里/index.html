Codex CLI 源码导读（八）：Skill、Plugin、Hook 与 Memory 各放在哪里

给 Codex 加一条团队约定时，很容易把几种扩展面混在一起。有人把测试步骤写成 Hook，结果每次工具调用都被打断；有人把一个可复用的审查流程做成 MCP 服务，却没有外部系统可连；还有人把一次会话里的临时事实当成长期 Memory。

这几种东西解决的问题不同。Skill 是一份可复用的工作方法；Plugin 是把若干能力装在一起的分发包；Hook 在特定生命周期节点执行规则；Memory 是为跨任务复用而保存的信息。MCP 已在第七篇讨论过，它负责连接外部工具和数据。

本文固定使用 openai/codex 的 268288a。重点是公开 CLI 源码中各机制的职责和连接点，不把它们的产品界面或任何私有服务实现混进来。

先从一个选择题开始

假设支付仓库希望每次修改接口时都遵循同一套约定：先阅读目录里的项目指令，查找受影响调用方，运行目标测试，最后检查不应改动的 API 定义。团队还想把这套方法分享给其他仓库。

这时应先问“要改变什么”，而不是先问“该写哪个配置文件”。

需求 更合适的扩展面 原因
给 Agent 一份按步骤工作的审查方法 Skill 内容是方法、检查清单和参考资料
将该 Skill、一个 MCP 服务和 Hook 一起安装 Plugin 内容需要作为一个版本化包分发
每次写文件前都检查路径或补充固定上下文 Hook 规则必须卡在工具调用的生命周期节点
让未来任务复用已沉淀的经验或摘要 Memory 内容的生命周期跨越当前 Thread
查询缺陷系统或内部文档 MCP 能力和数据位于外部系统

Plugin 可以包含 Skill、Hook 和 MCP 服务，所以它容易让人误以为自己也是一种任务方法。图里那条箭头说明了关系：Plugin 管的是打包与安装，组件本身仍按各自的机制运行。

Skill：让模型知道一套工作方法

一个 Skill 的主体是 SKILL.md。codex-rs/core-skills/src/loader.rs 会扫描配置、用户目录、项目目录和已启用 Plugin 提供的 Skill 根目录，读取前置元数据，例如名称和描述。这里的根目录是发现范围，不是把目录里所有文本一次性塞进模型上下文。

运行时先把可用 Skill 的名称、描述和位置整理成目录。这份目录会受上下文预算限制：描述过多时会缩短，极端情况下只保留足以发现它们的信息。任务中明确提到某个 Skill 后，build_skill_injections 才读取对应 SKILL.md 的正文，并把它注入当前任务。

sequenceDiagram
    participant L as Skill Loader
    participant M as 模型
    participant I as Skill Injection
    participant S as SKILL.md

    L->>L: 扫描根目录，读取名称和描述
    L-->>M: 可用 Skill 目录
    M->>I: 明确提到某个 Skill
    I->>S: 读取正文和相关资源
    S-->>I: 工作步骤与局部知识
    I-->>M: 当前任务的 Skill 指令

这解释了 Skill 的两个特点。它不是另一种模型，也不是自动执行的脚本。它先让模型知道“这里有一套支付接口审查流程”，在被选中后再提供具体步骤、参考文档或脚本位置。模型仍要在正常的 Agent 循环里选择工具，并接受第六篇的权限和审批约束。

Skill 也可以声明依赖，例如需要某种工具或 MCP 服务。core/src/mcp_skill_dependencies.rs 负责在被明确提及的 Skill 需要外部能力时检查依赖。这并没有改变 Skill 的本质：依赖让工作方法可以使用能力，能力本身仍由 MCP 或内置工具系统提供。

Plugin：把一组组件变成可安装包

Plugin 的入口是包内的 .codex-plugin/plugin.json。codex-rs/plugin/src/manifest.rs 解析清单中的名称、版本、描述和组件路径。清单可以列出 Skill 根目录、MCP 服务定义、App 声明和 Hook 声明，也可以附带界面元数据。

review-bundle/
├── .codex-plugin/plugin.json   # 包名与组件入口
├── skills/                     # 可复用工作方法
├── hooks/                      # 生命周期规则
└── .mcp.json                   # 可选的 MCP 服务定义

上面的目录只是说明组件如何归属，不是每个 Plugin 都必须包含四项。一个 Plugin 可以只提供 Skill，也可以只提供某个 MCP 服务。加载结果会分别保存有效的 Skill 根目录、MCP 服务、App 和 Hook 来源；只有 Plugin 已启用且没有加载错误时，这些组件才会进入后续运行时。

因此，Plugin 的价值在于分发边界。团队若要把“支付接口审查 Skill”和“查询发布状态的 MCP 服务”交给多个仓库使用，做成一个 Plugin 比让每个人手工复制几个文件更合适。若只想为当前仓库写一份审查方法，直接放一个 Skill 通常更清楚。

Hook：在动作发生的节点执行规则

Hook 不等待模型主动提到它。它由运行时在特定事件上调用。公开配置支持 SessionStart、PreToolUse、PermissionRequest、PostToolUse、Stop 等事件，每个事件可以按工具名匹配一个或多个处理器。

PreToolUse 很能说明 Hook 和 Skill 的差别。Skill 可以建议“改文件前检查路径”；PreToolUse Hook 则在工具准备执行时收到结构化输入，能够阻止调用、补充模型上下文，或在未阻止时改写工具参数。PostToolUse 则能查看工具输入和输出，添加反馈或停止后续处理。

sequenceDiagram
    participant M as 模型
    participant R as 工具运行时
    participant H as PreToolUse Hook
    participant T as 工具
    participant P as PostToolUse Hook

    M->>R: 请求工具调用
    R->>H: 工具名和结构化参数
    alt Hook 阻止
        H-->>R: 拒绝原因
        R-->>M: 工具结果
    else Hook 放行或改写参数
        H-->>R: 继续执行
        R->>T: 调用工具
        T-->>R: 工具输出
        R->>P: 输入和输出
        P-->>R: 反馈或停止信号
        R-->>M: 最终工具结果
    end

源码中的 core/src/hook_runtime.rs 负责把 Session、Turn、工具名和参数交给 Hook 引擎；hooks/src/events/pre_tool_use.rs 则处理选择匹配项、执行处理器和解析返回结果。Hook 的适用对象是可机械判断的规则，例如禁止向某个路径写入、在特定命令前要求说明、或在工具运行后收集固定反馈。

把模糊的工程判断硬塞进 Hook 往往会带来麻烦。比如“这个重构是否值得做”需要理解任务目标和代码上下文，适合让模型在 Skill 的方法指引下判断；“不得修改公开 API 目录”可以由 Hook 稳定拦截。

Memory：跨任务复用的信息，但有条件

Memory 不等于当前 Thread 的消息历史。历史记录的是这次任务已经说过什么、调用过什么；Memory 的读写模块面向更长的复用周期。公开源码把内存目录放在 CODEX_HOME/memories 下，并提供读取、引用和写入管线。

config.toml 中的 memories 配置至少区分三件事：是否生成新的 Memory、是否在任务中使用 Memory、外部上下文出现时是否停止让该 Thread 产生可复用 Memory。最后一项很实际：当任务大量依赖 MCP、网络或其他外部上下文时，运行时可以把 Thread 标记为 polluted，避免把来源和稳定性不明的内容直接沉淀到后续任务。

Memory 写入也不是每完成一个命令就立刻发生。codex-rs/memories/write/src/start.rs 显示，写入管线受功能开关、配置、是否为根任务、状态数据库和速率限制等条件控制，并在后台分阶段执行。读路径和写路径刻意分在不同 Rust 模块（crate）中。这个设计提醒我们：Memory 是受控的长期信息管理，不是把所有对话自动永久保存。

对支付仓库而言，适合进入 Memory 的是已经反复验证过的通用经验，例如某类测试的稳定运行方式。一次临时排查的日志、带有外部系统内容的回复，或尚未确认的猜测都不适合直接当作长期依据。

回到选择：先按生命周期切开

如果内容只在当前任务中指导模型怎么做，选 Skill。需要把几种组件一起发布和启用，选 Plugin。动作发生前后必须机械执行的规则，选 Hook。信息需要跨 Thread 留存，并且来源与质量允许沉淀时，才考虑 Memory。需要连外部系统时，选 MCP。

这些边界让扩展之间可以组合，而不必彼此冒充。一个 Plugin 可以带来审查 Skill 和路径保护 Hook；审查 Skill 可以依赖一个查询发布状态的 MCP 服务；Memory 可以记录经过验证的工作习惯。它们围绕同一次任务协作，分别负责方法、分发、强制、沉淀和连接。

接下来可以继续想

  • 一个 Hook 能改写工具参数时，审计与调试应记录原参数还是改写后的参数？
  • Plugin 带来的 MCP 服务应该沿用谁的权限与认证配置？
  • 哪些信息满足跨任务复用的条件，哪些信息只该留在当前 Thread？
