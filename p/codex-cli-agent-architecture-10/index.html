Codex CLI 源码导读（十一）：从一句任务到一次可验证的改动

前十篇把 Codex 拆成了运行时、上下文、工具、权限、协作与事件。最后把它们放回一项小任务里看，事情会具体许多：模型给出下一步的选择，Codex 让选择落到工作区；改动是否成立，还要靠构建和差异检查来回答。

本文使用本系列所在的 Hugo 站点仓库作为实例。这个仓库的部署工作流会运行 Hugo 构建，因此“新增一篇文章并让站点仍可构建”是一个真实、范围也足够小的任务。

1

3

在 content/post/ai/ 新增一篇 Codex CLI 源码导读文章。
保持同系列的 Hugo front matter，保留已有改动，
并验证站点可以渲染。

Hugo front matter 是文章开头由 +++ 包住的一段元数据，用来声明标题、日期、分类和标签。这个任务改的是内容文件，不涉及发布、推送或远程服务。本文固定对照 openai/codex 的 268288a，并以当前工作区可观察到的文件和构建结果为证据。

一条任务的完成条件

“写完文章”太模糊。对这个仓库而言，完成至少意味着：新文件位置正确，front matter 与同系列一致，Hugo 构建成功，改动没有引入空白格式错误，并且既有的未提交文件没有被覆盖。

flowchart LR
    A["用户任务"] --> B["Thread 与 Turn"]
    B --> C["收集工作区上下文"]
    C --> D["读取相邻文章和配置"]
    D --> E["应用补丁写入新文章"]
    E --> F["运行 Hugo 构建"]
    F --> G["检查 diff 与结果"]
    G --> H["最终汇报"]

    P["权限与审批"] -.约束.-> D
    P -.约束.-> E
    P -.约束.-> F

这张图没有画出模型的内部思路。公开源码可以证明 Codex 如何管理 Turn、工具和事件，却不能把模型的隐含推理逐字还原出来。阅读 Agent 架构时，这条边界很重要：我们分析的是可执行、可审计的流程，而不是猜测模型脑中的完整过程。

1. 任务先成为一次 Turn

无论用户从 codex 命令、TUI 还是富客户端提交这段话，运行时都会把它放进一个 Thread，并发起一个 Turn。Thread 保存持续的会话，Turn 表示这一轮由用户输入驱动的工作。

在 codex-rs/core/src/tasks/regular.rs 中，常规 Turn 先发出 TurnStarted 事件，再进入 run_turn。codex-rs/core/src/session/turn.rs 中的循环负责把模型响应、工具结果和后续输入串起来。若用户在任务中途补充限制，运行时会检查输入队列，再决定是否继续下一轮。

对这项任务，第一轮不需要调用 MCP，也不需要连接外部系统。工作区已经给出了足够多的本地证据：站点配置、文章目录、同系列文章和 Git 状态。

2. 上下文不是把整个仓库塞进提示词

一个稳妥的执行顺序会先观察，再修改。这里需要看的内容很有限：

要确认的事实 可读取的本地对象 它影响的决定
站点的文章类型和链接规则 hugo.yaml，以及现有 content/post/ 结构 新文件应放在 content/post/ 下
同系列怎样命名和标记 content/post/ai/codex-cli-agent-architecture-01.md 至 10.md 文件名和 front matter 保持连续
哪些改动已经存在 git status --short 只新增第十一篇，不覆盖前十篇或其他文件
发布时实际运行什么 .github/workflows/hugo_main.yaml 本地检查至少应覆盖 Hugo 渲染

这正对应第四篇的内容。运行时会从当前目录、项目指令、配置和已读文件构造当前步骤的上下文；文件正文通常在工具读取后才加入。codex-rs/core/src/session/step_context.rs 保存每一步的环境快照，工具在 codex-rs/core/src/tools/context.rs 中取得这份信息。

Git 状态尤其不能跳过。当前工作区已经有前十篇导读的改动。它们可能来自同一项写作，也可能来自用户的其他安排。Agent 的正确动作是把它们当成既有状态，而不是为了获得“干净工作区”去还原文件。

3. 模型选择工具，Codex 检查工具能否执行

在这项任务里，模型可能依次选择搜索、读取、写入和执行命令。选择本身来自模型；每一步实际如何落地，由工具系统决定。

rg --files content/post/ai/          # 找到相邻文章
sed -n ...                           # 读取 front matter 与写法
apply_patch                          # 新增第十一篇文章
hugo --renderToMemory                # 在内存中构建站点
git diff --check                     # 检查已跟踪改动的空白错误

apply_patch 是按补丁格式描述文件新增、删除或替换的工具。它比把整份文件交给 Shell 覆盖更容易验证目标和改动范围。codex-rs/core/src/tools/handlers/apply_patch.rs 会先解析补丁，再针对选定环境的文件系统核验路径与内容，之后才把改动交给运行时。

这里的权限判断也很具体。文章目录在工作区内时，写入是否需要确认取决于当前权限配置和审批策略；跑 Hugo 则属于执行命令，也会经过命令策略与沙箱判断。codex-rs/core/src/tools/sandboxing.rs 将命令分为可直接执行、需要审批和禁止三类。它不因为命令看上去“只是构建”就跳过边界。

本例没有申请更宽的目录权限，也没有运行发布或推送命令。任务只需要写当前工作区里的内容文件，并在本地验证。这个范围足以完成目标，也让副作用清楚可见。

4. 工具结果会改变下一步

工具不是一次性流水线。每个工具结果都会回到当前 Turn，成为下一轮可用的信息。

例如，读取 hugo.yaml 后可以知道文章的 permalink 由 post 规则生成；读取相邻文章后可以知道 front matter 使用 TOML 语法；git status --short 会提醒执行者不要碰已有文件。写入完成后，Hugo 构建的退出状态决定下一步：成功则检查差异，失败则读取报错并回到相关文件修正。

sequenceDiagram
    participant M as 模型
    participant T as 工具运行时
    participant W as Hugo 工作区

    M->>T: 读取配置和相邻文章
    T->>W: 只读访问
    W-->>T: 文件内容与 Git 状态
    T-->>M: 上下文中的新证据
    M->>T: apply_patch 新增文章
    T->>W: 校验后写入工作区
    W-->>T: 补丁结果
    T-->>M: 文件已变更
    M->>T: hugo --renderToMemory
    T->>W: 执行本地构建
    W-->>T: 退出状态与构建输出
    T-->>M: 验证结果

第五篇里的 ToolRouter 和处理器就在这条链上。模型只看到“读取文件”“执行命令”“应用补丁”这类能力；处理器负责将请求转换为真实的文件系统或进程操作，并把结果写回任务历史。

5. 这次任务有哪些可验证证据

本次写作完成后，hugo --renderToMemory 退出状态为 0。这个参数会把渲染结果保留在内存中，适合检查模板、front matter 和内容能否被 Hugo 解析，同时不会把生成目录当成一次内容改动。

还应运行两种差异检查。git diff --check 检查已跟踪文件的空白错误；新建但尚未跟踪的文件不在普通 Git diff 中，因此还要对它运行 git diff --no-index --check /dev/null <文件路径>。后者返回状态 1 表示两份文件不同，在这个用法里只要没有输出格式错误，就说明检查通过，不能把这个 1 当成构建失败。

仓库的 GitHub Actions 工作流在推送到 main 后运行 hugo -D 并发布静态文件。本地的内存构建没有替代发布环境，也没有替代真正的部署；它验证的是当前文章在本地 Hugo 配置下能够渲染。最终汇报应把这两层证据分开写。

证据 能证明什么 不能证明什么
hugo --renderToMemory 成功 当前内容可被本地 Hugo 渲染 远程发布是否成功
git diff --check 无输出 已跟踪改动没有 Git 能识别的空白错误 新文件是否也被覆盖
新文件的 --no-index --check 无输出 新文章没有同类格式错误 文章技术事实一定正确
Git 状态仍保留既有改动 执行没有擅自清理工作区 改动已经提交或发布

最后一点常被忽略。测试通过只说明某个完成条件成立，不会自动替执行者决定是否提交、推送或部署。它们是额外的外部动作，需要用户明确要求。

6. 回看整条链路

从一句“新增文章并验证站点”到最终结果，模型做的是不断选择下一步：先读什么、是否修改、该跑哪条验证命令。Codex CLI 提供的是任务容器、上下文记录、工具调度、权限判断和事件反馈。两者合在一起，才形成一个能在本地仓库中完成工作的 Agent。

以后遇到陌生版本或陌生模块，可以沿着这条路线读源码：

  1. 先找入口和 Thread、Turn 的创建位置。
  2. 再找上下文快照怎样进入一次步骤。
  3. 跟进某个工具从模型调用到处理器、权限判断和结果事件的路径。
  4. 最后看 TUI 或 app-server 如何把结果交还给用户。

不需要从所有 Rust 模块开始。选一条你能在工作区里验证的小任务，把每个抽象名词落到一次文件读取、一次补丁或一次测试上，源码的结构就会慢慢连起来。

接下来可以继续想

  • 哪些验证应该由 Agent 自动完成，哪些动作必须留给用户确认？
  • 如果 Hugo 构建成功但 git diff 显示改了意外文件，最终汇报应如何处理？
  • 一个任务需要外部缺陷系统数据时，MCP 放在这条链路的哪个位置？
  • 当任务跨越多个仓库或远程环境时，工作区边界应怎样重新定义？
