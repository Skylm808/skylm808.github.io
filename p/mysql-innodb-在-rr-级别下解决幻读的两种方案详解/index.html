阅读时长: 4 分钟

MySQL InnoDB 在 RR 级别下解决幻读的两种方案详解

关于 幻读 (Phantom Read)，很多人的理解仅停留在“插入了一条新数据”，但更深入的问题是：MySQL 默认的 Repeatable Read (RR) 到底有没有完全解决幻读？

答案是：大部分解决了，但没完全解决。

InnoDB 使用了两套不同的机制来应对不同的读取场景（快照读 vs 当前读）。

1. 场景一：快照读 (Snapshot Read)

SQL 语句：普通的 SELECT * FROM table ...（不加锁）

机制：MVCC (多版本并发控制)

在 RR 级别下，事务启动后的第一次查询会生成一个 Read View (一致性视图)，后续所有的查询都复用这个视图。

为什么能解决幻读？

  • 就像拍了一张照片。
  • 即使别的事务 B 插入 (Insert) 了几百条新数据并提交了。
  • 事务 A 再次查询时，拿着手里的 Read View 一比对，发现这些新数据的事务 ID (DB_TRX_ID) 比自己的 Read View 还大（属于未来数据）。
  • 结果：事务 A 根本看不见这些新数据。对它来说，幻读从未发生。

结论：在纯粹的快照读场景下，MVCC 完美解决了幻读。

2. 场景二：当前读 (Current Read)

SQL 语句：

  • SELECT ... FOR UPDATE
  • SELECT ... LOCK IN SHARE MODE
  • UPDATE ... / DELETE ... / INSERT ...

注意：这些操作需要读取数据库最新、最真实的数据，不能看老照片，否则会覆盖别人的更新，导致数据错乱。

机制：Next-Key Lock (临键锁)

Next-Key Lock = Record Lock (行锁) + Gap Lock (间隙锁)

InnoDB 不仅锁住扫描到的行，还会锁住行与行之间的间隙，彻底杜绝别人在这个范围内“插队”。

案例演示：如何锁死幻读？

Next-Key Lock 的核心在于“左开右闭”区间。

假设表里现有数据：id = 3, 5。

事务 A 执行：

1

SELECT * FROM table WHERE id = 3 FOR UPDATE;

注：虽然这里是等值查询，但如果 id 不是主键/唯一索引，或者查询条件是范围，InnoDB 依然会加间隙锁。为了方便理解范围锁，我们假设这是一个范围查询或者非唯一索引查询。

更典型的范围查询例子：

1

SELECT * FROM table WHERE id > 3 FOR UPDATE;

InnoDB 的锁定范围 (Next-Key Lock)：

  1. 记录锁：锁住 id = 5。
  2. 间隙锁：
  • 锁住 (3, 5]：即 3 到 5 之间的空隙（含 5）。
  • 锁住 (5, +∞)：5 之后的无限空间。

结果： 此时，如果事务 B 尝试操作：

  1. INSERT 4：落在 (3, 5] 区间 -> 阻塞 (Blocked)。
  2. INSERT 10：落在 (5, +∞) 区间 -> 阻塞 (Blocked)。
  3. INSERT 1：落在 (-∞, 3] 区间（没被锁） -> ✅ 成功插入！

结论： Next-Key Lock 精准地锁住了查询条件覆盖的范围及其后续间隙。它并没有锁住全表，不相关的区间（比如 id=1）依然可以自由插入，既防止了幻读，又保留了一定的并发性能。

3. 特殊场景：幻读在哪里“露馅”了？

虽然 MVCC 和 Next-Key Lock 守卫森严，但在一种极其刁钻的“先查后改”场景下，幻读依然会发生。

欺骗场景复现：

  1. 初始状态：表中无 id=3。
  2. 事务 A：SELECT * FROM table WHERE id=3;
  • 结果：空集（MVCC 快照读，正常）。
  • 状态：生成了 Read View，此时看不到 id=3。
  1. 事务 B：INSERT INTO table VALUES (3); COMMIT;
  • 动作：插入并提交。因为事务 A 没加锁（只是快照读），所以事务 B 成功插入。
  • 状态：数据库物理文件里有了 id=3，其 DB_TRX_ID = 事务 B。
  1. 事务 A：UPDATE table SET name='Hack' WHERE id=3;
  • 动作：UPDATE 是当前读！ 它必须读取物理上最新的数据。
  • 结果：更新成功！
  • 关键变化：这行数据的 DB_TRX_ID 被修改成了事务 A 自己的 ID！
  1. 事务 A：SELECT * FROM table WHERE id=3;
  • 灵异事件：查到了！
  • 原因：再次复用 Read View 进行检查时：
    • 数据的 DB_TRX_ID 是事务 A 自己。
    • 符合 MVCC 规则一：“我自己修改的数据可见”。
    • 所以，幻读发生了。

总结： 这并不是 Bug，而是 MVCC 的规则逻辑自洽的结果。这也提醒我们：不要在同一个事务里混用快照读和当前读来处理同一批数据。如果业务逻辑需要“先查后改”，第一次查询就必须使用 FOR UPDATE 加锁。

4. 最终结论

“MySQL 的 RR 级别有没有解决幻读？”

  1. 对于普通查询 (快照读)：
  • 靠 MVCC 解决。通过复用 Read View，像看老照片一样屏蔽了外界的插入。
  1. 对于加锁查询/修改 (当前读)：
  • 靠 Next-Key Lock 解决。通过锁住行和间隙，物理上阻止了新数据的插入。
  1. 例外情况：
  • 如果一个事务先进行快照读，然后进行当前读（UPDATE/DELETE），再进行快照读，可能会因为自己更新了别事务插入的数据，导致幻读显形。
