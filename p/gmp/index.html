Go GMP 调度模型

一、进程、线程、协程的区别

1. 进程 (Process)

  • 定义：操作系统资源分配的基本单位
  • 特点：
    • 拥有独立的内存空间
    • 进程间通信需要 IPC（管道、消息队列、共享内存等）
    • 创建和切换开销大（需要切换页表、刷新 TLB）
    • 隔离性强，一个进程崩溃不影响其他进程

2. 线程 (Thread)

  • 定义：CPU 调度的基本单位
  • 特点：
    • 共享进程的内存空间和资源
    • 拥有独立的栈、寄存器、程序计数器
    • 创建和切换开销比进程小，但仍需内核态切换
    • 线程间通信简单（共享内存），但需要同步机制
    • 一个线程崩溃可能导致整个进程崩溃

3. 协程 (Coroutine/Goroutine)

  • 定义：用户态的轻量级线程
  • 特点：
    • 完全在用户态调度，无需内核参与
    • 创建和切换开销极小（仅保存少量寄存器）
    • Go 的 Goroutine 初始栈仅 2KB，可动态扩容
    • 由语言运行时调度，而非操作系统
    • 适合高并发场景（可轻松创建百万级协程）

对比总结

特性 进程 线程 协程
调度者 操作系统 操作系统 用户态调度器
切换开销 最大（ms级） 中等（μs级） 最小（ns级）
内存占用 最大（MB级） 中等（MB级） 最小（KB级）
通信方式 IPC 共享内存 共享内存
并发数量 数十个 数百个 数百万个

二、Go GMP 调度模型

核心组件

G (Goroutine)

  • 定义：Go 协程，用户态的轻量级线程
  • 特点：
    • 初始栈大小 2KB，最大可扩展到 1GB
    • 包含执行栈、程序计数器、状态信息
    • 由 Go 运行时管理，而非操作系统

M (Machine)

  • 定义：操作系统线程，真正执行计算的实体
  • 特点：
    • 对应一个内核线程
    • 数量默认最大 10000（可通过 GOMAXPROCS 间接影响）
    • 负责执行 G 的代码

P (Processor)

  • 定义：逻辑处理器，调度的上下文
  • 特点：
    • 数量等于 GOMAXPROCS（默认为 CPU 核心数）
    • 维护一个本地 G 队列（最多 256 个）
    • M 必须绑定 P 才能执行 G

1. G 的创建与调度

  1. 使用 go func() 创建 G
  2. 优先放入当前 P 的本地队列
  3. 本地队列满了放入全局队列

2. M 获取 G 的顺序

  1. 从当前 P 的本地队列获取
  2. 本地队列为空，从全局队列获取
  3. 全局队列为空，从其他 P 的本地队列窃取（Work Stealing）
  4. 都为空则阻塞或自旋等待

3. 调度时机

  • G 主动让出 CPU（runtime.Gosched()）
  • G 阻塞在系统调用（如 I/O）
  • G 执行时间过长（10ms，协作式抢占）
  • G 在函数调用时检查抢占标志（基于信号的异步抢占）

关键机制

1. Work Stealing（工作窃取）

  • 当 P 的本地队列为空时，会从其他 P 的本地队列尾部窃取一半的 G
  • 减少全局队列竞争，提高并发性能

2. Hand Off（移交机制）

  • 当 M 因系统调用阻塞时，会将 P 移交给其他空闲 M
  • 保证 P 不会因 M 阻塞而空闲，充分利用 CPU

3. 抢占式调度

为什么需要抢占

如果一个 G 一直占着 CPU，同一个 P 本地队列里的其他 G 就会饿死。抢占不是因为 G 出了问题，而是公平性 — 你用够了时间片，该让别人执行了。

sysmon（system monitor）

Go runtime 启动时创建的特殊后台监控线程，不需要绑定 P，独立运行。职责包括：

  • 检查 G 是否执行过久，触发抢占
  • 检查长时间阻塞在系统调用的 M，触发 Hand Off
  • 触发 GC
  • 管理网络轮询（netpoll）

异步抢占（Go 1.14+）

不再依赖函数调用，通过 OS 信号直接打断 M。

G 连续执行超过 10ms
    ↓
sysmon 向该 G 所在的 M 发送 SIGURG 信号（signal urgent，TCP 紧急数据信号）
    ↓
OS 强制中断 M 当前的执行，跳到信号处理函数
    ↓
信号处理函数：保存 G 的寄存器现场，把 G 放回队列
    ↓
M 回到调度器，执行下一个 G

为什么选 SIGURG： 不能用 SIGKILL（会杀进程），SIGUSR1 等可能跟用户代码冲突，SIGURG 在绝大多数应用中不会被使用，最安全。

解决了什么： 即使是没有函数调用的死循环，SIGURG 信号也能从 OS 层面强制打断，彻底解决了协作式抢占的缺陷。

两种抢占对比
协作式（1.2~1.13） 异步抢占（1.14+）
触发者 sysmon（10ms 巡检） sysmon（10ms 巡检）
通知方式 设置 G 的抢占标志 向 M 发送 SIGURG 信号
生效时机 G 的下一次函数调用入口 立即，无论 G 在做什么
空循环能抢占吗 不能 能
本质 G 自己检查标志，主动让出 OS 强制中断，被动让出

优势

  1. 高并发：轻量级协程，可创建百万级 Goroutine
  2. 高效调度：用户态调度，避免内核态切换开销
  3. 负载均衡：Work Stealing 机制自动平衡负载
  4. 资源利用：Hand Off 机制保证 CPU 不空闲

package main

import (
    "fmt"
    "runtime"
    "time"
)

func main() {
    // 设置使用的 CPU 核心数
    runtime.GOMAXPROCS(2) // 2 个 P

    // 创建 10 个 Goroutine
    for i := 0; i < 10; i++ {
        go func(id int) {
            fmt.Printf("Goroutine %d running\n", id)
            time.Sleep(time.Second)
        }(i)
    }

    time.Sleep(2 * time.Second)

    // 查看当前 Goroutine 数量
    fmt.Printf("NumGoroutine: %d\n", runtime.NumGoroutine())
}

三、总结

  • 进程/线程/协程：从重到轻，从内核态到用户态，适应不同并发场景
  • GMP 模型：通过 G（协程）、M（线程）、P（处理器）三层架构，实现高效的用户态调度
  • 核心优势：轻量级、高并发、自动负载均衡、充分利用多核 CPU
