go-zero 的 Model 层 Redis 缓存机制

Jan 16, 2026
阅读时长: 3 分钟

在构建高性能微服务时，数据库往往是性能瓶颈。go-zero 在 Model 层内置了一套高效且自动化的 Redis 缓存机制。本文基于源码逻辑，讲清楚它如何处理主键缓存、唯一索引缓存以及数据一致性。

1. 核心设计哲学：Index Cache（索引缓存）

go-zero 并没有采用“大杂烩”式的缓存策略，而是严格遵循 “所有查询最终都回归到 ID” 的设计原则。

1.1 两种缓存类型

  1. Primary Cache (主键缓存)
  • Key: cache:user:id:1
  • Value: {"Id":1, "Username":"admin", ...}（整行数据 JSON）
  • 作用: 真正存储数据的地方。
  1. Index Cache (唯一索引缓存)
  • Key: cache:user:username:admin / cache:user:phone:138xxxx
  • Value: 1（对应的 UserID）
  • 作用: 只是一个路标，指向主键。

2. 查询流程：从“两步走”到“一步到位”

场景 A：基于主键查询（FindOne(id)）

这是最快路径，一步到位。

  1. 代码调用 FindOne(1)。
  2. 框架直接查 Redis cache:user:id:1。
  3. Hit: 拿到 JSON，反序列化返回。
  4. Miss: 查 MySQL -> 拿到整行数据 -> 写入 Redis -> 返回。

场景 B：基于唯一索引查询（FindOneByUsername("admin")）

这是典型的“以空间换时间 + 数据归一化”策略。

  1. 第一步（找 ID）
  • 查 Redis cache:user:username:admin。
  • 拿到 UserID: 1。
  1. 第二步（找数据）
  • 框架内部自动复用 FindOne(1) 的逻辑。
  • 查 Redis cache:user:id:1 拿到详细数据。

为什么这么设计？
如果在 username 的缓存里也存一份完整的 User 数据，当用户修改昵称时，需要同时更新 id:1、username:admin、phone:xxxx 等多份缓存，极易产生不一致。
现在的方案是只维护主键缓存：删除 id:1 后，所有索引查询都会回到主键缓存，天然一致。

3. 写入与更新流程：Cache-Aside Pattern

go-zero 严格遵循 旁路缓存 (Cache-Aside) 模式。

3.1 什么时候写入缓存？

只有在读取失败（Cache Miss）时才写入。即 Lazy Load（懒加载）。

  • 并不是 update 完数据库马上写 Redis。
  • 而是 update 完只删缓存。下次谁来读，谁负责去把数据从 MySQL“搬运”到 Redis。

3.2 更新/删除时的“自动清理”

当你调用 Update(user) 或 Delete(id) 时，框架底层（sqlc.CachedConn）会自动执行以下操作：

// 伪代码逻辑
func Delete(id int64) error {
    // 1. 先查出老数据（为了拿到 username、phone 等索引键）
    data, _ := FindOne(id)

    // 2. 准备所有相关的 Redis Keys
    keys := []string{
        "cache:user:id:1",
        "cache:user:username:admin",
        "cache:user:phone:138xxxx",
    }

    // 3. 执行数据库操作 + 删除缓存
    _, err := Exec(func() error {
        return DB.Delete(id)
    }, keys...)

    return err
}

下面是 github.com/zeromicro/go-zero@v1.9.4/core/stores/sqlc/cachedsql.go 中的具体代码，ExecCtx 是生成的 model 层代码里执行数据库操作并删除缓存的方法：

func (cc CachedConn) ExecCtx(ctx context.Context, exec ExecCtxFn, keys ...string) (
    sql.Result, error) {
    res, err := exec(ctx, cc.db)
    if err != nil {
        return nil, err
    }

    return res, cc.DelCacheCtx(ctx, keys...)
}
  • 如果 Redis 里有缓存 -> DEL 命令被执行，缓存被清空。
  • 如果 Redis 里没缓存 -> DEL 命令依然被执行（返回 0），没有任何副作用。

4. 总结

go-zero 的缓存机制通过 Model 代码生成 帮我们屏蔽了复杂的细节：

  1. 数据归一：所有数据实体只存一份（在 ID 缓存里）。
  2. 索引映射：唯一索引只存 ID，通过两次查找解决问题。
  3. 一致性保证：修改数据库自动级联删除对应的所有缓存 Key（包括 ID Key 和 Index Keys）。

这套机制简单、健壮，是解决高并发读问题的最佳实践之一。
