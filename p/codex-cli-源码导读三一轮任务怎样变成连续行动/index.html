阅读时长: 5 分钟

Codex CLI 源码导读（三）：一轮任务怎样变成连续行动

现在给 Codex 一项具体任务：运行某个测试，找出第一个失败点，修复后再运行一次。

任务不会在你按下回车时立刻结束。它通常会经历几步：先读测试，再搜索相关代码，接着修改，最后运行验证。每一步的结果都会影响下一步。这一章要看的就是这段连续过程怎样在 Codex 中被保存、推进和显示。

本文固定使用 openai/codex 的 268288a 版本。

先用任务语言理解状态

先不看目录名或函数名。对一名使用者来说，任务里最常见的是下面四样东西：

名称 可以这样理解
Thread 这项长期任务的完整记录，例如围绕同一个问题连续交谈
Turn 记录中的一次请求，例如“先运行测试”
动作 一次具体操作，例如读文件、搜索、修改或运行命令
事件 任务过程发出的结构化进度，例如“正在执行命令”

假如你先提交“运行测试”，过一会儿又补充“只改最小范围”，两次输入属于同一个 Thread，却是两个 Turn。一个 Turn 内部可能产生多个动作和事件。

界面偶尔会显示 inProgress。这是源码和协议使用的英文状态值，意思就是“正在执行”。它只说明这次输入已经被接住，绝不等于问题已经修好。

flowchart LR
    U["用户：运行测试并修复"] --> T["一个 Turn 开始"]
    T --> A1["读测试、搜索代码"]
    A1 --> A2["修改文件"]
    A2 --> A3["运行目标测试"]
    A3 --> E["Turn 完成或停止"]
    A1 -.过程事件.-> V["终端或 IDE"]
    A2 -.过程事件.-> V
    A3 -.过程事件.-> V

模型负责根据已知结果选下一步；Codex 负责执行动作、记录结果，并把过程变成事件。公开源码能让我们追到这条控制流，但不能得知模型内部为何选择某个搜索词。

turn/start 到底开始了什么

第二篇提过，turn/start 是“对 Turn 做 start 动作”的协议名。客户端发送它，是在说：请在这个 Thread 中处理这一次输入。

它的返回很快，因为服务要做的第一件事只是创建并登记这轮任务。随后模型和工具仍在后台工作，事件会持续送回界面。可以把它想成餐厅取号：拿到号码表示订单已进入队列，不表示菜已经做好。

sequenceDiagram
    participant U as 开发者
    participant C as TUI 或 IDE
    participant S as app-server
    participant R as 任务运行时

    U->>C: 提交“运行测试并修复”
    C->>S: turn/start
    S->>R: 交给运行时处理
    S-->>C: inProgress（正在执行）
    R-->>S: 读文件、工具执行、文本等事件
    S-->>C: 转发事件
    R-->>S: 本轮完成
    S-->>C: turn/completed

turn/completed 同样是“Turn 已完成”的协议名。界面真正可以据此收起等待状态，而不是只凭某一段文本看起来像结束了。

进入 core 后，谁在推进任务

现在再对应源码名。app-server 中的 request_processors/turn_processor.rs 接收 turn/start，检查请求，并把用户输入交给 core。这里的 core 是本章第一篇提到的任务运行时。

core 里有几个名称，第一次见到时可以这样翻译：

源码名 它在这条任务链上的职责
CodexThread 持有一条任务记录，并把新的输入送到执行现场
Session 当前这条任务的执行现场，管理运行中的状态
SessionIo 执行现场内外传递输入、控制信号和事件的通道
Op::UserInput “用户输入”在内部使用的一种操作表示

这些不是四种新的用户概念。它们是实现 Thread、Turn 和事件时用到的内部对象。阅读时，先把 Session 记成“当前任务的执行现场”就够了。

同一时刻，Session 会让一项任务按顺序推进。用户在执行中补充的话会先进入等待位置，不会硬插进一条正在运行的命令中。这种排队看起来普通，却避免了“前一轮还在写文件，后一轮又要求撤销”的状态混乱。

重复发生的部分：决定、执行、观察

真正的多轮逻辑在 codex-rs/core/src/session/turn.rs 的 run_turn。函数名可以直译为“运行这一轮”。它做的事与前面的任务图一致：把当前材料发给模型，处理模型回复；若模型请求工具，就执行工具，再把结果放回下一轮。

flowchart TD
    A["准备当前任务材料"] --> B["请求模型给出下一步"]
    B --> C{"模型的回答"}
    C -->|"给出最终文本"| D["检查本轮是否结束"]
    C -->|"请求工具"| E["执行读取、搜索、修改或命令"]
    E --> F["记录工具结果"]
    F --> A
    D --> G["发出完成事件"]

这里的关键不是某个大而神秘的 AgentLoop 函数。Agent 的循环由这几个明确步骤组成：模型选择动作，工具产生观察，观察回到下一次模型请求。第五篇会沿着“请求工具”那条线继续看。

工具完成时，结果不会只打印在屏幕上。测试输出、文件搜索结果和补丁是否应用成功，都会进入任务记录，成为模型下一步可以参考的材料。第四篇会解释这些材料怎样被整理。

事件为什么重要

任务运行时不应该知道终端如何排版，也不应该知道 IDE 如何展示差异。它只要发出结构化事件：开始了什么、拿到了什么输出、完成了什么。app-server 把事件转交给客户端，客户端再按自己的界面方式显示。

所以，同一次“运行测试”可以在终端里成为一条进度消息，在编辑器中成为可展开的任务项。界面不必重写模型与工具之间的循环。

codex-rs/rollout 容易让人误以为是主循环所在处。以本文版本为准，它更多负责会话记录的保存、查找和恢复。追正在运行的一轮任务时，先从 turn_processor.rs、CodexThread、Session 和 run_turn 看起；追历史任务时，再去看 rollout。
