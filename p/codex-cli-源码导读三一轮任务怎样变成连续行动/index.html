阅读时长: 5 分钟

Codex CLI 源码导读（三）：一轮任务怎样变成连续行动

第二篇的终点是 turn/start。客户端把用户输入提交给 app-server，很快得到一个状态为 inProgress 的 Turn。真正的工作并没有在这次 RPC 返回前完成：模型要读到任务，可能要请求工具，工具还会把结果带回模型。整个过程在后台继续，并通过事件推回界面。

这一章要找的不是一个名为 AgentLoop 的大函数，而是这条具体路径：turn/start 怎样进入 core，输入如何排队，模型和工具如何一轮轮接力，最后又怎样发出完成事件。

本文仍以 openai/codex 的 268288a 为准。

先把四个词分开

同一项任务在不同层有不同名字。混在一起看源码，很容易觉得每个模块都在“管理会话”。

名称 它表示什么 可以先把它想成
Thread 一段可保存、恢复或分叉的对话 一条长期任务记录
Turn Thread 中的一轮用户请求 “这次请 Codex 做什么”
Session core 内部承载运行的对象 这条记录此刻的执行现场
Event 执行过程对外发出的状态变化 界面可以消费的进度消息

用户输入“修复这个失败测试”时，客户端会在已有 Thread 中启动一个 Turn。Session 接到这轮输入后才开始运行。它会不断产生 Event：某个步骤开始了，模型消息有了增量，工具正在执行，某个步骤结束了。TUI 或 IDE 只需订阅这些事件，不需要自己实现任务循环。

turn/start 做的是提交，不是同步执行

turn_processor.rs 是 app-server 接收 turn/start 的位置。它先加载目标 Thread，检查输入和当前状态，再把协议层的输入转换为 core 使用的输入类型。

随后，它把用户输入、当前目录、模型选择、审批策略、权限覆盖等内容组合成 Op::UserInput，交给 CodexThread。CodexThread 再通过 SessionIo 把这项操作提交给内部的 Session。

sequenceDiagram
    participant C as TUI 或 IDE
    participant A as app-server
    participant T as CodexThread
    participant S as Session

    C->>A: turn/start
    A->>A: 加载 Thread，校验输入和设置
    A->>T: 提交 Op::UserInput
    T->>S: 送入 SessionIo
    A-->>C: Turn inProgress
    S-->>C: 后续事件流

这段代码解释了一个常见现象：turn/start 成功，不代表任务已经成功。它只表示输入已被接住，并为这一轮分配了标识。命令输出、文件改动和最终答复都要等事件流继续推进。

Session：一轮任务的执行现场

在 codex_thread.rs 中，CodexThread 包着 Session 和 SessionIo。前者保存这条任务运行所需的状态，后者负责把操作和事件放进双向通道。

Session 有一个很明确的约束：同一时刻最多运行一个任务。用户在模型工作时补充内容，输入不会随便插进某条工具调用的中间，而是进入输入队列，在合适的时机被当前 Turn 接收。中断也不是让 UI 自己清屏，而是发给 Session 的一次控制操作。

这个约束并不华丽，却很实际。没有它，用户连续发送两条消息、工具还在运行、模型又准备请求下一步时，系统很容易把状态混在一起。把排队和中断放到运行时，界面层才能保持相对简单。

Agent 循环藏在 run_turn 里

真正的循环在 core/src/session/turn.rs 的 run_turn 中。它的注释已经给出一个朴素的描述：模型在一次采样请求后，要么给出助手消息，要么请求函数调用；若请求工具，就先执行工具，再把输出放进下一次请求。

flowchart TD
    A["Turn 输入进入 Session"] --> B["准备本轮上下文与可用工具"]
    B --> C["向模型发送采样请求"]
    C --> D{"模型返回什么？"}
    D -->|"助手消息"| E["记录消息，检查是否结束"]
    D -->|"工具请求"| F["工具运行时执行调用"]
    F --> G["记录工具结果"]
    G --> B
    E --> H["发送 Turn 完成事件"]

图中的“采样请求”就是一次向模型请求输出。“下一步”的选择发生在模型响应里；运行时负责准备请求、解释响应、调度工具和维护状态。源码可以让我们观察这套控制流，但不能告诉我们模型内部为什么偏偏选择某个搜索词或某条命令。

循环开始前，run_turn 会准备本轮需要的状态，例如上下文快照、Skill 和 Plugin 的注入信息、启动 Hook 等。它们都可能影响模型看到的内容。第四篇会专门解释这些信息从哪里来；现在只要记住，模型并不是对着一条孤立的用户消息工作。

当模型请求工具时，run_turn 使用 ToolCallRuntime 和工具路由器调度调用。工具完成后，结果会被记录进任务历史，循环再次向模型发起请求。第五篇会沿着这条工具路径继续向下走。

当模型只给出最终消息，且没有等待中的补充输入或继续条件，循环才会走向结束。完成状态经过事件映射后，客户端看到 turn/completed。这也是 UI 判断“任务真的结束了”的可靠信号。

事件不是日志，是运行时和界面的边界

第一篇里把 Event 画在任务链路上，这里可以看出它的作用。Session 不应该知道终端怎样换行，也不应该知道 IDE 怎样渲染 Diff。它只需发出结构化事件。app-server 再把这些事件转为协议通知，客户端按自己的方式显示。

同一件工具调用因此可以在终端里显示为进度行，在 IDE 里显示为可展开的卡片，也可以被另一个客户端记录成机器可读日志。任务核心不用为每种界面重写一遍。

这也解释了为什么第二篇中的 TUI 会通过 app-server 接口工作。TUI 关心事件如何显示；core 关心 Turn 怎样执行。两层共享 Thread、Turn 和 Item 这些协议概念，但不共享渲染细节。

rollout 在哪里

第一次看目录时，很容易把 rollout 当成“任务不断推进”的地方。以本文固定的源码版本来看，这个判断不准确。

codex-rs/rollout 主要负责会话记录的持久化、检索和恢复相关能力，例如保存 session 文件、查找历史 Thread、读取记录。它参与“任务留下了什么”，而不是 run_turn 中“下一步如何继续”的主循环。

这个区别很有用。想追一次正在运行的 Turn，先从 turn_processor.rs、CodexThread、Session 和 run_turn 入手；想追历史会话如何恢复，再去看 rollout 和 thread-store。目录名看起来相近，职责却不同。
