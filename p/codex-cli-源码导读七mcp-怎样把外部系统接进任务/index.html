Codex CLI 源码导读（七）：MCP 怎样把外部系统接进任务

前六篇里的工具都围绕本地仓库：搜索文件、改补丁、运行测试。可是在真实工作里，完成一次修改常常还需要仓库之外的信息，例如缺陷系统里的复现步骤、内部文档中的接口约定，或某个服务的当前状态。复制内容进对话适合一次性补充；长期使用时，更新、访问范围和调用记录都会散掉。

这里最容易混淆三件事：模型发起一次工具调用，和 Codex 怎样执行它，是一回事；工具的能力从哪里来，是另一回事。MCP 处理第二件事。它是一套让 Codex 连接外部能力的协议：Codex 作为客户端连接 MCP 服务，服务声明工具和资源，Codex 再把符合条件的工具交给当前任务使用。

本文固定使用 openai/codex 的 268288a。阅读范围是公开 CLI 源码中的 MCP 客户端路径，不讨论某个具体公司的 MCP 服务实现或其中保存的数据。

先把工具调用和 MCP 分开

“function calling”在这里可以先理解为模型与运行时之间的一种约定：模型不直接执行程序，而是返回“调用哪个工具、参数是什么”的结构化请求。比如模型可能请求：

1
2
3
4

{
  "name": "search_issues",
  "arguments": { "project": "payment", "status": "open" }
}

Codex 收到这份请求，找到对应处理器，执行动作，再把结果交还给模型。这条来回过程适用于所有工具，不只适用于 MCP。

名称 它解决的问题 在一次任务中的位置
工具调用 模型怎样提出“请做这个动作” 模型给出工具名和 JSON 参数，等待结果
内置工具 Codex 在本地已经会做什么 读文件、搜索、执行 Shell、应用补丁
MCP 外部系统怎样接进 Codex 发现外部工具，并把调用转发给 MCP 服务

所以，MCP 工具通常也会成为模型可调用的工具，但两者不能画等号。模型请求 exec_command 时，Codex 会在本地找到 Shell 工具的处理器；模型请求 search_issues 时，Codex 可能需要将同样形式的工具调用转给一个 MCP 服务。前者不经过 MCP，后者才经过。

下面仍以 search_issues 为例。先看整条路径，再进入配置和源码细节。

sequenceDiagram
    participant A as 模型
    participant C as Codex 工具运行时
    participant M as MCP 客户端
    participant S as 缺陷系统的 MCP 服务

    A->>C: 请求 search_issues(JSON 参数)
    C->>M: 确认这是 MCP 工具
    M->>S: tools/call
    S-->>M: 问题列表
    M-->>C: 工具结果
    C-->>A: 本轮可继续使用的信息

MCP 解决的是“怎么接上外部系统”，不负责替模型判断任务，也不自动授予外部系统的全部权限。

再看它解决的缺口

假设团队已经把缺陷系统接成一个 MCP 服务，提供 search_issues 工具。用户要求：“根据支付模块当前的阻塞问题，修复仓库里的失败测试。”

这里有两类动作：在本地仓库运行测试，和到缺陷系统查询问题。前者适合内置的 Shell、文件和补丁工具；后者的账号、数据模型和访问方式属于外部系统。让 Codex 自己为每一种系统编写一套集成，既难维护，也无法复用。

MCP 把外部能力收敛成统一的工具接口：先列出工具，再以工具名和 JSON 参数调用。Codex 从协议返回的工具说明中取得名称、用途和参数结构，无须为每个缺陷系统、文档系统或内部服务单独接入 API。

flowchart LR
    U["用户任务"] --> C["Codex 本地运行时"]
    C --> L["内置工具\n读文件、Shell、补丁"]
    C --> M["MCP 客户端"]
    M --> S["MCP 服务\n外部系统的适配层"]
    S --> X["缺陷系统、文档或业务服务"]
    L --> R["本地结果"]
    S --> Q["外部工具结果"]
    R --> C
    Q --> C

图中的 MCP 服务是适配层，不是模型本身。它可以是 Codex 在本机启动的一个进程，也可以是通过网络连接的服务。模型不会直接拿着数据库凭证或 HTTP 客户端绕过 Codex；它看到的是 Codex 最终暴露给它的工具规格。

配置先描述“要连谁”

codex-rs/config/src/config_toml.rs 的 mcp_servers 保存 MCP 服务定义。每个服务至少有名字、是否启用和一种传输方式。当前源码支持两类常见传输：

传输 适合的服务形态 Codex 需要的信息
stdio 本机启动的服务进程 命令、参数、环境变量和可选工作目录
streamable_http 已部署的远程服务 URL、可选 Bearer Token 环境变量和 HTTP 头

stdio 的意思是 Codex 启动子进程，并通过标准输入输出交换协议消息；streamable_http 则是在 HTTP 连接上交换协议消息。两者都不是“模型直接访问网络”的别名，它们只是 Codex 与服务建立连接的两种传输方式。

命令行里的 codex mcp add 会把服务定义写入 Codex 的全局配置。codex-rs/cli/src/mcp_cmd.rs 在创建配置时还记录了服务是否必需、启动和调用超时、默认审批方式，以及每个工具的覆盖配置。MCP 服务由运行时配置管理，而不是靠一段临时提示词维持。

有两个设置值得在读源码时先记住。enabled = false 会让 Codex 跳过该服务的初始化；required = true 则会让 codex exec 在该服务启动失败时返回错误。前者适合暂时关闭一个集成，后者适合任务离开某个外部系统就无法继续的场景。

连接成功后，先发现工具

每个 Thread 都有自己的 MCP 运行时快照。这里的 Thread 是一条持续任务的会话容器，第三篇已经解释过。每轮任务还会取得一份 Step Context，也就是这轮环境、权限和可用能力的快照。McpRuntime 保存当前的连接集合；配置或可用环境变化时，运行时可以原子地替换这组连接，让正在使用旧快照的工作继续完成。

McpConnectionManager 负责管理一组服务客户端。启动后，它通过 MCP 的初始化握手建立连接，并用 tools/list 读取每个可用服务提供的工具。list_all_tools 会汇总这些工具，跳过当前不可用的服务，再把工具名整理成模型可见的形式。

sequenceDiagram
    participant S as Step Context
    participant R as MCP Runtime
    participant M as Connection Manager
    participant P as MCP 服务
    participant T as ToolRouter

    S->>R: 取得本轮连接快照
    R->>M: 获取全部 MCP 工具
    M->>P: initialize + tools/list
    P-->>M: 工具名、说明、参数 schema
    M->>M: 过滤、规范化名称
    M-->>T: ToolInfo 列表
    T-->>S: 本轮模型可见的工具

这里的 schema 是参数结构说明。例如 search_issues 可以声明 project、status 和 query 三个字段。模型据此生成参数；服务据此验证参数。模型只需要看到稳定的名称、说明和结构，不需要了解服务背后是 SQL、REST API 还是另一套 RPC。

源码刻意区分“给协议路由用的原始名称”和“给模型看的名称”。codex-rs/codex-mcp/src/tools.rs 的 ToolInfo 同时保存服务名、原始工具名和规范化后的可调用名称。这样，模型侧的名称可以清理非法字符、解决重名；真正调用时仍会准确回到原服务的原工具名。

发现不等于全部暴露

外部服务报出十几个工具，不代表模型这一轮必然能看到十几个。Codex 还要过三道筛选。

第一道是服务配置。enabled_tools 可以成为显式允许列表，disabled_tools 可以明确屏蔽工具。ToolFilter 在列工具和真正调用时都会检查这两项，所以不能只靠“模型没看到”来防止调用。

第二道是模型可见性。tool_is_model_visible 会检查工具的协议元数据；build_mcp_tool_runtimes 只为可见工具创建处理器。若开启工具搜索，一部分 MCP 工具还会以延迟加载方式出现，模型先搜索能力，再取得具体工具规格。这和第五篇的 Tool Plan 是同一层机制。

第三道是当前 Thread 的环境与配置。Session::mcp_runtime_for_step 会根据本轮可用环境、插件贡献和服务配置生成运行时投影。服务写在配置文件里，并不保证它此刻已经成功连接，也不保证它的工具会出现在本轮的列表中。

因此，MCP 工具的可见性是一份运行时结果，不是一张静态清单。

模型调用 MCP 工具时，Codex 做了什么

当模型选中一个 MCP 工具，McpHandler 先把模型参数交给 handle_mcp_tool_call。后者解析 JSON，检查该服务和工具仍可用，发出“工具调用开始”的 Turn 事件，然后决定是否需要审批。

flowchart TD
    A["模型请求 MCP 工具"] --> B["McpHandler"]
    B --> C["解析 JSON 参数\n检查工具元数据"]
    C --> D{"需要审批？"}
    D -->|"拒绝或取消"| E["完成事件和拒绝结果"]
    D -->|"通过或无需审批"| F["McpConnectionManager.call_tool"]
    F --> G["MCP tools/call"]
    G --> H["CallToolResult"]
    H --> I["完成事件与下一轮模型输入"]

McpConnectionManager::call_tool 仍会检查配置的工具过滤规则，然后调用客户端的 tools/call。返回值被转换成 CallToolResult，其中可以有文本内容、结构化内容、错误标记和元数据。Codex 还会按模型支持的输入类型清理结果，并对发往事件流的内容施加长度限制。

对前面的缺陷系统例子而言，模型拿到的是一条绑定了调用 ID 的工具结果。它可以据此继续读本地测试、修改代码，或者提出需要用户确认的下一步。MCP 调用仍在第三篇的 Turn 循环里，并进入这条任务的历史记录。

外部能力仍要经过边界

MCP 把能力接进来，同时也把新的信任边界接进来。外部工具的参数可能触发写操作、发消息或读取私有数据；即便工具名称叫 search，也不该只根据名字判断安全性。

Codex 在 MCP 调用前沿用权限 Hook 和审批流程。mcp_tool_call.rs 处理一次性允许、会话内允许、记住允许、拒绝和取消等结果。对于服务配置，管理员或用户还可以规定默认审批方式，并按工具覆盖。这个流程和第六篇中的 Shell 审批相似，但它保护的是“让外部服务执行某个协议动作”，不是替外部服务实施它自己的权限模型。

这里要留一条清晰边界：对于 stdio 服务，Codex 负责启动并连接本地进程；对于远程 HTTP 服务，Codex 负责协议连接与本地侧的配置、认证和审批。服务最终能读什么、写什么、如何保存凭证，仍由服务和它连接的系统决定。接入 MCP 前，要分别审查 Codex 的配置范围和该服务本身的权限范围。

读源码时从这五处开始

想确认的问题 源码起点
服务怎样写入配置 codex-rs/config/src/mcp_types.rs、cli/src/mcp_cmd.rs
一条任务怎样持有 MCP 运行时 codex-rs/codex-mcp/src/runtime.rs、core/src/session/mcp.rs
怎样列出并规范化工具 codex-rs/codex-mcp/src/connection_manager/tool_catalog.rs、tools.rs
模型调用怎样变成 tools/call core/src/tools/handlers/mcp.rs、core/src/mcp_tool_call.rs
调用结果怎样返回 codex-rs/codex-mcp/src/connection_manager.rs

先按这个顺序读，会比从 codex-mcp 的目录顶层一路向下更容易建立因果关系。尤其不要先把 Connector、Plugin 和 MCP 当成同义词。MCP 是协议接入面；第八篇会再解释什么是可复用的 Skill，什么是安装与分发用的 Plugin，Hook 和 Memory 又各自放在哪里。
