Codex CLI 源码导读（九）：多 Agent 怎样拆分任务、通信与收束

一个任务变大后，主 Agent 不一定要按顺序做完所有事。排查失败测试时，定位报错、检查配置和阅读最近改动可能彼此独立。把这些工作拆开，能更快获得证据；但如果几个 Agent 同时修改同一个文件，结果也会立刻混乱。

因此，多 Agent 不是“多开几个 Shell 命令”。它要处理任务拆分、独立上下文、结果汇总和状态追踪。本章固定使用 openai/codex 的 268288a，只讨论公开源码中可观察到的 Thread 派生与状态关系。

先分清三种并行

“并行”在 Codex 里至少可能指三件不同的事。

情况 例子 它们怎样关联
Shell 管道 rg "TODO" src | head -n 20 两个操作系统进程用标准输入输出传递字节流
并行工具调用 同时读取两个互不相关的文件 同一个 Turn 中的多个工具请求由运行时收集结果
Subagent 一个查测试，一个查配置，一个审查改动 多个独立 Thread 分别完成子任务，再由上层收束

管道里的 | 是进程间通信方式。rg 的输出直接流给 head，二者通常会同时运行。它不是 Agent A 把调查结论交给 Agent B。

并行工具调用也还不是多 Agent。模型可以在一次回复中提出多个没有依赖关系的工具请求；是否真正并发执行，取决于模型能力、工具类型和运行时安排。读取两个文件适合并行，“修改文件后运行测试”则必须等待修改完成。

Subagent 走的是另一条路径：每个子任务有自己的 Thread、上下文和工具循环。它们不会共享完整的模型内部状态。

什么任务值得拆

假设主任务是“修复支付模块的失败测试，并确认没有改坏配置”。可以这样拆：

flowchart TD
    P["主任务：修复支付测试"] --> A["子任务 A：定位失败断言和调用路径"]
    P --> B["子任务 B：检查测试配置和项目指令"]
    P --> C["子任务 C：阅读近期改动，识别风险"]
    A --> R["主 Agent 汇总证据"]
    B --> R
    C --> R
    R --> D["决定最小改动"]
    D --> T["运行验证"]

前三项主要读取信息，彼此干扰很小，适合并行。修改文件和最终测试则应由一个明确的执行者串起来：多个 Agent 同时写同一个工作区，会产生覆盖、冲突和无法解释的测试结果。

一个实用判断是：子任务的输入能否写成一段独立说明，输出能否写成一份可复用的结论？能，就适合拆；若后一步必须依赖前一步刚写入的文件，则更适合留在同一个 Turn 中顺序执行。

子 Agent 不是共享脑袋

公开源码把被派生的任务记录为 SessionSource::SubAgent(SubAgentSource::ThreadSpawn { ... })，并保存 parent_thread_id、深度等信息。它说明子任务从哪个父 Thread 派生，而不表示父子任务共享同一段完整对话。

可以把主 Agent 给子 Agent 的内容理解成一份任务单：目标、必要约束、可用工作区和希望返回的结果。子 Agent 在自己的 Thread 中读取文件、调用工具、产生事件；它的价值是交回可被主任务使用的结论，例如“失败来自哪个条件分支”“配置没有覆盖这条路径”“改动会影响哪些调用方”。

主 Agent 的任务单
  ├── 子 Agent A：只读排查 payment_test.go 的首个失败原因
  ├── 子 Agent B：检查当前目录下的 AGENTS.md 与测试命令

主 Agent 收到结果后
  └── 决定是否修改、修改哪里、跑哪些验证

这与 Shell 管道的差别很大。管道传的是连续字节；Subagent 交回的是任务级结果、状态和事件。主 Agent 也不应该把子 Agent 的原始长日志全部塞回自己的上下文，而应保留对决策有用的结论和证据位置。

谁记录进度，谁决定下一步

codex-rs/agent-graph-store 保存父 Thread 与子 Thread 之间的有向边。每个子 Thread 至多有一个已持久化的父任务；边有 Open 和 Closed 等生命周期状态。存储层可以列出某个父任务的直接子任务，也可以按层级查询后代任务。

这已经是一个任务协调中心，但它的职责很具体：登记关系、保存状态、支持查询。它不是一个自动替主 Agent 分配工作的“总指挥”。是否创建子任务、给出什么任务单、拿到结果后是否继续，仍由上层编排和主任务的决策流程处理。

flowchart LR
    P["父 Thread / 主 Agent"] -->|"创建子任务"| A["子 Thread A"]
    P -->|"创建子任务"| B["子 Thread B"]
    P -->|"创建子任务"| C["子 Thread C"]
    A -->|"状态与结果"| O["协调层：Thread 关系、事件、状态"]
    B -->|"状态与结果"| O
    C -->|"状态与结果"| O
    O -->|"可查询的进度"| P
    P -->|"收束结论"| F["最终改动与验证"]

界面要显示“哪个子任务还在跑”，不必猜测模型状态。每个 Thread 和 Turn 都会产生结构化事件；协调层保存父子关系，app-server 再按 thread_id 把事件送到客户端。下一篇会继续解释这些事件怎样送到 TUI 和富客户端。

并行并不自动提高质量

多 Agent 最常见的问题不是“不会并行”，而是边界太模糊：三个子任务都在查同一件事，或两个子任务同时改同一个文件。拆分时应把输出形式写清楚，例如“返回失败的条件、证据文件和行号，不修改文件”。

对有写入需求的任务，更稳妥的做法是先让多个子任务并行收集证据，再由主 Agent 统一修改和验证。若必须并行修改，应提供隔离工作区并明确合并责任；否则子 Agent 的完成状态不能证明最终工作区仍然一致。
