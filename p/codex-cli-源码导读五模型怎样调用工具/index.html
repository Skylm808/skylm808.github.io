Codex CLI 源码导读（五）：模型怎样调用工具

模型说“我来看看这个测试”，不会让文件凭空出现在上下文里。它必须先选择一个工具，带上参数；运行时执行工具；结果再以机器可识别的形式回到模型。这条闭环才让“找代码、改代码、跑测试”成为实际操作。

Codex 的工具系统不只是把 Shell 命令拼成字符串。它要决定本轮有哪些工具可见，解析模型返回的调用，交给正确的处理器，还要把成功、失败和取消都变成下一轮可以使用的输入。本文从 ToolRouter 开始，追一条工具调用完整走完的路径。

源码仍固定为 openai/codex 的 268288a。

工具先以“规格”进入模型请求

第四篇讲到，每次采样请求都有一份 Step Context。工具也从这份快照开始。run_turn 在向模型请求输出前构建 ToolRouter，再把其中“模型可见的工具规格”放进 Prompt。

工具规格由工具名、参数结构和描述组成。它告诉模型这轮可以请求什么，却不直接执行任何动作。ToolRouter 同时保存另一份注册表，供运行时在模型真正请求调用时查找处理器。

flowchart LR
    S["Step Context"] --> R["ToolRouter"]
    R --> P["模型可见的工具规格"]
    R --> G["运行时工具注册表"]
    P --> M["模型选择调用"]
    M --> G

这个分离很重要。模型看到的是受控的能力清单，不是运行时内部全部函数。某些工具只在内部调度中使用，某些能力需要先通过工具搜索发现，还有些扩展工具只在满足配置和环境条件时出现。工具列表本身就是本轮任务边界的一部分。

ToolCallRuntime 会保留创建它的 Step Context。工具调用即使稍后才执行，也仍对应模型当时看到的那份工具和环境，而不会悄悄切到另一份配置。

从模型输出到 ToolCall

模型流式输出时，run_turn 会收到不同种类的响应项。遇到函数调用、客户端执行的工具搜索或自定义工具调用时，codex-rs/core/src/tools/router.rs 中的 ToolRouter::build_tool_call 会把它们转成统一的 ToolCall。

一条 ToolCall 至少包含三样东西：工具名、调用 ID 和参数载荷。调用 ID 看似只是字符串，实际上承担了配对责任。工具执行完后，结果会带着相同 ID 回到模型，模型才能知道“这是刚才那次搜索的结果”，而不是把它错当成另一条命令的输出。

sequenceDiagram
    participant M as 模型
    participant R as ToolRouter
    participant T as ToolCallRuntime
    participant H as 工具处理器

    M->>R: FunctionCall(name, arguments, call_id)
    R->>T: ToolCall(name, payload, call_id)
    T->>H: 依名称和参数调度
    H-->>T: 工具输出或错误
    T-->>M: FunctionCallOutput(call_id, output)

模型不需要知道 Rust 里的 handler 类型，handler 也不需要知道模型如何生成参数。ToolCall 是二者之间的协议对象。

读、改、跑：工具不是同一件事

在编码任务中，最常见的动作看上去都像“执行命令”，但运行时仍会区分它们。

exec_command 和 write_stdin 处理可持续的命令执行与输入。模型可以借此运行 rg、读取文件、执行目标测试，必要时继续向已有进程写入内容。很多“搜索文件”或“查看测试输出”的体验，底层走的就是这类执行能力。

apply_patch 则处理结构化补丁。它会解析模型生成的 diff、计算受影响路径，再进入后续的权限与写入路径。Git 相关信息、图像查看、等待环境、工具搜索等能力也各自有处理器和规格。

不要把目录名和用户动作硬绑在一起。例如，file-search crate 提供独立能力，git-utils 提供 Git 操作支撑；一次任务真正暴露给模型的工具集合仍取决于这轮的 Tool Plan、模型模式、环境和配置。读源码时，先看 spec_plan.rs 如何组装工具，再追具体 handler，会少绕很多弯路。

调度器做了什么

ToolCallRuntime 接过 ToolCall 后，会询问注册表：这个工具是否允许并行，应该由哪个运行时处理，取消时是否需要等待处理器收尾。随后，它把 Session、Turn、Step Context、取消令牌、调用 ID 和参数一起封装成 ToolInvocation，交给注册表分派。

这一步把工具调用放回任务现场。处理器不只拿到一段参数，还能知道它属于哪个 Thread、哪一轮 Turn、哪个工作目录，以及当前的权限和环境快照。

有些调用可以并发，例如互不依赖的读取；有些则必须按顺序等待。源码把这个决定收在工具注册表和 ToolCallRuntime 中，而不是让模型自己猜“这两条命令能不能同时跑”。并发不是额外炫技，它避免了工具结果的顺序和状态失控。

结果为什么必须回到模型

工具处理器完成后，输出会转换成 ResponseInputItem。成功结果会回到历史，错误通常也会以可报告给模型的形式返回。下一次采样请求会带上这些新输入项，模型据此决定继续搜索、修改代码、换一种命令，或结束任务。

flowchart TD
    A["模型请求执行目标测试"] --> B["exec_command"]
    B --> C{"测试结果"}
    C -->|"失败"| D["失败输出进入历史"]
    C -->|"通过"| E["结果进入历史"]
    D --> F["模型决定继续定位或修改"]
    E --> G["模型汇报验证结果"]
    F --> A

这也是为什么工具输出不能只显示给用户。假如测试失败文本不回到模型，下一轮模型只能继续猜；假如补丁应用失败不回到模型，它也不知道应该修正路径还是调整上下文。Agent 的反馈回路靠的正是这份结果。

工具失败不一定意味着整项任务失败。一个命令返回非零退出码，可能只是排查过程中的证据；一个补丁解析失败，可能提示模型改用更小的修改。真正需要中止的致命错误由运行时向上报告。把这几种结果区分开，Agent 才能在失败后继续工作，而不是把每次命令报错都当成最终结论。

工具能力与安全能力要一起读

到这里，工具已经能让模型读、改、跑。但“能调度”不等于“可以直接执行”。ToolInvocation 携带的权限、审批和环境信息，会在命令或写文件真正发生前参与判断。apply_patch 也会根据目标路径计算额外写入权限。

因此，工具系统有两半：前半负责把模型意图变成具体调用，后半负责决定调用是否可以跨过本地边界。把它们拆开读容易产生一种错觉，好像模型拿到 exec_command 就拥有机器上的全部权限。下一篇专门沿着这条安全路径继续追。

读源码时的路线

问题 起点
