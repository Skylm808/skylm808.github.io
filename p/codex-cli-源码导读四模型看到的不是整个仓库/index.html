Codex CLI 源码导读（四）：模型看到的不是整个仓库

上一章的任务会反复问模型“下一步做什么”。可模型做决定时，看不到整个仓库，也不应该每次都重新阅读所有历史。它真正拿到的是一份不断更新的工作记录。

这份记录至少要回答两件事：**刚才发生了什么？现在在什么环境里工作？**前者是任务历史，后者是当前环境。先把这两个问题分开，上下文管理就不再像一段神秘的大提示词。

本文固定使用 openai/codex 的 268288a 版本。

上下文其实是两类材料

还是沿用“运行测试并修复”的任务。模型要选择下一步时，至少需要知道：测试刚才报了什么错、已经读过哪些文件、当前在哪个目录、项目有什么约定，以及现在允许做什么操作。

flowchart LR
    H["任务历史\n用户输入、模型消息、工具结果"] --> P["整理后发给模型"]
    E["当前环境\n目录、项目指令、权限与可用能力"] --> P
    P --> M["模型选择下一步"]
材料 它回答的问题 例子
任务历史 刚才已经做过什么？ 测试报错、搜索结果、是否已经改过文件
当前环境 此刻在什么条件下工作？ 当前目录、AGENTS.md、权限设置

代码文件不在这张表里，是因为 Codex 不会自动把整个仓库塞给模型。模型仍要通过读取和搜索工具逐步取得代码。这样做既控制信息量，也让每次读取都有明确目的。

项目指令从哪里来

AGENTS.md 是项目给 Agent 的工作说明。它可以写构建命令、测试要求、目录约定或不该触碰的内容。对新人来说，最需要知道的一点是：它和当前目录有关。

例如：

repo/
├── AGENTS.md                 # 全仓库通用约定
└── services/payment/
    ├── AGENTS.md             # 支付服务的额外约定
    └── internal/handler/     # 当前工作目录

当任务从 services/payment/internal/handler 启动时，Codex 会沿项目目录查找相关的 AGENTS.md。仓库根的说明可以给出通用规则，子目录的说明可以补充更局部的规则。AGENTS.override.md 则会优先于同目录的普通 AGENTS.md。

当前实现位于 codex-rs/core/src/agents_md.rs。它默认以 .git 作为项目边界：不会无限向父目录搜索，也不会把机器上无关目录的说明带进当前任务。

项目指令也不是越长越好。运行时会限制它的总大小；真正影响操作的约定应该写得具体，例如“运行哪条测试命令”或“不得修改哪类生成文件”。

为什么要把环境固定成一张快照

任务执行中，当前目录、权限或可用能力可能变化。为了避免“模型以为自己在 A 环境，工具却在 B 环境执行”，Codex 在一次模型请求前会捕获当前环境的快照。

可以把快照理解成拍照：在本轮决定和工具调用开始前，先记录此刻的目录、项目指令、权限和可用能力。后续处理使用同一份记录，任务才容易解释。

sequenceDiagram
    participant R as 任务运行时
    participant M as 模型
    participant T as 工具

    R->>R: 记录本轮环境快照
    R->>M: 发送历史与当前环境
    M->>R: 请求一个工具动作
    R->>T: 按这轮环境执行动作
    T-->>R: 返回结果

在源码中，这个快照叫 StepContext，由 Session::capture_step_context 准备。名字里的 Step 可以理解成“当前这一步”，不是新的任务层级。它不会把业务代码全文装进去；它记录的是这一步运行所依赖的环境条件。

历史不是原样堆积的聊天记录

任务历史会包含用户消息、模型消息、工具请求和工具结果。它们需要被整理，才能成为下一次模型请求的材料。

例如，测试输出很长，连续几轮工具也会产生大量日志。直接重复全部内容会很快超过模型一次能接收的输入量。Codex 因此会保留对下一步有用的证据，并在接近限制时整理历史。它并不保证模型永远记住每个字，而是尽力让下一步仍有足够的信息可用。

源码中的 ContextManager 就是这个“整理工作记录”的部件。它不是另外一种模型，也不是一个外部服务；它管理按时间排列的消息和工具结果，并在发出请求前准备合适的历史。

有时你会看到 Codex 再读一次刚刚看过的文件。这不一定表示它忘了：文件可能已被修改，早前内容也可能不适合回答新的问题。是否重复读取，要看它能否带来新的证据。

历史太长时，Codex 留下什么

先说一个容易误解的点：新的 Turn 不会固定只看“最近十条”或“最近二十条”消息。只要上下文窗口还有余量，ContextManager 会把当前保留下来的整段历史整理后发给模型。消息条数没有固定上限，真正受限的是这些材料占用的 token 数量。

当历史接近模型可用的上下文窗口，Codex 会在新 Turn 前自动压缩；如果一轮任务跑得很长，也可能在这一轮中途压缩。压缩不是直接从头删消息，而是先让模型根据当前历史写一段工作总结，再用更短的新历史替换原来的长历史。

当前环境：工作目录、AGENTS.md 指令、权限和可用工具

本地压缩实现还保留了一个明确的细节：它从最新的用户消息开始向前选，最多保留约 20,000 token 的原始用户要求；放不下的最早一条会被截断。早期模型回复、工具调用和工具输出则主要由工作总结替代。工具输出在写入历史时本身也可能先被截断，所以长日志并不会无限留在会话里。

flowchart TD
    A["当前历史接近 token 上限"] --> B["根据已有历史生成工作总结"]
    B --> C["保留近期用户要求"]
    C --> D["重新注入当前环境"]
    D --> E["用较短的新历史替换旧历史"]
    E --> F["下一 Turn 在新历史上继续"]

如果连压缩请求本身都放不进上下文窗口，当前本地实现会从最早的历史项开始移除，优先让最近的内容留下来。压缩越多次，越可能丢掉边缘细节；需要长期成立的规则仍应写在 AGENTS.md，关键进度也值得在下一次提问里再明确一遍。

当前环境怎样告诉模型“发生了变化”

除了历史，Codex 还要让模型知道环境变化。例如，用户调整了权限设置，或任务切换了工作目录。第一次需要时，运行时会把完整的环境说明写入模型可见的记录；后续则尽量只补充变化的部分。

源码把这份可见环境称为 WorldState，实现位于 core/src/session/world_state.rs。这个名字容易显得宏大，实际就是“本轮任务需要对模型说明的环境状态”。WorldState 包含项目指令、权限和环境信息等片段；它配合 StepContext，让模型看到的条件与工具执行的条件保持一致。

flowchart TD
    A["第一次需要环境说明"] --> B["写入完整环境状态"]
    B --> C["后续任务继续运行"]
    C --> D{"环境变了吗？"}
    D -->|"没有"| C
    D -->|"变了"| E["只补充变化部分"]
    E --> C

MCP、Skill 或 Plugin 也可能影响可用能力和环境说明，但它们不是理解上下文的前提。先抓住“历史 + 当前环境”这条主线，扩展机制会更容易放到正确位置。

读源码时，先找这四处

关心的问题 起点
项目指令从哪里找 core/src/agents_md.rs
当前环境何时固定 Session::capture_step_context
哪些环境状态对模型可见 session/world_state.rs
历史如何整理成模型输入 context_manager/history.rs

这些文件共同构成“上下文”，却不应被当成一个目录或一段固定提示词。按问题找代码，比从目录树逐个展开更容易建立判断。
