Codex CLI 源码导读（四）：模型看到的不是整个仓库

“把项目上下文交给模型”这句话听起来很顺，实际做起来却容易出问题。一个仓库可能有几万个文件，历史会话里还有工具输出、测试日志和早先的改动。全部塞进去，模型既读不完，也未必能抓住眼下真正相关的部分。

Codex 的做法不是维护一段永远不变的大提示词。每次向模型发起请求前，运行时会整理任务历史，再构造当前工作环境的快照；第一次需要时注入完整状态，后续尽量只补充变化。上下文因此是一份持续更新的工作记录。

本文从 Session 的这条路径出发，解释 AGENTS.md、历史、当前目录、权限和动态状态怎样进入模型视野。源码版本仍固定为 openai/codex 的 268288a。

上下文不是一个东西

对 Codex 来说，模型眼前的信息至少来自两条线：已经发生过的任务记录，以及此刻的运行环境。

flowchart LR
    A["Thread 历史\n用户消息、模型消息、工具结果"] --> E["ContextManager\n规范化与裁剪"]
    B["当前工作状态\n目录、项目指令、权限、环境"] --> F["Step Context\n一次请求的快照"]
    F --> G["World State\n完整状态或变化片段"]
    G --> E
    E --> M["模型请求"]

历史回答“刚才做过什么”。当前状态回答“现在允许做什么、在哪里做、有哪些项目约定”。二者都缺不得。

假设用户要求“只修复这个测试，不要升级依赖”。模型至少要知道：当前工作目录在哪，项目是否规定了专用测试命令，前一轮测试报了什么错，以及这次会话允许写入哪些位置。单独的一句用户消息并不能带来这些信息。

AGENTS.md 怎样进入任务

AGENTS.md 不是在 CLI 启动时读一次就结束。当前源码在 core/src/agents_md.rs 中把它当作项目级指令来源，并按当前工作目录查找。

默认情况下，Codex 会向上寻找项目根目录，默认根标记是 .git；随后从项目根一路回到当前目录，按顺序收集沿途找到的 AGENTS.md。因此，一个仓库根目录可以写通用约定，某个子目录可以补充只对该目录生效的规则。AGENTS.override.md 还会优先于同目录的普通 AGENTS.md。

repo/
├── AGENTS.md                 # 通用构建与测试约定
└── services/payment/

    └── internal/
        └── handler/          # 当前工作目录

当当前目录是 services/payment/internal/handler 时，模型得到的是根目录和 services/payment 的两份指令，顺序从上到下。它不会继续越过项目根去读取父目录的文档。

这并不表示项目指令可以无限增长。源码里有总字节数限制，读到预算用完就截断。它还会记录每段指令来自哪个环境、哪个路径，供客户端和运行时追溯。对使用者而言，最实用的结论是：把真正需要执行的约定写具体，别把整套团队手册原样放进 AGENTS.md。

每次请求前，先做一次 Step Context

第三篇的 run_turn 在每次模型采样前会调用 capture_step_context。这个函数不是简单拿一下当前目录，而是捕获一次请求范围内的动态状态。

它会等待当前环境准备就绪，刷新与环境选择相符的项目指令，解析已选择的能力根目录，并准备本轮可用的 MCP 运行时。结果被包装成 StepContext。接下来的世界状态构建、工具准备和模型请求都使用同一份快照。

这样做解决了一个看似细小的问题：假如模型看到的是 A 目录和 A 权限，工具执行时却按 B 目录和 B 权限运行，任务就会变得不可解释。把同一轮的环境选择固定下来，模型看到的条件和工具实际面对的条件才有机会一致。

这里要区分“项目文件内容”和“运行环境信息”。Step Context 会告诉模型当前目录、环境、项目指令和能力状态；它不会自动把业务源代码全文读进历史。代码文件仍要由模型通过搜索和读取工具逐步取得。这个界限能让上下文保持可控，也让每次读取有明确目的。

World State：把变化写进历史

session/world_state.rs 把当前条件拆成若干段状态：项目指令、权限与审批规则、环境信息、协作模式、可用插件或应用说明等。扩展机制也可以贡献自己的状态段。

第一次真正需要这些信息时，Session 会把完整上下文写入模型可见的历史，并记下一个基线快照。后面的 Turn 再运行时，不会机械地重复整份说明；它会比较新的 World State 和基线，把变化渲染成上下文片段。例如，权限配置或环境选择变化了，模型只收到相应更新。

sequenceDiagram
    participant S as Session
    participant H as ContextManager
    participant M as 模型

    S->>S: 捕获 Step Context
    S->>H: 首次：写入完整 World State
    H->>M: 历史加完整上下文
    S->>S: 下一轮捕获新快照
    S->>H: 比较新旧 World State
    H->>M: 仅追加变化片段

这也是为什么一条任务运行久了，模型仍然能知道权限或环境已经发生变化，而不需要每轮重新阅读所有配置。它不是“记住了一切”，而是运行时把可见状态维护在历史里。

历史会被整理后再发送

ContextManager 保存的是按时间排列的响应项。用户消息、模型消息、工具调用和工具输出都可能进入其中。到真正发模型请求时，for_prompt 会先规范化这段历史，移除不适合发送的项，并按模型支持的输入类型处理图像、音频等内容。

历史不是日志文件的原样副本。工具输出可能很长，模型又有上下文窗口限制。运行时会记录 token 使用情况，并在接近窗口限制时走压缩路径。压缩的实现细节后面会穿插出现；这一章只需要建立一个判断：上下文管理的职责不是尽可能多地保留字符，而是在下一步行动前保留足够的证据。

这也能解释 Agent 有时会重新读取文件。不是每一次重新读取都表示它“忘了”。文件可能变化了，前一次读取的内容可能不再适合当前上下文，或者它需要用新的问题重新定位证据。是否重复读取，要结合任务进度看。

读源码时，先找这四处

关心的问题 起点
项目指令从哪里找 core/src/agents_md.rs
环境与指令何时刷新 Session::capture_step_context
哪些动态状态对模型可见 session/world_state.rs
历史如何整理成模型输入 context_manager/history.rs

它们共同解释了“上下文”这个词，却不应该被当成同一个文件夹。先按问题找代码，读起来会比从目录树一路展开轻松得多。
