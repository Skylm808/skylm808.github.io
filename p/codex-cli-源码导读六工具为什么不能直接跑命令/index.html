Codex CLI 源码导读（六）：工具为什么不能直接跑命令

第五篇的工具系统已经能把“运行测试”变成一次 Shell 调用。但模型会犯错，命令也可能写错路径、删错文件或访问不该访问的网络。把工具交给模型，不等于把整台机器交给模型。

这里最容易混淆的是：沙箱、Docker、命令策略和审批都在谈“限制”，却不是同一层东西。先把它们分开，第六篇后面的源码名才有实际含义。

本文仍使用 openai/codex 的 268288a。讨论范围只限公开仓库中的本地 CLI 运行路径，不推测云端服务或桌面 App 的内部权限设计。

先把沙箱和 Docker 分开

沙箱的目标是限制一个即将运行的进程：它能读哪些路径、能写哪些路径、能不能访问网络。对 Codex 而言，Shell、读取文件或其他工具最后都会变成某个本地进程或本地操作；沙箱是在它真正运行时套上的边界。

Docker 容器也能隔离程序，但它解决的是更完整的运行环境问题。容器通常会带来独立的进程视图、文件系统视图、网络配置和镜像依赖；沙箱不必创建这些东西。可以把 Docker 想成“让程序在哪个房间里运行”，把沙箱想成“即使在这个房间里，它还能碰哪些门和抽屉”。两者可以一起使用，也可以分别使用。

以本文固定版本的公开源码为准，Codex 的本地沙箱按操作系统选择实现：macOS 使用 Seatbelt，Linux 使用 seccomp 路径，Windows 使用受限令牌沙箱。它们不是 Codex 自动启动的一套 Docker 容器。若你本来就在 Docker 中运行 Codex，Docker 是外层运行环境；Codex 的权限策略与沙箱仍是另外一层限制。

概念 它解决的问题 不要把它误解为
Docker 容器 程序运行在哪个隔离环境中 Codex 的命令审批机制
Codex 沙箱 某次工具执行实际能读、写、联网到哪里 一个完整的容器镜像
命令策略 这条命令应直接允许、请求确认还是禁止 操作系统级强制隔离
审批 人是否愿意为这一次操作放行 放开所有后续命令的永久权限

沙箱也不是“绝对安全”的同义词。它是多层边界中的执行层：策略决定要不要执行，审批决定人是否同意，沙箱决定即使执行了进程仍能碰到什么。

再看一次具体命令

假设模型想在仓库里执行两件事：

1

rg "TODO" src

1

curl https://example.com/script.sh | sh

第一条通常只是读取工作区中的文本；第二条涉及网络访问，还会把网络内容交给 Shell。它们最终是否可以执行，要看当前会话和本地配置，不能只凭命令表面判断。Codex 会从四个层面判断它。

问题 对应机制 它负责什么
这次任务的默认边界在哪 权限配置（permission profile） 定义文件系统与网络的可用范围
这条命令按规则应如何处理 命令策略（exec policy） 给出允许、需要审批或禁止的判断
人是否要在这里作出选择 审批（approval） 展示具体动作，接收拒绝、一次放行或会话内放行等决定
进程启动后实际能碰什么 沙箱（sandbox） 用操作系统级机制约束文件、网络等运行时能力

审批不是沙箱。用户同意一条命令，是一次决策；沙箱是进程运行时仍然存在的约束。反过来，沙箱也不会替用户判断“这次是否值得运行”。因此，一个操作既可能需要确认，也可能在确认后仍只有有限的可写范围。

从模型请求到进程启动

前一篇的 ToolCallRuntime 把模型请求交给具体工具运行时。对 Shell 请求，codex-rs/core/src/tools/runtimes/shell.rs 中的 ShellRuntime 会携带工作目录、命令、环境和本次请求的审批要求。真正执行之前，调用会经过策略判断和沙箱转换。

flowchart LR
    M["模型请求 Shell 工具"] --> R["ShellRuntime"]
    R --> P["exec policy\n允许 / 需审批 / 禁止"]
    P -->|"禁止"| X["拒绝结果回到模型"]
    P -->|"需审批"| A["Hook、自动审查或用户确认"]
    A -->|"拒绝"| X
    A -->|"通过"| S["SandboxManager 转换启动请求"]
    P -->|"允许"| S
    S --> E["受当前权限配置约束的进程"]
    E --> O["输出回到下一轮"]

图里最容易被忽略的是“拒绝结果回到模型”。拒绝同样会形成一条工具结果。模型据此缩小操作范围、换成只读检查，或向用户说明为什么需要额外权限。

当前代码把 Shell 的执行要求表示成 ExecApprovalRequirement，只有三种结果：Skip、NeedsApproval 和 Forbidden。这比“有弹窗”更准确：有些命令不需要问，有些命令必须先问，还有些命令在当前政策下根本不能发起确认。这个类型位于 codex-rs/core/src/tools/sandboxing.rs。

权限配置先给出工作范围

codex-rs/core/src/config/permissions.rs 定义了三种内置权限配置：read-only、workspace-write 和 danger-full-access。名字已经说明了它们的相对范围：只读、可在工作区写入，以及关闭这类限制的危险模式。

名称只给出相对范围。真正要看的是配置的形状：权限配置会变成文件系统策略和网络策略，规定哪些路径能读、哪些路径能写、哪些路径明确拒绝，以及网络是否受限。工作区写入模式也可以附带具体可写根目录和网络开关。“允许改代码”会落实到路径和网络规则上。

源码中的默认选择也体现了保守原则：项目被标记为可信或不可信时，默认使用工作区写入配置；否则使用只读配置。这里的“默认”不代表每次任务都一样，用户、项目配置和运行环境仍能改变最终生效的权限。

1
2
3

工作区                    允许读取、允许写入
工作区之外的个人文件       是否可读、可写，取决于当前配置
网络                      是否可访问，取决于当前配置和运行时策略

这就是为什么同一条 rg 在不同会话里可能有不同结果。命令本身没有“天然安全”的标签；它要放进当前目录、权限配置和网络条件里解释。

命令策略不是简单的黑名单

权限配置描述的是边界，exec policy 描述的是对具体命令的判断。codex-rs/core/src/exec_policy.rs 中的 ExecPolicyManager 会先把 Shell 命令解析成要检查的命令段，再用规则和启发式判断得到 Allow、Prompt 或 Forbidden。

这样做是因为一条 Shell 输入未必只有一个程序：管道、条件执行和 Shell 包装都可能把多个动作放进同一行。策略需要面向实际的命令段，而不能只看字符串是否包含某个词。

规则还可以给出很窄的前缀许可。例如，反复运行某个固定测试命令时，配置可以让这一类命令少打断用户。源码也专门拒绝为过宽的前缀自动提出建议，例如通用 Shell、解释器或 rm；否则“记住这次允许”很容易变成“以后允许任意脚本”。

即使策略给出 Allow，也不必然代表裸进程直接运行。当前实现只有在每个解析出的命令段都被显式允许规则匹配时，才会把首次执行标记为可绕过沙箱。这个细节表明：命令许可和沙箱绕过是两份独立判断，前者不应被误读成后者。

审批把一次高风险动作变成可见决定

当策略结果是 NeedsApproval，Shell 运行时会按命令、工作目录、沙箱权限和额外权限组成审批键，并请求审批。相同键若被选择“本次会话内允许”，会被缓存，后续相同范围的请求不必反复打断用户。

审批的来源不只可能是终端里的用户。公开代码先运行权限请求 Hook，也就是在固定生命周期节点执行的一小段本地规则；它可以允许或拒绝。若没有给出决定，再根据配置交给自动审查组件或用户。无论来源是什么，最后都收敛为 ReviewDecision；拒绝、超时或中止都会转换为工具拒绝结果。

图里的 alt 是 Mermaid 时序图的语法，等同于程序中的 if / else；它不是 Codex 的术语。原图把几个分支压在一起，读起来不直观。换成下面这个判断过程会更清楚：

flowchart TD
    A["ShellRuntime 收到需要审批的命令"] --> B{"本会话是否有完全匹配的允许记录？"}
    B -->|"有"| C["直接按会话内允许执行"]
    B -->|"没有"| D["先询问权限 Hook"]
    D --> E{"Hook 是否已经作出决定？"}
    E -->|"允许"| F["执行命令"]
    E -->|"拒绝"| G["拒绝结果返回模型"]
    E -->|"未决定"| H["向用户或自动审查展示命令、目录和理由"]
    H --> I{"审批结果"}
    I -->|"仅本次允许"| F
    I -->|"本会话允许"| J["写入审批缓存"]
    J --> F
    I -->|"拒绝或取消"| G

这里要把“命令策略”和“审批缓存”分成两步看。

命令策略先给出这次请求的初步结论：可以直接执行、需要审批，或当前直接禁止。它不是只有“命令在黑名单里”和“命令在白名单里”两种答案。原因有三点：一条 Shell 输入可能有管道和多个子命令；同一个命令在不同目录、不同网络权限下风险不同；某些规则只允许非常具体的命令前缀，而不是放开整个 Shell。

当策略的结论是“需要审批”时，才进入上图。此时缓存也不是按“类似命令”模糊匹配。Shell 的审批键至少包含规范化后的命令、工作目录和环境标识。只有所有键都已经是 ApprovedForSession，Codex 才跳过这次询问；用户选择“仅本次允许”不会写入缓存。

例如，在同一个仓库和同一个会话中：

操作 可能发生什么 为什么
npm test，用户选择“本会话允许” 后续再次运行完全相同的 npm test 不再问 命令、目录和环境相同，审批键命中
npm test -- --runInBand 仍可能再问 命令内容变了，不是同一个审批键
在另一个仓库运行 npm test 仍可能再问 工作目录变了
npm test && curl https://example.com/x.sh | sh 不会因为前半段曾获允许就放行 这是一条不同的复合命令，还增加了网络和执行外部内容的风险
npm test，用户选择“仅本次允许” 下一次仍可能再问 这次允许没有被缓存

“不是简单黑白名单”的意思也在这里：策略可以让安全范围内的固定命令少打断人，但它不会把一次对 npm test 的同意扩大为“以后所有 npm 命令都不用问”，更不会扩大为“以后所有 Shell 脚本都可执行”。对补丁这类会影响多个文件的工具，源码还会按目标路径分别建立审批键；一次改 src/a.rs 的会话内同意，不应自动覆盖对 src/b.rs 的修改。

沙箱在操作系统边界上执行限制

审批通过后，Shell 运行时会构造 SandboxCommand，再交给 SandboxManager::transform。这个转换阶段把工作目录、环境、有效权限配置、网络代理条件和当前平台的沙箱类型组合成实际启动请求。

codex-rs/sandboxing 公开了平台选择：macOS 使用系统提供的 Seatbelt 沙箱，Linux 使用 seccomp 路径，Windows 使用受限令牌沙箱。具体能力会随操作系统而变，但上层接口保持同一件事：启动时带上由权限配置推导出的文件系统和网络策略。

有一个细节很能说明沙箱不是摆设：当有效文件系统策略包含“拒绝读取”的路径时，代码不允许因为升级请求就简单跳过沙箱。原因写在 sandboxing.rs 的注释里：这些拒绝读取规则只能在沙箱中得到执行，直接取消沙箱会把原本拒绝的读取悄悄放开。

Unix 下还有一个更细的路径。codex-rs/shell-escalation/README.md 说明了它如何拦截 Shell 的 execve 请求；execve 是 Unix 启动新程序的底层系统调用。协议端会返回“继续在沙箱内运行”“升级到沙箱外运行”或“拒绝”。它服务的是 Shell 子进程的升级处理，不是另一套通用权限系统。第一次读源码时，把它放在“执行过程中如何保留或升级边界”的位置即可，不必一开始就钻进补丁过的 zsh 实现。

用一个判断框架读执行结果

面对一次命令执行，不妨按下面的顺序追源码：

  1. 当前 Permission Profile 给了哪些文件和网络能力？
  2. ExecPolicyManager 对这条命令给出了 Allow、Prompt 还是 Forbidden？
  3. 如果需要审批，谁做决定，决定的范围能否被会话缓存？
  4. 进程最终由哪个平台沙箱启动，实际得到哪份有效策略？

这个顺序比问“这条命令危险吗”更有用。风险并不只来自命令名，也来自工作目录、可写路径、网络状态、用户给出的理由和当前会话的授权。

到这里，Codex 的本地 Agent 已经有了完整的动作边界：模型提出下一步，工具系统负责调度，权限、策略、审批和沙箱共同决定这一步能以什么范围发生。下一篇会离开本地工具，看看 MCP 怎样把外部系统接进这条受控路径。
