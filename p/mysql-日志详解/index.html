MySQL 日志体系详解

MySQL 的日志体系是数据库实现数据恢复、主从复制、高并发（MVCC）及故障排查的核心基石。

日志主要分为两大类：

  1. Server 层日志：MySQL 服务层产生，所有存储引擎通用（Binlog, Error Log, Slow Query Log）。
  2. 引擎层日志：特定存储引擎产生（InnoDB 的 Redo Log, Undo Log）。

1. 核心日志详解

1.1 二进制日志 (Binlog)

  • 归属：Server 层。
  • 性质：逻辑日志（记录 SQL 语句或行数据的变更）。
  • 作用：数据恢复、主从复制。
  • 写入方式：追加写。文件写满后会切换到下一个新文件。

❓ Binlog 如何实现主从一致 (Replication)？

主从复制本质上就是让从库（Slave）把主库（Master）做过的事情再做一遍。整个过程分为三步（3个线程）：

  1. Dump 线程 (Master)：
  • Slave 连接到 Master，告诉 Master：“我想要从 binlog-file-001 的第 890 位置开始同步”。
  • Master 启动一个 bin_log dump_thread，负责读取本地 Binlog，并通过网络发送给 Slave。
  1. I/O 线程 (Slave)：
  • Slave 启动一个 I/O thread，接收 Master 发来的 Binlog 数据。
  • 它不直接执行，而是将这些数据快速写入到 Slave 本地的 中继日志 (Relay Log) 中。
  • 注：Relay Log 结构和 Binlog 一模一样，只是它是存在 Slave 上的。
  1. SQL 线程 (Slave)：
  • Slave 启动一个 SQL thread，读取 Relay Log 中的内容。
  • 解析出 SQL 语句或行变更数据，在 Slave 的数据库中重放 (Replay) 执行。
  • 最终实现 Slave 数据与 Master 一致。

1.2 重做日志 (Redo Log)

  • 归属：InnoDB 引擎层。
  • 性质：物理日志。
  • 写入方式：循环写 (Circular Buffer)。Write Pos 追上 Checkpoint 时需停顿刷盘。

1.3 回滚日志 (Undo Log)

  • 归属：InnoDB 引擎层。
  • 作用：事务回滚、MVCC。
  • 写入方式：Insert 提交即删，Update/Delete 提交后放入 History List 待清理。

2. 核心机制辨析

3.1 Redo Log vs Undo Log

特性 Undo Log (回滚日志) Redo Log (重做日志)
记录的数据状态 事务开始前 (更新之前的值) 事务完成后 (更新之后的值)
主要目的 撤销操作 (回滚) 重做操作 (恢复)
崩溃恢复 提交前崩溃 -> 回滚 提交后崩溃 -> 恢复

3.2 Redo Log vs Binlog

维度 Redo Log Binlog
核心能力 Crash-safe (崩溃恢复) 归档、主从复制
写入特征 循环写 (空间固定) 追加写 (无限增长)

3. 事务执行与日志写入时序

  1. 写入 Undo Log (记录“更新前”)。
  2. 执行 Update (修改内存)。
  3. 写入 Redo Log (Prepare) (记录“更新后”，标记 Prepare)。
  4. 写入 Binlog (Server 层持久化)。
  5. 提交 Redo Log (Commit) (标记 Commit)。

4. 两阶段提交与崩溃恢复

崩溃恢复逻辑

MySQL 重启时检查 Redo Log 状态：

  1. Binlog 未写入 (Redo Log 处于 Prepare)：-> 回滚事务 (Undo)。
  2. Binlog 已写入 (Redo Log 处于 Prepare)：-> 提交事务 (Redo)。

❓ 疑难解答：为什么 Binlog 没写就要回滚？

答案：为了保证主从数据一致性。 Binlog 是主从同步的凭证。如果主库利用 Redo Log 恢复了数据（没票也拿货），但因为没写 Binlog 导致从库没收到数据，就会导致主库有数据、从库没数据。为了集群一致，主库必须回滚。
