Redis：常用数据结构与持久化原理

String（字符串）

Redis 最基础的类型，SET key value 中的 value 就是 String 类型（key 永远是字符串）。值最大 512MB。下面的编码以 Redis 7.4 为准；业务侧只使用 String、Hash 等数据类型，编码是 Redis 根据内容和大小挑选的内部存法。

底层实现 说明
int 值是可表示为 long 的整数时，直接把整数放进对象，不额外分配字符串
embstr 小字符串时，对象和 SDS 放在同一块内存里，分配与释放都更省
raw + SDS 较长或需要修改的字符串，用独立 SDS 保存字节内容

SDS 结构详解：

1
2
3
4
5
6
7

// 概念示意。Redis 会按长度选择 sdshdr8 / 16 / 32 / 64 等头部。
struct sdshdr {
    unsigned char len;    // 已使用长度
    unsigned char alloc;  // 已分配容量
    unsigned char flags;  // 头部类型
    char buf[];           // 实际存储的字节
};
  • 为什么不用 C 字符串：C 字符串以 \0 结尾，获取长度需要遍历 O(N)；SDS 记录长度，可以保存二进制字节。
  • 为什么有 embstr：SET greeting hello 这种小值很常见。把对象头和字符串分两次分配，会多出一次内存管理成本；embstr 把它们放在一起。
  • 预分配策略：SDS 扩容会留出额外空间，减少连续 APPEND 时反复分配和复制。

常用场景：

  • 缓存：SET user:1001 "{name:张三}" — 最经典的用法，缓存热点数据
  • 计数器：INCR article:1001:views — 原子自增，天然适合阅读量、点赞数
  • 分布式锁：SET lock:order NX EX 30 — NX 保证只有一个客户端能拿到锁
  • 分布式 Session：多个服务节点共享用户登录态

List（列表）

有序、可重复的元素集合，两端都能操作。

底层实现 说明
listpack 很短的 List 直接放在一块紧凑连续内存里
quicklist 较大的 List 由多个 listpack 节点组成双向链表；两端操作不必移动整条大列表，单个节点仍保持紧凑

quicklist 早期的节点使用 ziplist；当前 Redis 使用 listpack。不要把“quicklist = ziplist 链表”当作新版本结论。

常用场景：

  • 消息队列：LPUSH 生产 + BRPOP 阻塞消费，简易版 MQ
  • 最新列表：LPUSH + LTRIM 维护最新 N 条记录（如朋友圈动态、最近浏览）
  • 时间线 Feed：按时间顺序存储，LRANGE 分页获取
1
2

LPUSH news:feed "文章A" "文章B" "文章C"
LRANGE news:feed 0 9   # 取最新 10 条

Hash（哈希）

一个 key 下面挂多个 field-value 对，类似一张"小表"。

底层实现 说明
listpack field 和 value 交替连续存放。默认不超过 512 个 field，且单个 field/value 不超过 64 字节时采用
hashtable 超过阈值后转为真正的哈希表，查找 O(1)

常用场景：

  • 对象存储：比 String 存 JSON 更灵活，可以单独读写某个字段
  • 购物车：HSET cart:user1001 sku_001 2 — field 是商品 ID，value 是数量
1
2
3

HSET user:1001 name "张三" age 25 city "北京"
HGET user:1001 name       # → "张三"
HINCRBY user:1001 age 1   # 单独给 age +1，不用读写整个对象

Set（集合）

无序、不重复的元素集合，支持交并差运算。

底层实现 说明
intset 全是整数且元素少时使用，内存紧凑
listpack 小集合中出现非整数时也能紧凑保存。默认最多 128 个元素，单个元素最多 64 字节
hashtable 超过阈值后的常规实现，成员放在 key 中，value 为空

常用场景：

  • 去重：SADD 天然去重，如统计 UV（独立访客）
  • 标签系统：每个用户一个 Set 存标签，SINTER 求共同标签
  • 社交关系：共同关注 SINTER、可能认识 SDIFF、全部关注 SUNION
  • 抽奖：SADD 加入参与者，SRANDMEMBER 随机抽取，SPOP 抽完移除
1
2
3

SADD user:1001:follow "张三" "李四" "王五"
SADD user:1002:follow "张三" "赵六"
SINTER user:1001:follow user:1002:follow  # → "张三"（共同关注）

Sorted Set / ZSet（有序集合）

每个元素带一个 score（分数），按 score 排序，元素不重复。

底层实现 说明
listpack member 与 score 成对连续存放。默认不超过 128 个成员，且 member 不超过 64 字节时采用
skiplist + hashtable 跳表负责按 score 排序和范围查询 O(logN)，哈希表负责按 member 精确查找 O(1)，两者配合

跳表（Skip List）原理

跳表是一种多层链表结构，通过"跳跃"减少查找次数，性能接近平衡树但实现简单得多。

结构示意（以查找 score=50 为例）：

1
2
3

Level 3:  1 -----------------------> 50 ----------> 100
Level 2:  1 -------> 20 -----------> 50 ----------> 100
Level 1:  1 -> 10 -> 20 -> 30 -> 40 -> 50 -> 60 -> 100  (最底层完整链表)

查找过程：

  1. 从最高层开始，向右走到第一个 ≥ 目标值的节点
  2. 如果相等就找到了；如果大了就下降一层，从上一个节点继续向右找
  3. 重复直到找到或到达底层

为什么快：

  • 普通链表查找 O(N)，跳表通过"高速公路"跳过大量节点，平均 O(logN)
  • 插入/删除时，新节点随机决定层数（抛硬币），期望层数 logN，无需像红黑树那样旋转平衡

Redis 中的实现细节：

  • 每个节点最多 32 层
  • 同时维护一个哈希表，key 是 member，value 是 score，用于 O(1) 查找某个元素的分数
  • 跳表按 score 排序，score 相同时按 member 字典序排序

一条 ZSet 在内存里有几层

先把 Redis 数据库的 key 空间与 ZSet 内部结构分开。game:rank 是 Redis 数据库中的一个 key，它指向一个 ZSet 对象；它不在 ZSet 内部的哈希表里。

1
2

Redis 数据库 key 空间
└─ game:rank  ->  ZSet 对象

小 ZSet 使用一块 listpack，只有 member、score 的连续对，没有跳表和内部哈希表：

1

[listpack: player_B 50  player_A 100]

当 ZSet 超过 listpack 阈值后，ZSet 对象内部会有两份索引：

1
2
3
4
5
6
7
8
9

game:rank
└─ ZSet 对象
   ├─ hashtable（按 member 找分数）
   │  ├─ player_A -> 100
   │  └─ player_B -> 50
   │
   └─ skiplist（按 score 排序）
      50  -> player_B
      100 -> player_A

逻辑上，member 和 score 都会出现在两份索引中。物理上，Redis 会尽量复用内存：跳表节点持有 member 和 score；哈希表以 member 为 key，value 指向该跳表节点的 score。这样 ZSCORE game:rank player_A 先查哈希表，ZREVRANGE game:rank 0 9 则从跳表按分数取排名。

ZADD 怎样同时维护两份索引

先看一个已经转为“哈希表 + 跳表”的排行榜：

1
2

dict：      player_B -> 50, player_A -> 100
skiplist：  50 -> player_B -> 100 -> player_A

执行下面的命令：

1

ZADD game:rank 75 player_C

Redis 会这样做：

  1. 先在内部哈希表查 player_C。查不到，说明是新成员。
  2. 从跳表高层向右查找，定位到 score 为 50 的 player_B 和 score 为 100 的 player_A 之间。
  3. 创建跳表节点，将 player_C 插入 score 为 75 的位置。
  4. 在哈希表加入 player_C -> 75，保存对新跳表节点 score 的引用。

结果如下：

1
2

dict：      player_B -> 50, player_C -> 75, player_A -> 100
skiplist：  50 -> player_B -> 75 -> player_C -> 100 -> player_A

更新已有成员时，流程略有不同：

1
2

ZINCRBY game:rank 80 player_B
# player_B 的分数从 50 变成 130
  1. 哈希表先找到 player_B 的旧分数 50。
  2. 跳表检查新分数 130 是否还适合原来的位置。这里不适合，因为它应排到 player_A 后面。
  3. 跳表摘下 player_B，再插入 score 为 130 的新位置。
  4. 哈希表把 player_B 对应的 score 引用更新为新节点的 score。
1
2

dict：      player_C -> 75, player_A -> 100, player_B -> 130
skiplist：  75 -> player_C -> 100 -> player_A -> 130 -> player_B

若分数变化后仍夹在原来的两个邻居之间，跳表可以只更新节点分数，不必摘下再插入。无论哪条路径，命令执行结束时两份索引都指向同一组 member 与 score。

ZREVRANK game:rank "player_A"  # → 1（第二名）

先区分数据类型与内部编码

前面的 String、List、Hash、Set、ZSet 是命令层看到的类型。内部编码是另一层：同一条 HGET、ZRANGE 或 LRANGE 命令不变，Redis 会根据集合大小和元素长度选择不同布局。

先看三种转换

Hash 和 ZSet 的紧凑编码通常各自只有一块 listpack。它们达到阈值后会换成另一种索引结构，不会先切成多块 listpack。

 1
 2
 3
 4
 5
 6
 7

 9

小 Hash：
[listpack: field1 value1 field2 value2 ...]

大 Hash：
[hashtable: field1 -> value1, field2 -> value2 ...]

小 ZSet：
[listpack: member1 score1 member2 score2 ...]

大 ZSet：
[hashtable: member -> score]
[skiplist:  按 score 排序的 member]

List 是另一种情况。它很短时也可以只有一个 listpack；变大后，Redis 才把多个 listpack 节点串成 quicklist。这里的“节点大小”只适用于 List，不是 Hash、ZSet 的转换条件。

1
2
3
4
5
6
7
8

小 List：
[listpack: A B C D ...]

大 List：
quicklist
  ├─ node 1 -> [listpack: A B C D]
  ├─ node 2 -> [listpack: E F G H]
  └─ node 3 -> [listpack: I J]

Hash、ZSet 主要看元素数量和单个元素长度；List 还要看每个 listpack 节点允许的大小。这个区别先记住，后面的 listpack、quicklist 和跳表就不会混在一起。

用下面的命令可以在测试环境直接查看实际编码：

1
2
3

HSET user:1001 name "张三" age 25 city "北京"
OBJECT ENCODING user:1001
# 小 Hash 通常会显示 listpack

编码会随数据变化。它不是 Schema，也不应由业务代码依赖；它解释的是同一条 HGET 为什么在小对象和大对象上会走不同的内存布局。

listpack：现在的小对象紧凑编码

listpack 是一段连续字节数组。前一句“开头记录总长度和元素个数”说的是整段 listpack 的头部；后面的 [本项编码][数据][backlen] 说的是其中一个元素的布局。两者并不冲突，完整形状如下：

1
2
3

[总字节数][元素个数] [本项编码][数据][backlen] [本项编码][数据][backlen] [结束标记]
\________________/ \______________________________/ \______________________________/
    listpack 头部                  第一个元素                       第二个元素

“本项编码”不是业务数据，而是 Redis 为读这个元素放的元数据。它告诉解析器：这个元素是整数还是字符串；如果是字符串，后面有多少字节；如果是整数，用哪种宽度解码。随后才是实际的“数据”。例如 Hash 里的字段 age 是字符串元素，编码会说明字符串长度为 3；值 25 可以用紧凑的整数编码保存。

backlen 记录的是当前元素本身占了多少字节，放在元素尾部。Redis 从尾部向头部遍历时，先读 backlen，就能跳回这个元素的起点；再看前一个字节位置，就能读到上一个元素的 backlen。因此不需要让下一个元素保存前一个元素的长度。

Hash 中的 field、value 会交替存放；ZSet 中则是 member、score 成对存放。字符串长度和整数值都有紧凑编码，短整数不必先转成 ASCII 字符串。

backlen 不是 List 专属字段。只要数据使用 listpack，Hash、ZSet、Set，以及 List 自己或 quicklist 中的 listpack 节点，都会按这套元素格式保存。它描述的是 listpack 的二进制布局，不描述业务上的“列表元素”。

它适合小数据。比如用户资料只有 name、age、city 三个短字段时，如果每个字段都单独分配一个哈希节点、指针和 SDS，元数据往往比内容还大；listpack 把这些东西挤在一块内存里，节省的是大量对象头和指针。

连续存储也有代价：中间插入或删除需要移动后面的字节，按字段查找需要顺序扫描。因此 Redis 会在长度或单个值超过配置阈值时切成哈希表或跳表。以默认配置为例：

类型 紧凑编码 默认转出条件
Hash listpack field 数超过 512，或 field/value 超过 64 字节
Set intset 或 listpack intset 超过 512 个整数；listpack 超过 128 个元素或元素超过 64 字节
ZSet listpack 成员超过 128 个，或 member 超过 64 字节
List listpack 超过 list-max-listpack-size 所允许的单块大小，默认 -2，约为 8 KB

阈值来自 redis.conf，可以通过 CONFIG GET hash-max-listpack-*、CONFIG GET zset-max-listpack-* 等命令查看。调高阈值不必然更快：省内存与减少顺序扫描是两笔账，应当用真实数据测量。

ziplist：需要认识，但不该当成当前默认答案

ziplist 也是旧版 Redis 的紧凑连续结构。它的每个元素开头都记录前一个元素的长度，因此可以从尾部向头部走。这个字段同样不是 List 专属；旧版 Hash、ZSet 和 quicklist 节点使用 ziplist 时，都会带它。

问题出在长度扩容会向后传递。假设 A、B、C 三个元素的大小都刚好在边界附近：

1
2
3
4
5
6
7

旧 ziplist：
[A] [B 的 prevlen = A 的长度] [B] [C 的 prevlen = B 的长度] [C]

A 从 253 字节变成 254 字节：
1. B 的 prevlen 由 1 字节扩成 5 字节；
2. B 本身因此变长；
3. C 记录的 B 长度也可能要扩容。

如果后面还有更多卡在边界附近的元素，更新就会继续传递，带来多次内存移动。listpack 把长度信息留在元素自己的尾部，A 变长时只需更新 A 的内容和 A 自己的 backlen；B、C 不需要为了描述 A 而修改。

listpack 把“向后遍历所需的长度”放在元素尾部的 backlen，不再让后一个元素保存前一个元素长度，避开了这种级联更新。Redis 7.4 的源码仍保留 ziplist 代码和历史 RDB 格式兼容路径；新对象编码应以 listpack 为准。

1
2

旧 ziplist:  [前项长度][本项编码][数据] [前项长度][本项编码][数据]
listpack:    [本项编码][数据][backlen] [本项编码][数据][backlen]

旧版资料常把 ziplist 写成 Hash、ZSet 和 quicklist 节点的默认实现。读 Redis 7.x 时，应把它视为历史实现，把 listpack 视为当前默认答案。

quicklist：List 的分块形式

List 不能一直只用一块连续内存。假设一个待处理任务列表有 20 万条短消息；如果每次 LPUSH 都把所有元素塞进同一块 listpack，扩容、移动和内存碎片都会变得难看。quicklist 把它拆成许多节点，每个节点是一块 listpack，节点之间用双向指针连接。这里的节点分块只属于 List 的 quicklist，不会发生在 Hash 或 ZSet 的紧凑编码上。

1

head <-> [listpack: task-1 ... task-300] <-> [listpack: task-301 ... task-600] <-> ... <-> tail

LPUSH、RPUSH、LPOP、RPOP 多数只触碰头尾附近的节点；读取中间位置时，Redis 从更近的一端选择节点。list-max-listpack-size 控制每个节点装多少数据，list-compress-depth 还能选择只压缩中间节点，把两端若干节点留在未压缩状态，照顾常见的两端读写。

intset：小而纯的整数集合

intset 是一块有序整数数组。它不存字符串指针，只存 16、32 或 64 位整数；数组内元素按数值排序，查找用二分法。插入一个更大范围的数时，Redis 会把整块数组升级到更宽的整数类型。

1
2
3
4
5
6
7

SADD lottery:ids 7 12 42
OBJECT ENCODING lottery:ids
# 小型纯整数集合通常是 intset

SADD lottery:ids guest:1001
OBJECT ENCODING lottery:ids
# 出现非整数后，若集合仍很小，Redis 7.4 可以改用 listpack；再变大后使用 hashtable

这解释了一个容易忽略的点：Set 并非永远是“哈希表”。只有数据量或数据形态不适合紧凑编码时，才会转成哈希表。

hashtable：通用索引，也承担渐进式扩容

Redis 的字典结构服务于数据库 key 空间，也服务于大 Hash、Set 和 ZSet 的 member 查找。它使用哈希桶，平均查找、插入和删除是 O(1)。大 ZSet 会同时有一张 member 到 score 的字典和一座跳表：字典解决 ZSCORE player_A，跳表解决“分数在 100 到 200 的成员”。

哈希表扩容时不会在一条命令里搬完所有桶。Redis 同时保留旧表和新表，把后续操作遇到的桶逐步迁走，并由后台周期继续推进。这样能把一次大迁移拆开，避免某个 Hash 扩容时突然占满事件循环。

skiplist：为范围排序准备的索引

跳表已经在 ZSet 部分出现过。它的价值不只是“查找快”，而是天然维持排序顺序。排行榜里，ZSCORE 走字典；ZREVRANGE game:rank 0 9 和 ZRANGEBYSCORE game:rank 100 200 需要跳表沿着 score 的顺序移动。两个结构存的是同一批成员，内存会更多，换来的是两种查询都不用退让。

五种之外：这些命令复用了已有结构

Redis 还有一些常用功能，但并不都引入一套新的通用对象类型。

功能 内部位置 例子
Bitmap / Bitfield String 的 SDS 字节数组 SETBIT login:20260302 1001 1，一个 bit 对应一个用户是否登录
HyperLogLog 特殊 String 编码的稀疏或稠密寄存器 PFADD uv:20260302 user:1，近似统计 UV
Geo ZSet，score 是编码后的地理位置 GEOADD shops 116.397 39.908 tiananmen
Stream 基数树索引条目块，条目内容使用 listpack XADD orders * id 1001 status created

Stream 的消费者组、待确认消息等还会用字典维护状态。它比 List 消息队列多了消费组、确认和重放语义，因此不应只把 Stream 理解成“另一种 List”。

类型速查

类型 底层实现 有序 去重 一句话场景
String int / embstr / SDS — — 缓存、计数器、分布式锁
List listpack / quicklist ✅ 插入序 ❌ 消息队列、最新列表
Hash listpack / hashtable ❌ field 唯一 对象存储、购物车
Set intset / listpack / hashtable ❌ ✅ 去重、社交关系、抽奖
ZSet listpack / skiplist + hashtable ✅ score 序 ✅ 排行榜、延迟队列

持久化：RDB 与 AOF

Redis 数据全在内存里，一旦进程挂了数据就没了。持久化就是把内存数据写到磁盘，重启后能恢复。

RDB（Redis Database）— 快照

原理：在某个时间点，把内存中的全部数据生成一份二进制快照文件（dump.rdb）。

触发方式：

  • 手动：SAVE（阻塞主线程，生产环境别用）/ BGSAVE（fork 子进程，推荐）
  • 自动：配置 save 900 1（900 秒内至少 1 次写操作就触发 BGSAVE）

BGSAVE 执行流程：

1
2
3
4
5
6

1. 主进程 fork() 出一个子进程
2. 子进程拥有父进程的内存页表副本（此时并不复制实际数据）
3. 子进程遍历内存，将数据序列化写入临时 RDB 文件
4. 写完后替换旧的 dump.rdb
5. 期间主进程继续处理请求，如果有写操作 → 触发 COW（Copy-On-Write）：
   操作系统只复制被修改的那个内存页，子进程仍然读到 fork 时刻的旧数据

COW（写时复制） 是关键：fork 时不真正复制内存，只有主进程写数据时才按页复制，所以 fork 很快，内存开销也可控。

优点：文件紧凑、恢复速度快（直接加载二进制）、对主线程影响小 缺点：两次快照之间的数据可能丢失（比如配置 5 分钟一次，最多丢 5 分钟数据）

AOF（Append Only File）— 追加日志

原理：把每一条写命令以文本形式追加到 appendonly.aof 文件末尾。恢复时重放所有命令。

写回策略（appendfsync）：

策略 行为 数据安全 性能
always 每条命令都 fsync 到磁盘 最高，最多丢 1 条命令 最差
everysec（默认） 每秒 fsync 一次 最多丢 1 秒数据 折中，推荐
no 交给操作系统决定何时刷盘 可能丢较多数据 最好

AOF 重写（Rewrite）：

AOF 文件会越来越大（比如同一个 key 被 SET 了 100 次，文件里就有 100 条命令）。重写机制解决这个问题：

1
2
3
4
5
6
7
8

1. 主进程 fork() 子进程
2. 子进程根据当前内存数据，生成一份"最小命令集"的新 AOF 文件
   （100 次 SET 同一个 key → 只保留最后一条）
3. 重写期间主进程的新写命令同时写入：
   - 旧 AOF 文件（保证安全）
   - AOF 重写缓冲区
4. 子进程写完后，主进程把重写缓冲区的命令追加到新 AOF 文件
5. 用新文件替换旧文件

优点：数据安全性高（最多丢 1 秒）、文件可读（纯文本命令） 缺点：文件比 RDB 大、恢复速度比 RDB 慢（要重放命令）

RDB vs AOF 对比

RDB AOF
持久化内容 某一时刻的全量数据快照（二进制） 所有写命令的追加日志（文本）
数据安全 可能丢两次快照之间的数据 最多丢 1 秒（everysec）
文件大小 小（二进制压缩） 大（即使重写后仍比 RDB 大）
恢复速度 快（直接加载） 慢（逐条重放命令）
对性能影响 fork 时有短暂开销 everysec 模式下几乎无感
适用场景 冷备份、灾难恢复、对丢数据容忍度高 对数据安全要求高的业务

混合持久化（Redis 4.0+）

问题：RDB 恢复快但可能丢数据，AOF 数据安全但恢复慢，能不能结合两者优点？

解决方案：混合持久化——在 AOF 重写时，把文件拆成两部分：

1
2
3
4
5

┌─────────────────────────────────────────┐
│  RDB 格式的全量数据（二进制，紧凑）          │  ← fork 时刻的内存快照
├─────────────────────────────────────────┤
│  AOF 格式的增量命令（文本）                 │  ← 重写期间新产生的写命令
└─────────────────────────────────────────┘

开启方式：

1
2

# redis.conf 配置
aof-use-rdb-preamble yes

工作流程（以 AOF 重写为例）：

1
2
3
4
5
6
7

1. 主进程 fork() 子进程
2. 子进程把当前内存数据以 RDB 格式写入新 AOF 文件（快速、紧凑）
3. 重写期间，主进程的新写命令同时写入：
   - 旧 AOF 文件（保证不丢数据）
   - AOF 重写缓冲区
4. 子进程写完 RDB 部分后，主进程把重写缓冲区的增量命令以 AOF 格式追加到新文件末尾
5. 用新文件替换旧文件

恢复时的处理：

Redis 启动时读取混合格式的 AOF 文件：

  1. 先识别文件头是否为 RDB 格式（魔数 REDIS）
  2. 如果是，先快速加载 RDB 部分（全量数据）
  3. 再逐条重放后面的 AOF 命令（增量数据）

优势总结：

维度 纯 RDB 纯 AOF 混合持久化
数据安全 ❌ 可能丢几分钟 ✅ 最多丢 1 秒 ✅ 最多丢 1 秒
恢复速度 ✅ 快 ❌ 慢 ✅ 快（RDB 部分）+ 增量重放
文件大小 ✅ 小 ❌ 大 ✅ 比纯 AOF 小很多

一句话：混合持久化 = RDB 的快速恢复 + AOF 的数据安全，是生产环境的最佳实践。

实际生产建议

  1. 开启混合持久化（Redis 4.0+）：
1
2
3

appendonly yes
aof-use-rdb-preamble yes
appendfsync everysec

  1. 恢复优先级：Redis 启动时，如果 AOF 文件存在就优先用 AOF 恢复（数据更完整），否则用 RDB。

  2. 定期备份 RDB：即使开了 AOF，也建议定期把 dump.rdb 备份到远程（如对象存储），作为灾难恢复的最后一道防线。
